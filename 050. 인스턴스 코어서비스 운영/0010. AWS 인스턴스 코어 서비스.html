Compute Services

안전한 가상 서버 구축의 4대 핵심 요소:
인스턴스 코어 서비스

Amazon EC2 인스턴스는 단순히 CPU와 메모리만으로 동작하지 않습니다. 운영 체제 이미지, 안전한 접속을 위한 인증 키, 초기화 자동화 스크립트, 그리고 안전한 권한 관리가 결합되어야 비로소 실무에 투입 가능한 완전한 서버로 동작합니다.

AMI (Amazon Machine Image, 아마존 머신 이미지)

AMI는 인스턴스를 시작하는 데 필요한 소프트웨어 구성(운영 체제, 애플리케이션 서버, 애플리케이션 등)이 포함된 마스터 템플릿(Master Template)입니다. 인스턴스의 하드웨어(CPU/RAM)가 그릇이라면, AMI는 그 그릇에 담기는 내용물입니다.

AWS Quick Start AMI

AWS가 기본적으로 제공하고 유지 관리하는 운영 체제(Amazon Linux, Ubuntu, Windows 등) 이미지입니다.

Custom AMI (사용자 지정)

사용자가 소프트웨어를 설치하고 환경 설정을 마친 후 직접 구운(Baking) 이미지입니다. 일명 골든 이미지(Golden Image)1라고 불립니다.

AWS Marketplace

서드파티(제3자) 벤더가 보안 솔루션, 상용 방화벽, 특별한 소프트웨어를 설치하여 판매하는 유료/무료 이미지입니다.

실무 작업 예제: AWS CLI를 이용한 최신 AMI 조회

Terminal

인프라 자동화(IaC) 구성 시, 운영 체제의 최신 보안 패치가 적용된 AMI ID를 동적으로 가져오는 것은 필수적입니다. 아래는 AWS CLI(Command Line Interface, 명령줄 인터페이스)4를 통해 최신 Amazon Linux 2023 이미지를 검색하는 명령어입니다.

aws ec2 describe-images \
    --owners amazon \
    --filters "Name=name,Values=al2023-ami-2023.*-x86_64" "Name=state,Values=available" \
    --query "sort_by(Images, &CreationDate)[-1].[ImageId, Name]" \
    --output text
결과 예시: ami-0c9c942bd7bf113a2 al2023-ami-2023.4.20240401.1-kernel-6.1-x86_64

Key Pair (키 페어)

EC2 인스턴스에 안전하게 접속하기 위해 사용되는 비대칭 키 암호화(Asymmetric Key Cryptography) 방식의 인증 수단입니다. 암호화와 복호화에 서로 다른 키를 사용하므로 비밀번호 기반 접속보다 무차별 대입 공격(Brute-force attack)에 대해 훨씬 강력한 보안을 제공합니다.

  • Public Key (퍼블릭 키): AWS가 보관하며, 인스턴스 생성 시 운영체제 내부(~/.ssh/authorized_keys)에 자동으로 삽입됩니다.
  • Private Key (프라이빗 키): 사용자가 최초 1회 다운로드하여 로컬 PC에 안전하게 보관하는 .pem 또는 .ppk 파일입니다.

프라이빗 키 분실 및 유출 주의

프라이빗 키 파일은 단 한 번만 다운로드할 수 있습니다. 분실 시 복구가 불가능합니다.

외부에 유출될 경우 누구나 서버에 최고 관리자(Root) 권한으로 접속할 수 있으므로 절대 GitHub나 공개형 스토리지에 업로드하면 안 됩니다.

실무 작업 예제: SSH를 이용한 안전한 인스턴스 접속

Bash

리눅스 환경에서 SSH(Secure Shell, 보안 셸)5를 사용하여 접속할 때, 프라이빗 키 파일의 권한이 너무 열려있으면 접속이 차단됩니다(Unprotected Private Key File 에러). 반드시 권한을 제한한 후 접속해야 합니다.

# 1. 키 파일 권한 변경 (소유자만 읽기 가능하도록 400 설정)
chmod 400 my-production-key.pem

# 2. SSH를 통한 EC2 인스턴스 접속 (Amazon Linux의 기본 사용자명은 ec2-user)
ssh -i "my-production-key.pem" ec2-user@203.0.113.25

User Data (사용자 데이터)

인스턴스가 최초로 부팅될 때 딱 한 번 실행되는 자동화 스크립트입니다. 이를 통해 서버 관리자가 직접 접속하지 않아도 패키지를 업데이트하고, 웹 서버를 설치하고, 소스 코드를 다운로드하는 일련의 환경 구성 과정을 완전 자동화할 수 있습니다. 이러한 무인 자동화 과정을 부트스트래핑(Bootstrapping)2이라고 합니다.

실무 작업 예제: 웹 서버 자동 배포 스크립트

Bash Script 
#!/bin/bash
# 실행 과정 로깅 설정 (트러블슈팅 용이성 확보)
exec > >(tee /var/log/user-data.log|logger -t user-data -s 2>/dev/console) 2>&1

echo "==== 시스템 구성 시작 ===="
# 시스템 패키지 최신화
yum update -y

# 아파치(Apache) 웹 서버 및 기본 유틸리티 설치
yum install -y httpd wget jq

# 서비스 자동 실행 등록 및 즉시 시작
systemctl enable httpd
systemctl start httpd

# 로드밸런서(ELB) 상태 검사용 상태 페이지 및 메인 페이지 생성
echo "OK" > /var/www/html/health.html
echo "<h1>AWS EC2 자동 배포 성공! ($(hostname))</h1>" > /var/www/html/index.html

echo "==== 시스템 구성 완료 ===="

실전 트러블슈팅 (Troubleshooting)

User Data 스크립트에 오타가 있거나 실행 중 오류가 발생하면, EC2 콘솔에서는 정상적으로 '실행 중'으로 보이지만 원하는 웹사이트는 접속되지 않습니다. 이때는 인스턴스에 SSH로 접속하여 /var/log/cloud-init-output.log 파일을 확인하면 스크립트의 전체 실행 로그와 에러 원인을 파악할 수 있습니다.

Instance Metadata (인스턴스 메타데이터) 및 IAM 역할

인스턴스 내부의 거울: Metadata

EC2 인스턴스가 자기 자신의 환경 정보(예: 내부 IP, 퍼블릭 IP, 인스턴스 ID, 연결된 보안 그룹 등)를 스스로 조회할 수 있게 해주는 고유 데이터입니다. 인스턴스 내부에서만 접근 가능한 링크 로컬(Link-local) 주소인 169.254.169.254를 호출하여 데이터를 가져옵니다.

보안 권장: SSRF(서버 측 요청 위조)3 공격을 원천 방어하기 위해, 반드시 세션 토큰 발행을 요구하는 IMDSv2(Instance Metadata Service version 2)만을 사용하도록 강제하는 것이 현대 클라우드 보안의 표준입니다.

안전한 권한 부여: IAM Role (역할)

EC2 인스턴스가 애플리케이션 실행 중 S3 버킷의 파일을 읽거나, DynamoDB에 데이터를 기록하는 등 다른 AWS 서비스에 API 호출을 해야 할 때가 있습니다. 이때 절대 소스 코드나 환경 변수에 하드코딩된 영구 자격 증명(Access Key)을 두어서는 안 됩니다.

대신 IAM(자격 증명 및 액세스 관리) Role(역할)을 인스턴스에 부여(Attach)하십시오. 그러면 인스턴스가 메타데이터 서비스를 통해 일정 시간 후 자동 파기되는 임시 보안 자격 증명(STS)을 안전하게 자동 발급받아 사용합니다.

실무 작업 예제: IMDSv2를 이용한 안전한 메타데이터 호출

cURL

IMDSv2를 사용하려면 먼저 PUT 요청으로 일시적인 토큰(Token)을 발급받은 후, 해당 토큰을 헤더(Header)에 담아 GET 요청을 보내야 합니다. 아래는 인스턴스 자신의 고유 ID(Instance ID)를 안전하게 조회하는 스크립트 예제입니다.

# 1단계: 세션 토큰 발급 (유효시간 21600초 = 6시간 설정)
TOKEN=`curl -X PUT "http://169.254.169.254/latest/api/token" \
    -H "X-aws-ec2-metadata-token-ttl-seconds: 21600"`

# 2단계: 발급받은 토큰을 사용하여 인스턴스 ID 조회
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
    -s http://169.254.169.254/latest/meta-data/instance-id

# 출력 결과 예시: i-0abcd1234efgh5678

코어 서비스 설정 시 과금 및 보안 주의사항 (필독)

EIP (Elastic IP, 탄력적 IP) 미사용 과금 폭탄

변하지 않는 고정 공인 IP를 제공하는 EIP는 인스턴스에 연결(Associate)되어 인스턴스가 정상 실행 중일 때는 요금이 청구되지 않습니다(무료). 하지만, IP 고갈을 막기 위한 AWS의 정책에 따라 EIP를 할당만 받아 놓고 연결하지 않거나, 연결된 인스턴스를 중지(Stop) 상태로 두면 시간당 요금이 계속 청구됩니다. 인스턴스를 삭제하거나 중지할 때는 반드시 EIP를 먼저 릴리즈(Release, 계정에서 해제)해야 합니다.

Marketplace AMI 라이선스 요금 부과

AWS Quick Start에서 제공하는 Amazon Linux나 Ubuntu 기본 이미지는 무료(하드웨어 인프라 비용만 발생)입니다. 하지만, AWS Marketplace에서 판매하는 특정 서드파티 AMI(예: Windows Server with SQL Server, Palo Alto 상용 방화벽 등)는 기본 EC2 인스턴스 시간당 요금 외에 벤더사의 소프트웨어 라이선스 비용이 추가로 합산되어 이중 과금됩니다. 실습 생성 시 예상 요금 단가를 반드시 재차 확인하십시오.

주석 및 용어 설명

  1. 골든 이미지 (Golden Image): 실무 환경에서 수십~수백 대의 서버를 일관되게 배포(Scaling)하기 위해, 최신 보안 패치, 모니터링 에이전트, 필수 소프트웨어 등을 완벽히 설치하고 보안 검수를 완료한 후 생성해 둔 '표준 마스터 이미지'를 뜻하는 실무 용어입니다.
  2. 부트스트래핑 (Bootstrapping): 운영 체제가 가동된 직후, 사람의 수동 개입 없이 자동으로 시스템을 설정하고 필요한 소프트웨어를 설치하여 서비스가 즉시 동작 가능한 상태(Desired State)로 만드는 일련의 과정을 의미합니다.
  3. SSRF (Server-Side Request Forgery, 서버 측 요청 위조): 해커가 취약한 외부 대면 웹 애플리케이션을 조종하여, 서버가 내부 네트워크(이 경우 169.254.169.254 메타데이터 서버)로 악의적인 요청을 보내게 만들어 서버 내 임시 보안 자격 증명이나 중요 설정값을 탈취하는 치명적인 해킹 기법입니다.
  4. CLI (Command Line Interface, 명령줄 인터페이스): 마우스 기반의 그래픽 사용자 인터페이스(GUI) 콘솔을 사용하지 않고, 텍스트 터미널 창에서 명령어를 타이핑하여 시스템을 제어하고 클라우드 리소스를 관리하는 도구입니다. 인프라 관리를 스크립트로 자동화할 때 필수적입니다.
  5. SSH (Secure Shell, 보안 셸): 네트워크 상의 다른 컴퓨터(서버)에 로그인하거나 원격 시스템에서 명령을 실행하기 위해 사용되는 응용 프로그램 또는 그 프로토콜입니다. 주고받는 모든 데이터가 암호화되어 패킷 스니핑 등의 보안 위협으로부터 안전합니다.
