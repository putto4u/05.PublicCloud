**AWS(Amazon Web Services, 아마존 웹 서비스) 핵심 키(Key) 종류 및 사용 용도 총정리**

AWS 환경에서는 자원에 접근하거나 데이터를 암호화하기 위해 다양한 형태의 키(Key)가 사용됩니다. 각 키는 고유한 발급 목적과 사용 범위를 가지며, 이를 정확히 구분하여 사용하는 것은 클라우드 보안 아키텍처 설계의 가장 기본적이고 핵심적인 요소입니다.

| --- | --- | --- | --- |

<br>(액세스 키 아이디 및 비밀 액세스 키) | 프로그래밍 방식(CLI, SDK)으로 AWS API를 호출하여 클라우드 자원을 제어할 때 사용하는 장기 자격 증명 | **전면 무료**<br>

### 2. 키 종류별 상세 설명 및 추가 사용 용도

#### **A. Access Key (액세스 키 아이디 및 비밀 액세스 키)**

IAM(Identity and Access Management, 항등 및 접근 관리) 사용자에게 부여되는 영구적인 자격 증명입니다. 웹 브라우저(AWS 콘솔)를 통한 로그인이 아닌, 외부 시스템이나 스크립트에서 AWS 클라우드를 제어할 때 필수적입니다.

* **기본 용도:** 개인 PC의 CLI(Command Line Interface, 명령줄 인터페이스) 또는 Terraform(테라폼)과 같은 IaC(Infrastructure as Code, 코드형 인프라) 도구에서 환경 변수로 등록하여 AWS 계정을 인증합니다.
* **추가 사용 용도:**
* **온프레미스(On-premises, 자체 보유 전산실) 연동:** 사내 데이터 센터의 백업 서버가 정기적으로 AWS S3(Simple Storage Service, 단순 스토리지 서비스)로 백업 파일을 전송할 때, 전송 스크립트 내부에 인증 수단으로 사용됩니다.
* **CI/CD(Continuous Integration/Continuous Deployment, 지속적 통합/지속적 배포) 파이프라인:** GitHub Actions(깃허브 액션스)나 Jenkins(젠킨스) 서버가 빌드된 애플리케이션 코드를 AWS 자원에 자동 배포할 때 인증 키로 활용됩니다.



> **💡 실전 팁 및 자주 오해하는 부분**
> * **절대 주의(해킹 사례 1위):** 소스 코드를 [GitHub(깃허브)](https://github.com) 등에 커밋할 때 `Secret Access Key(비밀 액세스 키)`가 코드 내에 하드코딩(Hardcoding, 데이터를 코드 내부에 직접 입력하는 방식)된 상태로 퍼블릭 리포지토리에 올라가면, 1분 이내에 전 세계의 해킹 봇이 이를 스캔하여 악성 암호화폐 채굴 서버를 수백 대 생성합니다. 수천만 원의 **요금 폭탄**을 맞을 수 있으므로 반드시 `환경 변수(Environment Variables)`나 `.env` 파일로 분리하고 `.gitignore`에 등록해야 합니다.
> * **모범 사례 대안:** AWS 내부의 자원(예: EC2 서버)이 다른 AWS 자원(예: S3 버킷)에 접근할 때는 절대 Access Key를 발급받아 서버에 저장하지 마십시오. 대신 **IAM Role(아이엠 역할, 항등 및 접근 관리 역할)**을 EC2(엘라스틱 컴퓨트 클라우드)에 부여하여 임시 자격 증명을 자동으로 획득하게 하는 것이 보안 아키텍처의 핵심 원칙입니다.

#### **B. Key Pair (아마존 엘라스틱 컴퓨트 클라우드 키 페어)**

공개 키(Public Key) 암호화 방식을 사용하여 AWS 가상 머신에 접속할 때 패스워드를 대체하는 수단입니다. AWS가 퍼블릭 키를 서버에 보관하고, 사용자는 프라이빗 키(`.pem` 또는 `.ppk` 파일)를 다운로드하여 소유합니다.

* **기본 용도:** Linux(리눅스) 서버에 SSH(Secure Shell, 보안 셸)로 접속하거나, Windows(윈도우) 서버의 초기 관리자 비밀번호를 해독(Decryption)할 때 사용합니다.
* **추가 사용 용도:**
* **Ansible(앤서블) 자동화 환경 구성:** 마스터 PC(강사 PC)에서 여러 대의 예비 서버(수강생 PC)로 소프트웨어를 일괄 배포할 때, 이 프라이빗 키를 Ansible 설정 파일에 매핑하여 패스워드 입력 없이 다수의 서버에 동시 접속하고 제어하는 데 사용합니다.
* **Bastion Host(베스천 호스트, 외부 접속 차단망 내의 접근 제어용 서버) 구축:** 퍼블릭 서브넷(외부 접근 허용 네트워크)의 베스천 서버를 거쳐 프라이빗 서브넷(내부 보안 네트워크)의 데이터베이스 서버로 터널링(Tunneling, 안전한 데이터 전송 통로 생성) 접속을 할 때 인증 수단으로 사용됩니다.



> **💡 실전 팁 및 자주 오해하는 부분**
> 프라이빗 키는 EC2 인스턴스를 생성하는 시점에 **단 한 번만 다운로드**할 수 있습니다. 이를 분실하면 기존 방식으로는 해당 서버에 접근할 수 없습니다. 분실 시 인스턴스를 중지하고 디스크를 분리하여 다른 인스턴스에 붙인 뒤 초기화하는 복잡한 과정이 필요하므로, 키 관리에 각별히 유의해야 합니다. 최근에는 키 페어 없이도 안전하게 접속할 수 있는 **SSM Session Manager(시스템 매니저 세션 매니저)** 사용이 권장됩니다.

#### **C. KMS Key (아마존 웹 서비스 키 관리 서비스 키)**

데이터를 암호화하기 위한 마스터 키(Customer Master Key, CMK)를 생성하고 관리하는 관리형 서비스입니다.

* **기본 용도:** AWS 클라우드에 저장되는 민감한 데이터(RDS 데이터베이스 스토리지, S3 버킷 파일, EBS 볼륨 등)를 유출로부터 보호하기 위해 '저장 데이터 암호화(Encryption at Rest)'를 수행합니다.
* **추가 사용 용도:**
* **애플리케이션 봉투 암호화(Envelope Encryption):** 개발자가 만든 애플리케이션 내부에서 직접 사용자의 주민등록번호나 결제 정보 등 특정 필드 데이터만 암호화할 때, KMS API를 호출하여 데이터를 보호합니다.[^2]
* **비밀 정보 관리:** AWS Secrets Manager(시크릿 매니저)와 연동하여 데이터베이스 접속 패스워드나 외부 연동 API 토큰 자체를 암호화된 상태로 안전하게 보관합니다.



#### **D. API Key (아마존 에이피아이 게이트웨이 에이피아이 키)**

주로 서버리스(Serverless, 서버를 직접 관리하지 않는 아키텍처) 환경에서 사용자 고유의 API를 서비스할 때 사용하는 접근 제어 문자열입니다.

* **기본 용도:** 만들어둔 API 서비스를 외부 개발자나 파트너사가 호출할 때, 허가된 사용자만 접근할 수 있도록 식별하는 용도입니다.
* **추가 사용 용도:**
* **사용량 제어(Throttling) 및 할당량(Quota) 관리:** '고객사 A는 초당 10회 호출, 월간 10,000회 호출만 허용'과 같이 특정 키마다 접속 트래픽을 제한하는 사용량 계획(Usage Plan)에 맵핑하여, 시스템 과부하를 방지하고 서비스 티어(무료/유료 구독)를 분리하는 비즈니스 모델 구축에 사용됩니다.
