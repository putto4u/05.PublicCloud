## EC2 Auto Scaling과 EKS의 차이 및 필요성

AWS에서 **EC2 Auto Scaling**이 인스턴스 개수를 자동으로 조절해 주는데도 불구하고 **EKS(Elastic Kubernetes Service)**를 사용하는 이유는 '무엇을 스케일링하는가'와 '어떻게 관리하는가'의 단위가 다르기 때문입니다.

---

### 1. EC2 Auto Scaling vs EKS 핵심 비교

| 구분 | EC2 Auto Scaling (ASG) | Amazon EKS (Kubernetes) |
| --- | --- | --- |
| **관리 단위** | **가상 서버 (Instance)** | **컨테이너 (Pod)** |
| **스케일링 대상** | CPU/메모리 부하에 따른 서버 대수 | 서비스 요청량에 따른 컨테이너 개수 |
| **자원 효율성** | 서버 한 대를 통째로 추가 (무거움) | 기존 서버 내 남는 자원에 컨테이너 추가 (가벼움) |
| **복구 방식** | 서버 장애 시 새 서버 생성 | 컨테이너 장애 시 즉시 재시작 (Self-healing) |
| **배포 방식** | AMI 빌드 및 인스턴스 교체 (느림) | 컨테이너 이미지 교체 (매우 빠름) |

---

### 2. EKS를 써야 하는 결정적인 이유 (오토 스케일링 관점)

단순히 서버 대수를 늘리는 것 이상의 **입체적인 확장성**이 필요할 때 EKS가 필수적입니다.

* **HPA (Horizontal Pod Autoscaler)**: 트래픽이 몰릴 때 서버를 새로 띄우는 것이 아니라, 이미 떠 있는 서버의 남는 자원에 컨테이너(Pod)만 빠르게 복제합니다. 서버 한 대가 뜨는 데 몇 분이 걸린다면, 컨테이너는 수 초 내에 확장됩니다.
* **CA (Cluster Autoscaler) / Karpenter**: 컨테이너들을 다 수용하기에 서버 자원이 부족해지면, 그때 비로소 EC2 인스턴스를 추가합니다. 즉, **[앱 확장 -> 서버 확장]**으로 이어지는 지능적인 2단계 스케일링이 가능합니다.
* **빈 패킹 (Bin Packing)**: 여러 서비스의 컨테이너들을 서버 자원에 맞춰 촘촘하게 배치하여, 서버 대수를 최소화하고 비용을 절감합니다.

---

### 3. 기술 전문가의 실전 요약

EKS는 **'애플리케이션 단위의 유연함'**을 위해 사용합니다. 만약 서비스가 단일 모놀리식 구조이고 업데이트가 잦지 않다면 EC2 Auto Scaling만으로 충분할 수 있습니다. 하지만 다음과 같은 상황이라면 EKS가 압도적으로 유리합니다.

1. **마이크로서비스(MSA)**: 수십 개의 서비스를 각각 독립적으로 스케일링해야 할 때.
2. **빠른 배포**: 하루에도 몇 번씩 서비스를 업데이트해야 할 때.
3. **멀티 클라우드**: AWS뿐만 아니라 온프레미스나 타 클라우드에서도 동일한 운영 환경을 유지하고 싶을 때.

---

### 4. 비용 관련 주의사항

* **EKS 제어 평면 요금**: 클러스터당 **시간당 $0.10(월 약 $72)**이 고정적으로 발생합니다.
* **기타**: 워커 노드로 사용되는 EC2 요금은 동일하며, 컨테이너 간 통신이나 로드 밸런서(ALB/NLB) 사용에 따른 비용이 추가될 수 있습니다.
