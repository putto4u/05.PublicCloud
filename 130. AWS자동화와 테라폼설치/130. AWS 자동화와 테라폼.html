테라폼 & AWS
엔터프라이즈 자동화 아키텍처

단순한 스크립팅을 넘어, 상태(State) 기반의 인프라 형상 관리를 마스터하십시오.
확장성, 일관성, 그리고 협업을 위한 클라우드 운영의 새로운 표준입니다.

Paradigm Shift

절차적 명령 vs 선언적 상태

절차적 방식 (Shell Script/Ansible)

"서버 A를 만들어라 → 켜졌으면 IP를 가져와라 → 방화벽에 넣어라" (순서와 과정에 집중)

선언적 방식 (Terraform)

"나는 서버 A와 방화벽 규칙 B가 필요하다." (최종 결과물에 집중)

Terraform이 '현재 상태'와 비교해 부족한 부분만 자동 실행

AWS 자동화를 위한 최적의 선택

엔터프라이즈 환경에서 테라폼이 강력한 3가지 이유

Dependency Graph

테라폼은 리소스 간의 의존성을 자동으로 분석하여 병렬 처리(Parallelism)가 가능한 부분을 동시에 생성합니다.

VPC ← Subnet ← EC2
(EC2 생성 전 VPC 자동 선행)

Immutable Infrastructure

설정을 수정하는 것이 아니라, 교체(Replace)하는 방식입니다. 구성 드리프트(Drift)를 원천 차단합니다.

AMI 업데이트 시:
패치(X) → 새 인스턴스 배포(O)

Reusable Modules

검증된 아키텍처를 모듈(Module)로 캡슐화하여 팀 전체가 레고 블록처럼 재사용할 수 있습니다.

module "vpc" { source = "..." }
단 한 줄로 표준 네트워크 배포

IaC 도구 비교 분석 (vs Alternatives)

기능 Terraform AWS CloudFormation Ansible
주 목적 인프라 프로비저닝 (생성) 인프라 프로비저닝 (생성) 구성 관리 (설정/패치)
지원 범위 멀티 클라우드 (AWS/Azure/GCP) AWS 종속 (Vendor Lock-in) 멀티 플랫폼
언어 (Syntax) HCL (가독성 우수) JSON / YAML (매우 장황함) YAML (간결함)
상태 관리 State File (.tfstate) AWS 내부 관리 Stateless (상태 없음)
* 테라폼은 AWS뿐만 아니라 온프레미스, Kubernetes, SaaS 등 모든 API 기반 서비스를 통합 관리할 수 있는 것이 최대 강점입니다.

실전 코드 미리보기

인프라를 '자산'으로 코딩하다

테라폼 코드는 단순한 설정 파일이 아닙니다. 버전 관리(Git)가 가능하고, 동료와 리뷰할 수 있으며, 재사용 가능한 소프트웨어 자산입니다.

  • 가독성: HCL은 인간이 읽기 쉽도록 설계됨
  • 모듈화: 반복되는 인프라 패턴을 함수처럼 재사용
과금 주의 (Billing Alert)

'terraform apply' 명령어는 즉시 실제 AWS 리소스를 생성하므로 비용이 발생합니다. 실습 후에는 반드시 'terraform destroy'로 리소스를 정리하세요.
