**Amazon S3 정적 웹사이트 호스팅 실습**

이번 실습에서는 Amazon S3(아마존 심플 스토리지 서비스)를 활용하여 서버 인프라 구축 없이 HTML(HyperText Markup Language, 하이퍼텍스트 마크업 언어), CSS(Cascading Style Sheets, 종속형 시트), JS(JavaScript, 자바스크립트) 파일만으로 구성된 웹사이트를 배포하는 과정을 진행합니다.

### 1단계: Amazon S3 버킷 생성

1. AWS(Amazon Web Services, 아마존 웹 서비스) 관리 콘솔에 로그인한 후, **S3** 서비스로 이동하여 **버킷 만들기(Create bucket)**를 클릭합니다.
2. **일반 구성(General configuration)**에서 전 세계적으로 유일한 **버킷 이름(Bucket name)**을 입력합니다. (예: `my-static-web-20260223`)
3. **AWS 리전(AWS Region)**은 접속 사용자와 가장 가까운 리전(예: `ap-northeast-2 (Seoul)`)을 선택합니다.
4. **객체 소유권(Object Ownership)**은 **ACL 비활성화됨(ACLs disabled)**을 선택합니다.[^1]

### 2단계: 퍼블릭 액세스 차단 해제 설정

웹사이트는 외부 인터넷을 통해 누구나 접근할 수 있어야 하므로, S3에 기본적으로 설정된 강력한 보안 차단 기능을 일부 해제해야 합니다.

1. 버킷 생성 페이지의 **이 버킷의 퍼블릭 액세스 차단 설정(Block Public Access settings for this bucket)** 섹션으로 이동합니다.
2. **모든 퍼블릭 액세스 차단(Block all public access)**의 체크 박스를 **해제**합니다.
3. 하단에 나타나는 경고 메시지를 확인하고, **"현재 설정으로 인해 이 버킷과 그 안에 있는 객체가 퍼블릭 상태가 될 수 있음을 인지합니다."**라는 확인란에 체크합니다.
4. 페이지 맨 아래의 **버킷 만들기(Create bucket)**를 클릭하여 생성을 완료합니다.

### 3단계: 정적 웹사이트 호스팅 기능 활성화

1. 생성된 버킷의 이름을 클릭하여 내부로 진입한 뒤, **속성(Properties)** 탭을 선택합니다.
2. 페이지 최하단으로 스크롤하여 **정적 웹사이트 호스팅(Static website hosting)** 섹션의 **편집(Edit)**을 클릭합니다.
3. **정적 웹사이트 호스팅(Static website hosting)**을 **활성화(Enable)**로 변경합니다.
4. **인덱스 문서(Index document)**에 `index.html`을 입력합니다. (오류 문서(Error document)는 선택 사항이나 `error.html` 입력을 권장합니다.)
5. **변경 사항 저장(Save changes)**을 클릭합니다.

### 4단계: 외부 읽기 권한 부여 (버킷 정책 설정)

버킷을 공개 상태로 만들었더라도, 내부의 파일(객체)을 누구나 읽을 수 있도록 명시적인 허용 정책을 작성해야 합니다.

1. 버킷의 **권한(Permissions)** 탭으로 이동합니다.
2. **버킷 정책(Bucket policy)** 섹션의 **편집(Edit)**을 클릭합니다.
3. 아래의 JSON(JavaScript Object Notation, 자바스크립트 객체 표기법) 코드를 복사하여 붙여넣습니다. **단, `Resource` 항목의 버킷 이름은 본인이 생성한 이름으로 반드시 변경해야 합니다.**

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": "*",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::본인의버킷이름/*"
        }
    ]
}

```

4. **변경 사항 저장(Save changes)**을 클릭합니다. 정상적으로 적용되면 버킷 이름 옆에 붉은색의 **'퍼블릭(Public)'** 태그가 표시됩니다.

### 5단계: 웹사이트 파일 업로드 및 접속 테스트

1. 로컬 PC(Personal Computer, 개인용 컴퓨터)에 간단한 내용이 담긴 `index.html` 파일을 생성합니다.
2. 버킷의 **객체(Objects)** 탭으로 이동하여 **업로드(Upload)** 버튼을 누르고 해당 `index.html` 파일을 업로드합니다.
3. 다시 **속성(Properties)** 탭의 **정적 웹사이트 호스팅(Static website hosting)** 섹션으로 이동합니다.
4. 표시된 **버킷 웹사이트 엔드포인트(Bucket website endpoint)** 링크를 복사하여 웹 브라우저 주소창에 붙여넣고 사이트가 정상적으로 열리는지 확인합니다.

---

### 💡 실전 팁 및 자주 오해하거나 실수하는 부분

* **폴더 구조의 오해:** 정적 호스팅 설정 시 지정한 `index.html`은 반드시 버킷의 최상위 루트 경로(Root Directory)에 업로드해야 합니다. 임의의 폴더를 만들고 그 안에 넣으면 엔드포인트 접속 시 파일을 찾을 수 없어 에러가 발생합니다.
* **대소문자 구분:** S3의 파일명은 대소문자를 엄격하게 구분합니다. `Index.html`이나 `INDEX.HTML`로 업로드한 경우 브라우저가 자동으로 인식하지 못하므로, 파일명을 소문자 `index.html`로 통일하는 것이 필수적입니다.
* **객체 암호화(Object Encryption) 설정 확인:** 파일 업로드 시 **기본 암호화(Default encryption)**가 `SSE-KMS`(AWS Key Management Service 기반 서버 측 암호화)로 설정되어 있다면, 브라우저에서 접근 시 `403 Forbidden` 에러가 발생합니다. 퍼블릭 정적 웹사이트용 파일은 반드시 `SSE-S3`(Amazon S3 관리형 키를 사용한 서버 측 암호화) 방식을 사용해야 합니다.

> **⚠️ [과금 주의] 데이터 송신(Data Transfer Out) 요금**
> 웹사이트 파일 자체를 저장하는 스토리지(Storage) 비용 외에도, 방문자가 웹사이트에 접속하여 S3에서 외부 인터넷 망으로 데이터를 다운로드할 때마다 **데이터 송신(Data Transfer Out)** 과금이 발생합니다. 무료 계층(Free Tier)의 경우 월 100GB까지는 무료이나, 이미지가 많거나 트래픽이 집중될 경우 막대한 요금이 청구될 수 있습니다. 실무에서는 비용 절감과 로딩 속도 향상을 위해 Amazon CloudFront(아마존 클라우드프론트 - 콘텐츠 전송 네트워크)를 앞단에 연결하여 캐싱(Caching)하는 것이 표준 아키텍처입니다.

[^1]:
**ACL(Access Control List, 접근 제어 목록)**: 과거에 버킷과 개별 객체에 대한 권한을 세밀하게 관리하던 레거시(Legacy, 구형) 방식입니다. 현재 AWS(Amazon Web Services)는 보안 관리를 단순화하고 중앙 집중화하기 위해 ACL을 비활성화하고 버킷 정책(Bucket Policy)만을 사용하여 전체 권한을 통제하는 것을 강력히 권장하고 있습니다.
