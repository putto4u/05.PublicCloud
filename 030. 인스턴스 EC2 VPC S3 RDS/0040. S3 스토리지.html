S3 객체 스토리지의 이해

  • Bucket(버킷, 바구니): 객체를 저장하는 최상위 컨테이너. 전역적으로 유일한 이름을 가져야 함.
  • Object(객체): 데이터와 메타데이터의 결합체.
  • Key(키): 객체의 고유 식별자. 경로 형태를 띠지만 실제로는 평면 구조.
  • Durability(내구성): 99.999999999% (11-9s) 설계로 데이터 소실 방지.
[객체 스토리지와 블록 스토리지의 구조 차이 이미지]

S3는 API 기반의 HTTP/HTTPS 통신을 사용하여 데이터에 접근합니다.

용도별 스토리지 클래스 전략

클래스 특징 주요 용도 과금 핵심
Standard 자주 사용하는 데이터 웹 사이트 이미지, 실시간 서비스 높은 저장 비용
Standard-IA 가끔 접근 (Infrequent Access) 지난달 보고서, 재해 복구용 인출 비용 발생*
Intelligent-Tiering 패턴 자동 분석 사용 빈도를 모르는 데이터 자동 이동 수수료
Glacier Flexible 장기 보관 (아카이브) 세무 기록, 규정 준수 백업 매우 낮은 저장 비용

유료 과금 주의 사항

IA 및 Glacier 클래스는 저장 비용은 저렴하지만, 데이터를 읽을 때 Retrieval Fee(인출 비용)가 발생하며 최소 보관 기간 미준수 시 위약금이 청구될 수 있습니다.

효율적 관리를 위한 핵심 기능

Versioning (버전 관리)

실수 인한 삭제 복구 가능. 객체의 모든 수정 이력을 보관합니다.

Lifecycle (수명 주기)

특정 기간 후 클래스 변경이나 삭제를 자동화하여 비용을 절감합니다.

Replication (복제)

동일/타 리전으로 데이터를 자동 복제하여 가용성을 확보합니다.

* 주석: 버전 관리를 활성화하면 삭제된 파일도 실제로는 스토리지에 남아있으므로, 삭제 마커(Delete Marker)를 관리하지 않으면 지속적으로 과금될 수 있습니다.

S3 Static Web Hosting (정적 웹 호스팅)

서버리스 웹 서비스의 시작

HTML, CSS, JavaScript와 같은 정적 자산을 별도의 웹 서버(EC2 등) 없이 S3 버킷을 통해 직접 호스팅하는 기능입니다.

  • 고가용성: 서버 다운 우려 없는 안정적 서비스
  • 확장성: 무제한에 가까운 동시 접속 처리
  • 저비용: 사용한 스토리지 및 전송량만큼만 과금

설정 핵심 요소

1. 정적 웹 호스팅 활성화
2. Index Document(인덱스 문서) 지정
3. Error Document(에러 문서) 지정
4. Bucket Policy(버킷 정책) 퍼블릭 읽기 허용

사례 1: Single Page Application (SPA)

React, Vue, Angular 기반의 현대적 웹 앱 배포에 최적화된 방식입니다.

  • CI/CD(지속적 통합/배포) 연동: 빌드된 파일을 S3에 자동 동기화.
  • Client-Side Routing(클라이언트 사이드 라우팅): 404 에러 시 index.html로 리다이렉트 처리.
  • 글로벌 배포: CloudFront와 결합하여 전 세계 초고속 전송.

전송 비용 주의

S3에서 인터넷으로 나가는 Data Transfer Out(데이터 전송 비용)이 주된 과금 요소입니다. 트래픽이 많을 경우 CloudFront(클라우드프론트)를 통한 전송 최적화가 필수입니다.

사례 2: 기술 문서 및 SSG 활용

SSG (정적 사이트 생성기)

Hugo, Jekyll, Docusaurus 등을 활용한 사내 기술 블로그나 제품 사용 설명서 구축.

마케팅 랜딩 페이지

이벤트성 단발 페이지 배포 시 서버 관리 부담 없이 즉시 론칭 가능.

[S3와 Route 53을 연동한 커스텀 도메인 적용 이미지]

S3 보안 아키텍처 실무

  • IAM Policy(식별 및 액세스 관리 정책): 특정 사용자나 역할에 권한 부여.
  • Bucket Policy(버킷 정책): 버킷 단위의 세부 접근 제어.
  • Block Public Access(퍼블릭 액세스 차단): 실수로 인한 데이터 유출 방지 계층.
  • SSE(Server-Side Encryption, 서버 측 암호화): 저장 데이터 자동 암호화.

전문가 팁

정적 호스팅을 위해 버킷을 퍼블릭으로 열 경우, 반드시 민감한 데이터가 포함되지 않았는지 확인하십시오. 중요 데이터는 별도의 비공개 버킷에 관리해야 합니다.

CloudFront 연동 및 HTTPS 보안

S3 단독 호스팅보다 CloudFront(CDN 서비스) 연동이 권장되는 이유:

HTTPS 적용

S3 엔드포인트는 HTTP만 지원하지만, CloudFront를 통해 보안 인증서(SSL/TLS) 적용 가능.

OAC (원본 액세스 제어)

S3 버킷을 비공개로 유지하고 CloudFront를 통해서만 데이터에 접근하도록 강제.

Edge Caching (엣지 캐싱)

전 세계 사용자 근처의 서버에 콘텐츠를 복제하여 응답 속도 극대화.

* OAC(Origin Access Control, 원본 액세스 제어): S3 버킷에 대한 직접적인 외부 접근을 차단하고 CDN을 통한 검증된 요청만 허용하는 최신 보안 방식입니다.
