AWS 허브 앤 스포크(Hub and Spoke) 네트워크 아키텍처 구축 실무 가이드

본 과정은 기업 환경에서 가장 널리 사용되는 엔터프라이즈급 네트워크 구조인 허브 앤 스포크(Hub and Spoke) 아키텍처를 AWS 클라우드 환경에서 직접 설계하고 구축하는 실무 지침서입니다. 중앙 집중식 트래픽 제어, 보안 검사, 그리고 확장성을 고려한 최고 수준의 아키텍처 설계 패턴을 다룹니다.

아키텍처 목표 설계도

[목적] 모든 스포크(Spoke) VPC의 인터넷 향 트래픽을 허브(Hub) VPC로 일원화하여 중앙 통제 및 보안(Centralized Egress)을 달성합니다.

[AWS 허브 앤 스포크 네트워크 논리적 토폴로지 다이어그램 이미지]
[과금 발생 서비스 사전 안내] 주의! 본 실습을 진행하는 동안 아래의 서비스 항목에서 시간당 과금 및 데이터 처리 비용이 발생합니다. 실습 완료 후 반드시 리소스를 삭제(Clean-up)하여 불필요한 비용 청구를 방지하시기 바랍니다.
  • NAT GW (NAT Gateway, 네트워크 주소 변환 게이트웨이): 시간당 요금 및 데이터 처리 요금 부과
  • TGW (Transit Gateway, 전송 게이트웨이): 시간당 연결 요금 및 데이터 처리 요금 부과
  • EIP (Elastic IP, 탄력적 IP 주소): NAT Gateway 연결 시 사용되며, 인스턴스 미연결 시 시간당 과금

사전 필수 용어 정리

  • VPC (Virtual Private Cloud, 가상 사설 클라우드): AWS 클라우드 내에서 논리적으로 격리된 사용자 전용 가상 네트워크 공간입니다.
  • AZ (Availability Zone, 가용 영역): 하나의 리전(Region) 내에 존재하는 물리적으로 분리된 데이터 센터 그룹입니다. 서울 리전의 경우 4개의 AZ(a, b, c, d)를 제공하여 고가용성을 보장합니다.
  • IGW (Internet Gateway, 인터넷 게이트웨이): VPC 내부의 리소스가 외부 인터넷과 통신할 수 있도록 연결해 주는 수평 확장형 논리적 구성 요소입니다.
  • NAT GW (Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이): 프라이빗 서브넷에 있는 리소스가 외부 인터넷으로 나갈 수 있게 해주지만, 외부에서 프라이빗 서브넷으로 직접 접근하는 것은 차단하는 관리형 서비스입니다. [과금 발생 서비스]
  • TGW (Transit Gateway, 전송 게이트웨이): 여러 개의 VPC와 온프레미스 네트워크를 중앙에서 연결해 주는 클라우드 라우터(네트워크 허브) 역할을 합니다. 복잡한 피어링(Peering) 연결을 단순화합니다. [과금 발생 서비스]
  • CIDR (Classless Inter-Domain Routing, 사이더): IP 주소 할당 및 라우팅을 위한 유연한 인터넷 프로토콜 주소 할당 방법입니다. 예: 10.0.0.0/16

Phase 1. 중앙 허브 네트워크 구축: Hub VPC (AZ 1, AZ 3)

가장 먼저 모든 네트워크 트래픽의 중심이 될 Hub VPC를 구축합니다. 이 VPC는 인터넷 게이트웨이(IGW)를 보유하며, 스포크(Spoke) VPC들이 인터넷으로 나갈 때 거쳐가는 중앙 통로 역할을 수행합니다. 고가용성(HA) 설계를 위해 서울 리전의 AZ 1과 AZ 3에 자원을 분산 배치합니다.

아키텍트의 실무 팁: 서브넷 분리 전략

실무 환경에서는 Transit Gateway를 연결할 때 전용 서브넷(TGW Attachment Subnet)을 생성하는 것이 AWS 권장 모범 사례(Best Practice)입니다. 라우팅 테이블 관리가 분리되어 복잡한 트래픽 흐름을 명확하게 제어할 수 있기 때문입니다. TGW 전용 서브넷은 많은 IP를 필요로 하지 않으므로 /28 크기로 충분합니다.

1단계: Hub VPC 생성
  1. AWS 관리 콘솔에 로그인 후, 상단 검색창에 VPC를 입력하고 이동합니다.
  2. 우측 상단의 지역(Region)이 아시아 태평양 (서울) [ap-northeast-2]로 선택되어 있는지 반드시 확인합니다.
  3. 좌측 내비게이션 창에서 VPC를 클릭하고, 우측 상단의 주황색 VPC 생성 버튼을 클릭합니다.
  4. 생성 옵션에서 다음을 선택 및 입력합니다.
    • 생성할 리소스: VPC만 (사용자 지정 서브넷 설정을 위해 VPC만 단독으로 생성합니다.)
    • 이름 태그: vpc_hub
    • IPv4 CIDR 블록: IPv4 CIDR 수동 입력 선택 후 10.0.0.0/16 입력
    • IPv6 CIDR 블록: IPv6 CIDR 블록 없음
    • 테넌시: 기본값 (Default)
  5. 하단의 VPC 생성 버튼을 클릭합니다.
[Hub VPC 생성 완료 화면 및 CIDR 정보 이미지]
2단계: Hub VPC 서브넷 생성 (AZ 1, AZ 3)

요구사항에 따라 서울 리전의 AZ-1(ap-northeast-2a)과 AZ-3(ap-northeast-2c)에 서브넷을 구성합니다. 퍼블릭(인터넷 연결용), 프라이빗(내부용), 그리고 TGW 연결 전용 서브넷을 각각 생성합니다.

  1. 좌측 메뉴에서 서브넷을 클릭하고 서브넷 생성을 클릭합니다.
  2. VPC ID: 앞서 만든 vpc_hub를 선택합니다.
  3. 아래 표의 구성에 맞게 새 서브넷 추가 버튼을 눌러 총 6개의 서브넷을 생성합니다.
서브넷 이름 (이름 태그) 가용 영역 (AZ) IPv4 CIDR 블록 용도 설명
hub-public-sub-2a ap-northeast-2a (AZ-1) 10.0.1.0/24 NAT Gateway 및 외부 인입 로드밸런서 배치용 퍼블릭망
hub-public-sub-2c ap-northeast-2c (AZ-3) 10.0.2.0/24 동일 (고가용성용)
hub-private-sub-2a ap-northeast-2a (AZ-1) 10.0.10.0/24 Hub 내부 보안 인스턴스 배치용 프라이빗망
hub-private-sub-2c ap-northeast-2c (AZ-3) 10.0.20.0/24 동일 (고가용성용)
hub-tgw-attach-sub-2a ap-northeast-2a (AZ-1) 10.0.254.0/28 Transit Gateway 전용 연결 인터페이스(ENI) 배치 서브넷
hub-tgw-attach-sub-2c ap-northeast-2c (AZ-3) 10.0.254.16/28 동일 (고가용성용)

입력이 완료되면 하단의 서브넷 생성을 클릭합니다.

3단계: 인터넷 게이트웨이(IGW) 생성 및 Hub VPC 연결

Hub VPC가 외부 인터넷과 통신하기 위한 관문인 IGW를 만듭니다.

  1. 좌측 메뉴에서 인터넷 게이트웨이를 클릭하고 인터넷 게이트웨이 생성을 누릅니다.
  2. 이름 태그: igw_hub 입력 후 생성합니다.
  3. 생성된 igw_hub를 선택하고 우측 상단의 작업 ▼ 드롭다운에서 VPC에 연결을 클릭합니다.
  4. 사용 가능한 VPC: vpc_hub를 선택하고 연결을 완료합니다.
4단계: NAT 게이트웨이 생성 [과금 발생 서비스]

스포크(Spoke) VPC의 프라이빗 리소스들이 인터넷에 접속하려면 공인 IP로의 주소 변환이 필요합니다. Hub VPC의 퍼블릭 서브넷에 NAT GW를 배치하여 이 역할을 수행하게 합니다.

  1. 좌측 메뉴에서 NAT 게이트웨이를 클릭하고 NAT 게이트웨이 생성을 누릅니다.
  2. 이름: nat_hub_2a
  3. 서브넷: hub-public-sub-2a (반드시 '퍼블릭' 서브넷을 선택해야 합니다.)
  4. 탄력적 IP 할당 ID: 우측의 탄력적 IP 할당 버튼을 클릭하여 새로운 고정 공인 IP(EIP)를 즉시 발급받아 연결합니다.
  5. NAT 게이트웨이 생성을 클릭합니다. (상태가 'Available'로 변경될 때까지 약 2~3분 소요됩니다.)
아키텍처 설계 참고 (단일 NAT vs 다중 NAT)

프로덕션(운영) 환경에서는 완벽한 고가용성을 위해 AZ-3(hub-public-sub-2c)에도 추가로 NAT GW를 생성해야 하지만, 본 실습에서는 비용 최적화를 위해 AZ-1에 단일 NAT GW만 구축하여 전체 트래픽을 처리하도록 구성합니다.

Phase 2. 스포크 네트워크 구축: Spoke VPC (AZ 2, AZ 4)

실제 업무용 서버나 데이터베이스가 위치할 Spoke VPC(vpc_prod)를 생성합니다. 요구사항에 따라 서울 리전의 AZ-2(ap-northeast-2b)와 AZ-4(ap-northeast-2d)에 자원을 배치합니다. 이 VPC는 자체적인 인터넷 게이트웨이(IGW)를 가지지 않고 완벽히 격리됩니다.

1단계: Spoke VPC (vpc_prod) 생성
  1. 좌측 메뉴에서 VPC 클릭 -> VPC 생성 클릭
  2. 생성할 리소스: VPC만
  3. 이름 태그: vpc_prod
  4. IPv4 CIDR 블록: 10.1.0.0/16 (Hub VPC와 IP 대역이 겹치지 않도록 주의해야 합니다.)
  5. VPC 생성 클릭
2단계: Spoke VPC 서브넷 생성 (AZ 2, AZ 4)

프로덕션 워크로드가 실행될 프라이빗 서브넷과 TGW 연결을 위한 전용 서브넷을 생성합니다.

  1. 좌측 메뉴 서브넷 -> 서브넷 생성 클릭
  2. VPC ID: vpc_prod 선택
  3. 아래 표에 맞춰 4개의 서브넷을 추가합니다.
서브넷 이름 (이름 태그) 가용 영역 (AZ) IPv4 CIDR 블록 용도 설명
prod-private-sub-2b ap-northeast-2b (AZ-2) 10.1.10.0/24 실제 업무용(Spoke) 서버가 위치할 완전 격리 프라이빗망
prod-private-sub-2d ap-northeast-2d (AZ-4) 10.1.20.0/24 동일 (고가용성용)
prod-tgw-attach-sub-2b ap-northeast-2b (AZ-2) 10.1.254.0/28 Transit Gateway 연결용 전용 서브넷
prod-tgw-attach-sub-2d ap-northeast-2d (AZ-4) 10.1.254.16/28 동일 (고가용성용)

서브넷 생성을 완료합니다.

[Spoke VPC의 서브넷 생성 완료 화면 이미지 - AZ b와 d 할당 확인]

Phase 3. 클라우드 라우터, Transit Gateway (TGW) 생성 및 연결

격리된 Hub VPC와 Spoke VPC를 네트워크 수준에서 하나로 묶어줄 중앙 라우터인 Transit Gateway를 생성하고 각각의 VPC에 연결(Attachment)합니다.

1단계: Transit Gateway 생성 [과금 발생 서비스]
  1. 좌측 하단 내비게이션 메뉴의 Transit Gateway 섹션에서 Transit Gateway를 클릭합니다.
  2. 우측 상단의 Transit Gateway 생성을 클릭합니다.
  3. 설정 정보 입력:
    • 이름 태그: tgw_main
    • 설명: Central Router for Hub and Spoke
    • Amazon 측 ASN (Autonomous System Number): 64512 (기본값 유지. BGP 라우팅 프로토콜에서 사용되는 고유 식별 번호입니다.)
    • DNS 지원: Enable (활성화)
    • VPN ECMP 지원: Enable (활성화)
    • 기본 라우팅 테이블 연결/전파: Disable (비활성화)로 체크 해제합니다. (보안과 세밀한 라우팅 제어를 위해 라우팅 테이블을 수동으로 구성하는 것이 실무 아키텍처 원칙입니다.)
  4. Transit Gateway 생성을 클릭합니다. 생성까지 약 3~5분이 소요되며 상태가 'Available'이 될 때까지 대기합니다.
2단계: TGW Attachment (연결) 생성 - Hub VPC

TGW의 네트워크 인터페이스(ENI)를 Hub VPC의 지정된 서브넷에 삽입하는 과정입니다.

  1. 좌측 메뉴 Transit Gateway 연결 (Attachments) 클릭 -> Transit Gateway 연결 생성 클릭
  2. 이름 태그: tgw-attach-hub
  3. Transit Gateway ID: 앞서 만든 tgw_main 선택
  4. 연결 유형 (Attachment type): VPC
  5. VPC ID: vpc_hub 선택
  6. 서브넷 ID: 중요! 가용 영역 ap-northeast-2a와 ap-northeast-2c에서, Phase 1에서 생성한 hub-tgw-attach-sub-2a와 hub-tgw-attach-sub-2c 전용 서브넷을 선택합니다. 다른 서브넷이 선택되지 않도록 체크박스를 신중히 확인하세요.
  7. 생성을 클릭합니다.
3단계: TGW Attachment (연결) 생성 - Spoke VPC
  1. 다시 Transit Gateway 연결 생성을 클릭합니다.
  2. 이름 태그: tgw-attach-prod
  3. Transit Gateway ID: tgw_main 선택
  4. 연결 유형: VPC
  5. VPC ID: vpc_prod 선택
  6. 서브넷 ID: 가용 영역 ap-northeast-2b와 ap-northeast-2d에서 prod-tgw-attach-sub-2b, prod-tgw-attach-sub-2d를 각각 선택합니다.
  7. 생성을 클릭합니다.

두 연결의 상태가 모두 'Available'이 될 때까지 기다립니다.

Phase 4. 라우팅 테이블(Route Table) 설계 및 구성 (핵심 단계)

허브 앤 스포크 아키텍처의 가장 중요한 부분입니다. Spoke VPC 내부의 리소스가 "Spoke -> TGW -> Hub VPC(NAT GW) -> IGW -> 인터넷" 순서로 안전하게 통신할 수 있도록 각 지점의 이정표(라우팅 테이블)를 정확히 설정해야 합니다.

트래픽 흐름(Traffic Flow) 분석

  1. [출발] Spoke VPC 내부의 EC2 서버에서 구글(8.8.8.8)로 핑(Ping)을 보냅니다. 목적지 IP는 0.0.0.0/0(모든 외부 IP)에 해당합니다.
  2. [Spoke 라우팅] Spoke 라우팅 테이블은 0.0.0.0/0 트래픽을 TGW로 보냅니다.
  3. [TGW 라우팅] TGW는 0.0.0.0/0 트래픽을 받아서 Hub VPC (tgw-attach-hub)로 전달합니다.
  4. [Hub TGW 서브넷 라우팅] 트래픽이 Hub VPC에 진입했습니다. TGW 전용 서브넷 라우팅 테이블은 외부로 나가는 트래픽(0.0.0.0/0)을 NAT Gateway로 보냅니다.
  5. [Hub 퍼블릭 서브넷 라우팅] NAT Gateway는 사설 IP를 공인 IP로 변환 후, 자신의 라우팅 테이블을 참조하여 트래픽을 IGW로 보냅니다. 비로소 트래픽이 인터넷으로 나갑니다.
  6. [복귀 흐름] 인터넷에서 돌아온 응답 트래픽은 NAT Gateway에 도착합니다. NAT Gateway가 위치한 퍼블릭 서브넷 라우팅 테이블은 목적지가 Spoke VPC (10.1.0.0/16)인 것을 확인하고 트래픽을 다시 TGW로 돌려보냅니다.
1단계: Spoke VPC 라우팅 테이블 구성

Spoke 내의 서버들이 외부로 통신할 때 모든 트래픽을 TGW로 보내도록 설정합니다.

  1. 좌측 메뉴 라우팅 테이블 클릭 -> 라우팅 테이블 생성
  2. 이름: rt_prod_private / VPC: vpc_prod 선택 후 생성
  3. 생성된 라우팅 테이블 선택 후 하단 [서브넷 연결] 탭 -> 서브넷 연결 편집 클릭
  4. Spoke VPC의 4개 서브넷(private 2개, tgw-attach 2개)을 모두 선택하고 연결 저장.
  5. 하단 [라우팅] 탭 -> 라우팅 편집 클릭 -> 라우팅 추가
  6. 대상(Destination): 0.0.0.0/0
  7. 대상(Target): Transit Gateway 선택 후 드롭다운에서 tgw_main 선택
  8. 변경 사항 저장 클릭.
2단계: Transit Gateway 라우팅 테이블 구성

TGW가 중앙에서 트래픽의 방향을 지시할 수 있도록 TGW 전용 라우팅 테이블을 설정합니다.

  1. 좌측 하단 메뉴 Transit Gateway 라우팅 테이블 클릭 -> Transit Gateway 라우팅 테이블 생성
  2. 이름 태그: tgw_rt_main / Transit Gateway ID: tgw_main 선택 후 생성
  3. 생성된 tgw_rt_main 선택 후 하단 [연결(Associations)] 탭 -> 연결 생성 클릭.
    tgw-attach-hub와 tgw-attach-prod를 각각 하나씩 선택하여 총 2개의 연결을 생성합니다.
  4. 하단 [경로(Routes)] 탭 -> 경로 생성(Create route) 클릭
    • 인터넷으로 나가는 경로 추가:
      CIDR: 0.0.0.0/0
      연결(Attachment): tgw-attach-hub (인터넷 관문인 Hub로 보냄)
  5. 다시 경로 생성 클릭
    • Spoke VPC로 찾아가는 경로 추가:
      CIDR: 10.1.0.0/16 (Spoke VPC 대역)
      연결(Attachment): tgw-attach-prod
[TGW 라우팅 테이블의 경로(Routes) 설정 화면 이미지: 0.0.0.0/0 은 Hub로, 10.1.0.0/16은 Spoke로 향하는 화면]
3단계: Hub VPC - TGW 서브넷 라우팅 테이블 구성

Spoke에서 TGW를 타고 Hub VPC로 들어온 트래픽(인터넷 향)을 제일 먼저 맞이하는 곳이 TGW Attach 서브넷입니다. 여기서 트래픽을 NAT Gateway로 밀어 넣어주어야 합니다.

  1. 좌측 메뉴 라우팅 테이블 클릭 -> 라우팅 테이블 생성
  2. 이름: rt_hub_tgw / VPC: vpc_hub 선택 후 생성
  3. [서브넷 연결] 탭 -> 서브넷 연결 편집 -> hub-tgw-attach-sub-2a, hub-tgw-attach-sub-2c 2개 선택 후 저장.
  4. [라우팅] 탭 -> 라우팅 편집 -> 라우팅 추가
  5. 대상: 0.0.0.0/0 / 대상(Target): NAT 게이트웨이 선택 -> nat_hub_2a 선택
  6. 변경 사항 저장 클릭.
4단계: Hub VPC - 퍼블릭 서브넷 라우팅 테이블 구성 (가장 중요)

NAT Gateway가 위치한 퍼블릭 서브넷의 설정입니다. 첫째, NAT에서 나온 트래픽을 IGW로 보내 인터넷으로 나가게 합니다. 둘째, 인터넷에서 처리된 후 돌아오는 응답 트래픽(목적지: Spoke VPC 10.1.0.0/16)을 다시 TGW로 안내하여 Spoke까지 무사히 도착하게 합니다.

  1. 라우팅 테이블 생성
  2. 이름: rt_hub_public / VPC: vpc_hub 선택 후 생성
  3. [서브넷 연결] 탭 -> 서브넷 연결 편집 -> hub-public-sub-2a, hub-public-sub-2c 2개 선택 후 저장.
  4. [라우팅] 탭 -> 라우팅 편집 -> 라우팅 추가 (2개의 라우팅 룰을 추가합니다.)
    • 첫 번째 룰 (외부로): 대상 0.0.0.0/0 -> Target: 인터넷 게이트웨이(IGW) -> igw_hub
    • 두 번째 룰 (내부 Spoke로 회귀): 대상 10.1.0.0/16 -> Target: Transit Gateway -> tgw_main
  5. 변경 사항 저장 클릭.
흔히 하는 실수 주의 (Troubleshooting Point)

많은 수강생들이 퍼블릭 라우팅 테이블에서 10.1.0.0/16 -> TGW 라우팅 룰을 누락합니다. 이 룰이 없으면 외부로 요청은 나가지만, 돌아오는 응답이 퍼블릭 서브넷에서 길을 잃어(Drop) Spoke 인스턴스에서 통신 타임아웃(Timeout)이 발생하게 됩니다.

Phase 5. 아키텍처 검증 및 테스트

완성된 아키텍처를 테스트하기 위해, 완전히 격리된 Spoke VPC(AZ-2)에 가상 서버(EC2)를 하나 띄우고 시스템 관리자(SSM) 기능을 통해 접속한 뒤, 인터넷 통신이 정상적으로 허브를 거쳐 이루어지는지 검증합니다.

1단계: Spoke VPC에 테스트용 EC2 인스턴스 생성
  1. AWS 콘솔 상단 검색창에 EC2 입력 후 이동.
  2. 좌측 메뉴 인스턴스 클릭 -> 우측 상단 인스턴스 시작 클릭.
  3. 이름: spoke-test-server
  4. AMI: Amazon Linux 2023 AMI (무료 티어 적용)
  5. 인스턴스 유형: t2.micro
  6. 키 페어: 키 페어 없이 진행 (권장하지 않음) 선택. (보안 셸 접속 대신 AWS Systems Manager의 Session Manager를 사용하여 웹 브라우저에서 직접 콘솔에 접근할 예정입니다.)
  7. 네트워크 설정:
    • 편집 클릭
    • VPC: vpc_prod (Spoke VPC)
    • 서브넷: prod-private-sub-2b (AZ-2의 프라이빗 서브넷)
    • 퍼블릭 IP 자동 할당: 비활성화(Disable) (내부망이므로 공인 IP가 필요 없습니다.)
    • 방화벽(보안 그룹): 기본 생성 보안 그룹 사용 (또는 새로 생성). 보안을 위해 Inbound 규칙은 비워두고, Outbound 규칙만 모든 트래픽 0.0.0.0/0 허용 상태인지 확인합니다.
  8. 고급 세부 정보 (매우 중요):
    • 스크롤을 내려 IAM 인스턴스 프로파일 항목을 찾습니다.
    • 사전에 생성된 AmazonSSMManagedInstanceCore 정책이 부여된 IAM 역할을 선택합니다. (이 역할이 있어야 EC2가 프라이빗 환경에서도 AWS Session Manager를 통해 통신을 맺을 수 있습니다. 역할을 미리 만들지 않았다면 IAM 콘솔에서 먼저 생성해야 합니다.)
  9. 우측 하단 인스턴스 시작 클릭.
2단계: 연결 및 라우팅 경로 추적 검증
  1. 인스턴스 상태가 실행 중(Running)이고 상태 검사가 완료되면, spoke-test-server를 체크하고 상단의 연결(Connect) 버튼을 클릭합니다.
  2. [Session Manager] 탭을 선택하고 연결 버튼을 클릭합니다. (검은색 웹 터미널 창이 열립니다.)
  3. 터미널에서 구글 퍼블릭 DNS로 핑(Ping) 테스트를 수행합니다.
    sh-5.2$ ping 8.8.8.8 PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data. 64 bytes from 8.8.8.8: icmp_seq=1 ttl=111 time=30.2 ms 64 bytes from 8.8.8.8: icmp_seq=2 ttl=111 time=30.1 ms ...

    응답이 정상적으로 수신된다면, 완벽히 격리된 Spoke VPC 내부망에 위치한 서버가 TGW를 타고 Hub VPC의 NAT Gateway와 IGW를 거쳐 인터넷에 접속하는 엔터프라이즈급 허브 앤 스포크 아키텍처 구축에 성공한 것입니다.

아키텍처 확장 과제 (Advanced)

현 단계에서는 외부로 나가는 인터넷 트래픽(Egress)만 중앙 집중화했습니다. 실무 보안 규정 컴플라이언스를 만족하기 위해서는, Hub VPC에 AWS Network Firewall이나 써드파티 차세대 방화벽(Palo Alto, Fortinet 등) 어플라이언스를 추가로 배치하여 Spoke에서 나가는 모든 패킷을 심층 패킷 분석(DPI) 하도록 아키텍처를 고도화할 수 있습니다.
