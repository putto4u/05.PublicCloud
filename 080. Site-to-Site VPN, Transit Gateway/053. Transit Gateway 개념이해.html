AWS NETWORK ARCHITECTURE SERIES

AWS Transit Gateway (TGW) 전문 과정

전송 게이트웨이를 활용한 엔터프라이즈 하이브리드 클라우드 구축 및 최적화

1. 클라우드 네트워킹의 진화와 도전 과제

전형적인 기업용 클라우드 환경은 보안, 운영, 프로젝트별 요구사항에 따라 수십 개의 VPC (Virtual Private Cloud, 가상 사설 클라우드)를 생성하게 됩니다. 초기에는 이들을 연결하기 위해 1:1 방식인 VPC Peering (가상 사설 클라우드 간 연결)을 사용했습니다.

VPC Peering의 한계

  • Full Mesh (풀 메시, 망형)의 복잡성: 연결해야 할 VPC가 N개일 때 필요한 페어링 수는 N(N-1)/2개로 기하급수적으로 증가합니다.
  • Transitive Routing (전이적 라우팅) 불가: 중간 VPC를 경유지로 사용하여 제3의 VPC로 패킷을 전달할 수 없습니다.

Transit Gateway의 해결책

  • Hub-and-Spoke (허브 앤 스포크, 바퀴살 구조): 중앙 허브를 통한 단순화된 연결 구조를 제공합니다.
  • 중앙 집중식 관리: 모든 트래픽의 흐름을 단일 지점에서 제어하고 모니터링할 수 있습니다.

N:N의 복잡한 연결을 1:N의 효율적인 구조로 전환

VPC 피어링은 연결당 관리가 필요하지만, TGW (Transit Gateway, 전송 게이트웨이)는 중앙에서 모든 연결을 흡수하여 관리 포인트를 하나로 통합합니다.

2. Transit Gateway의 구조적 실체와 연결 접점

TGW는 특정 VPC의 리소스가 아닌 **AWS 리전(Region, 지역) 단위의 독립 서비스**입니다. VPC와 통신하기 위해서는 VPC 내부의 서브넷에 "접점"을 만들어야 합니다.

2.1 VPC 어태치먼트와 ENI의 역할

연결을 시도하면 AWS는 해당 VPC의 서브넷 내부에 ENI (Elastic Network Interface, 탄력적 네트워크 인터페이스)를 생성합니다. 이 인터페이스가 TGW로 향하는 물리적인 통로가 됩니다.

VPC 서브넷 내부에 생성된 TGW 전용 ENI를 통한 트래픽 전송 원리

각 VPC는 리전 단위의 TGW와 연결될 때 서브넷 내부에 생성된 ENI (Elastic Network Interface, 탄력적 네트워크 인터페이스)를 통해 패킷을 주고받습니다.

3. 중앙 집중식 인터넷 연결 (Centralized Egress) 및 패킷 흐름

실무에서는 보안 검사(Security Inspection)와 비용 절감을 위해 모든 VPC의 인터넷 트래픽을 한곳으로 모읍니다. 이를 Centralized Egress (중앙 집중형 이그레스, 중앙 집중식 외부 통신) 아키텍처라고 합니다.

왜 별도의 Egress VPC를 구성해야 하는가?

  • 비용 최적화: 각 VPC마다 NAT Gateway (NAT 게이트웨이, 네트워크 주소 변환 게이트웨이)를 설치하면 시간당 요금과 데이터 처리 비용이 중복 발생합니다. 중앙에서 하나만 운영하면 비용이 크게 절감됩니다.
  • 중앙 집중형 보안 통제: 모든 나가는(Outbound) 트래픽이 하나의 VPC를 거치므로, 이곳에 Network Firewall (네트워크 방화벽)이나 IDS/IPS를 배치하여 통합 모니터링 및 차단이 가능합니다.
  • IP 관리의 단순화: 외부로 나가는 공인 IP를 특정 VPC의 NAT Gateway IP로 고정할 수 있어 화이트리스트 관리에 용이합니다.

중앙 집중식 이그레스 구조에서의 패킷 이동 순서

3.2 패킷의 여정: VPC에서 인터넷까지 (Step-by-Step)

패킷이 Spoke VPC(사용자 영역)에서 출발하여 외부 인터넷에 도달하는 정교한 흐름은 다음과 같습니다.

1 VPC A → TGW: VPC A의 EC2가 인터넷(0.0.0.0/0)으로 요청을 보냅니다. VPC 라우트 테이블은 목적지가 외부이므로 이를 TGW (Transit Gateway, 전송 게이트웨이) 어태치먼트로 전달합니다.

2 TGW → Egress VPC: 패킷을 받은 TGW는 자신의 라우팅 테이블에서 0.0.0.0/0의 Next Hop (다음 홉, 다음 경로)을 확인합니다. 목적지가 Egress VPC 어태치먼트로 지정되어 있으므로 해당 VPC로 패킷을 라우팅합니다.

3 Egress VPC → NAT GW: Egress VPC의 TGW 서브넷 라우트 테이블은 들어온 패킷을 NAT Gateway (NAT 게이트웨이, 네트워크 주소 변환 게이트웨이)로 보냅니다. 여기서 사설 IP가 공인 IP로 변환됩니다.

4 NAT GW → IGW: NAT Gateway가 위치한 퍼블릭 서브넷의 라우트 테이블은 패킷을 IGW (Internet Gateway, 인터넷 게이트웨이)로 보냅니다.

5 IGW → Internet: 최종적으로 인터넷 게이트웨이를 통해 패킷이 공용 인터넷으로 나갑니다.

이 과정에서 각 구간의 Route Table (라우트 테이블, 경로 표) 설정이 정확해야 데이터가 유실되지 않고 흐를 수 있습니다.

4. 2단계 라우팅 계층 (Double-layer Routing)

패킷이 이동하기 위해서는 VPC 내부 라우팅과 TGW 내부 라우팅이 모두 유기적으로 작동해야 합니다.

계층 1 VPC 라우팅 테이블 (Ingress Side)

VPC 내부 자원이 외부로 나갈 때, Next Hop (다음 홉)을 TGW ID(tgw-xxxx)로 지정합니다.

Destination: 0.0.0.0/0 | Target: tgw-0abc123456...

계층 2 Transit Gateway 라우팅 테이블 (Hub Side)

TGW 내부에서 패킷이 어느 어태치먼트로 갈지 결정합니다.

Destination: 0.0.0.0/0 | Attachment: tgw-attach-egress-vpc

5. VPC Peering vs. Transit Gateway 최종 비교

항목 VPC Peering (피어링) Transit Gateway (TGW)
구조 1:1 지점 간 연결 중앙 집중형 허브 (1:N)
전이적 라우팅 지원 안 함 완벽 지원
관리 포인트 연결마다 개별 관리 단일 허브에서 통합 관리
확장성 수십 개 수준의 한계 수천 개 수준으로 무한 확장 가능
비용 연결비 없음 (트래픽 요금만) 시간당 요금 + 트래픽 요금

6. 과금 체계 및 비용 최적화 (유료 서비스)

과금 유의사항

TGW는 고정비(시간당 요금)가 발생하므로, 소규모 환경에서는 VPC Peering이 유리할 수 있습니다.

  • TGW Attachment 시간당 요금: 약 $0.05/hour (연결당)
  • 데이터 처리 요금 (Data Processing): 1GB당 약 $0.02

심화 용어 및 주석

IGW (Internet Gateway, 인터넷 게이트웨이): VPC와 인터넷 간의 양방향 통신을 가능하게 하는 수평 확장형 게이트웨이입니다.

NAT Gateway (Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이): 프라이빗 서브넷의 자원이 외부 인터넷으로 나갈 수 있도록 사설 IP를 공인 IP로 변환해주는 서비스입니다.

Next Hop (다음 홉): 데이터 패킷이 목적지로 가기 위해 거쳐야 하는 네트워크 상의 다음 단계 경로입니다.

ENI (Elastic Network Interface, 탄력적 네트워크 인터페이스): VPC 내의 가상 서버 등에 연결되어 통신을 가능하게 하는 가상 네트워크 카드입니다.

Centralized Egress (중앙 집중식 외부 통신): 여러 네트워크의 인터넷 출구를 하나로 통합하여 보안과 비용을 최적화하는 아키텍처 패턴입니다.

Propagation (전파): 연결된 네트워크의 경로 정보를 라우팅 테이블에 자동으로 등록하는 기능입니다.
