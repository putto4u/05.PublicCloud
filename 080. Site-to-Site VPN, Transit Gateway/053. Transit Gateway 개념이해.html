AWS NETWORK ARCHITECTURE SERIES

AWS Transit Gateway (TGW) 전문 과정

전송 게이트웨이를 활용한 엔터프라이즈 하이브리드 클라우드 구축 및 최적화

1. 클라우드 네트워킹의 진화와 도전 과제

전형적인 기업용 클라우드 환경은 보안, 운영, 프로젝트별 요구사항에 따라 수십 개의 VPC (Virtual Private Cloud, 가상 사설 클라우드)를 생성하게 됩니다. 초기에는 이들을 연결하기 위해 1:1 방식인 VPC Peering (가상 사설 클라우드 간 연결)을 사용했습니다.

VPC Peering의 한계

  • Full Mesh (풀 메시, 망형)의 복잡성: 연결해야 할 VPC가 N개일 때 필요한 페어링 수는 N(N-1)/2개로 기하급수적으로 증가합니다.
  • Transitive Routing (전이적 라우팅) 불가: 중간 VPC를 경유지로 사용하여 제3의 VPC로 패킷을 전달할 수 없습니다.

Transit Gateway의 해결책

  • Hub-and-Spoke (허브 앤 스포크, 바퀴살 구조): 중앙 허브를 통한 단순화된 연결 구조를 제공합니다.
  • 중앙 집중식 관리: 모든 트래픽의 흐름을 단일 지점에서 제어하고 모니터링할 수 있습니다.

N:N의 복잡한 연결을 1:N의 효율적인 구조로 전환

2. Transit Gateway의 구조적 실체와 연결 접점

전문가 수준에서 가장 혼동하기 쉬운 지점은 TGW의 위치입니다. TGW는 **특정 VPC의 리소스가 아닙니다.** AWS 리전 내에 존재하는 독립적인 서비스 엔터티이며, VPC와 통신하기 위해 VPC 내부 서브넷에 연결 지점(Contact Point)을 만듭니다.

2.1 VPC 어태치먼트와 ENI의 역할

VPC가 TGW에 "연결(Attachment)"될 때, AWS는 해당 VPC의 각 Availability Zone (AZ, 가용 영역)에서 지정된 서브넷에 Elastic Network Interface (ENI, 탄력적 네트워크 인터페이스)를 생성합니다. 이 ENI가 바로 VPC 내부 트래픽이 TGW 허브로 들어가는 물리적인 입구가 됩니다.

VPC 서브넷 내부에 생성된 TGW 전용 ENI를 통한 트래픽 전송 원리

3. 중앙 집중식 인터넷 연결 (Centralized Egress) 실전 예제

모든 VPC에 각각 NAT Gateway (NAT 게이트웨이, 네트워크 주소 변환 게이트웨이)를 설치하면 비용이 크게 발생합니다. 대신, 하나의 공통 VPC(Egress VPC)를 만들고 TGW를 통해 모든 VPC의 인터넷 트래픽을 한곳으로 모아 처리할 수 있습니다.

3.1 아키텍처 시나리오: 3개의 VPC 통합 인터넷 연결

VPC A, B, C는 프라이빗 서브넷으로만 구성되어 있으며, 외부 인터넷으로 나가기 위한 모든 트래픽을 중앙의 Egress VPC (이그레스 VPC, 외부 통신 전용 가상 사설 클라우드)로 보냅니다.

3-VPC 중앙 집중형 이그레스 아키텍처 흐름도

3.2 트래픽 흐름 상세 설명

1단계: Spoke VPC의 판단

VPC A의 인스턴스가 인터넷(0.0.0.0/0)으로 패킷을 던지면, VPC 라우트 테이블은 이 패킷을 TGW로 보냅니다.

2단계: TGW의 중계

패킷을 받은 TGW는 자신의 라우트 테이블을 확인합니다. 0.0.0.0/0에 대한 목적지가 Egress VPC 어태치먼트로 설정되어 있으므로 패킷을 그쪽으로 넘깁니다.

3단계: Egress VPC의 처리

패킷은 Egress VPC의 서브넷에 도달합니다. 이곳의 라우트 테이블은 인터넷 트래픽을 NAT Gateway로 보내고, 최종적으로 Internet Gateway (인터넷 게이트웨이)를 통해 외부로 나갑니다.

비용 절감 효과: 각 VPC마다 NAT Gateway를 둘 경우 월 수백 달러가 발생하지만, 이 구조를 사용하면 단 1개의 NAT Gateway 비용으로 3개(혹은 그 이상)의 VPC가 외부와 통신할 수 있습니다.

4. 2단계 라우팅 계층 (Double-layer Routing)

패킷이 목적지에 도달하기 위해서는 VPC 내부 라우팅과 TGW 내부 라우팅이 모두 올바르게 설정되어야 합니다.

계층 1 VPC 라우팅 테이블 (Ingress Side)

VPC 내부의 자원(EC2 등)이 다른 네트워크로 패킷을 보낼 때, 대상 IP 대역의 Next Hop (다음 홉)을 TGW ID로 지정해야 합니다.

Destination: 10.2.0.0/16 (상대 VPC) | Target: tgw-0abc123456789...

계층 2 Transit Gateway 라우팅 테이블 (Hub Side)

TGW에 도달한 패킷은 TGW 자체의 라우팅 테이블을 조회합니다. 여기서 해당 IP 대역이 어떤 어태치먼트(VPC B, VPN 등)와 연결되어 있는지 확인하고 전달합니다.

Destination: 10.2.0.0/16 | Attachment: vpc-b-attachment-id

5. VPC Peering vs. Transit Gateway 최종 비교

항목 VPC Peering (피어링) Transit Gateway (TGW)
연결 방식 1:1 Point-to-Point (지점 간 연결) 1:N Hub-and-Spoke (중앙 집중형)
전이적 라우팅 지원 안 함 (A-B, B-C 연결 시 A-C 통신 불가) 지원 (A-TGW-C 통신 완벽 지원)
관리 복잡도 VPC 개수 증가 시 기하급수적으로 복잡해짐 중앙에서 단일 지점 관리 가능
확장성 리전당 페어링 수 제한(보통 50~125개) 수천 개의 VPC 연결 가능
대역폭 제한 없음 (인스턴스 성능에 좌우) VPC 어태치먼트당 기본 50Gbps 지원

6. 과금 체계 및 비용 시뮬레이션

유료 서비스 주의사항

Transit Gateway는 VPC 피어링과 달리 고정적인 시간당 점유 요금이 발생하므로 설계 단계에서 비용 효율성을 반드시 고려해야 합니다.

  • AWS Transit Gateway Attachment 시간당 요금: 약 $0.05/hour (서울 리전 기준, 각 VPC 연결당 부과)
  • 데이터 처리 요금 (Data Processing): 전송되는 데이터 1GB당 약 $0.02

비용 산정 시나리오

조건: 5개의 VPC를 TGW에 한 달(730시간) 동안 연결하고, 월간 총 500GB의 데이터를 처리할 경우

1. 연결 점유료 (Hourly): 5개 × $0.05 × 730시간 $182.50
2. 데이터 처리료 (Traffic): 500GB × $0.02 $10.00
월 예상 총 비용 $192.50

심화 용어 및 주석

Elastic Network Interface (ENI, 탄력적 네트워크 인터페이스): VPC 내부에서 IP 주소를 가지는 가상 네트워크 카드입니다. TGW 연결 시 서브넷에 생성되어 통로 역할을 합니다.

NAT Gateway (Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이): 프라이빗 서브넷의 자원이 외부 인터넷으로 나갈 수 있게 도와주는 서비스입니다.

Internet Gateway (IGW, 인터넷 게이트웨이): VPC와 인터넷 간의 통신을 가능하게 하는 수평 확장형 가용성이 높은 게이트웨이입니다.

Next Hop (다음 홉): 데이터 패킷이 목적지로 가기 위해 거쳐야 하는 네트워크 상의 다음 단계 게이트웨이입니다.

Transitive Routing (전이적 라우팅, 중간 거점 경유 라우팅): 중간 네트워크 장치를 거쳐 제3의 네트워크로 트래픽을 전달하는 기능입니다.

Egress (이그레스, 외부 통신): 네트워크 내부에서 외부(인터넷 등)로 나가는 트래픽을 의미합니다.
