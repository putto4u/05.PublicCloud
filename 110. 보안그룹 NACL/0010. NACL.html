AWS NACL 완벽 이해 및 실무 활용

Network Access Control List (네트워크 접근 제어 목록)

💰

과금 안내 (Billing Information)

NACL 기능 자체는 무료(추가 요금 없음)입니다. VPC (Virtual Private Cloud, 가상 사설 클라우드)를 생성할 때 기본적으로 제공되며, 규칙을 추가하거나 수정하는 데 비용이 발생하지 않습니다. 단, NACL을 통과하는 트래픽이 NAT 게이트웨이 등 유료 데이터 전송 서비스를 거칠 경우 해당 서비스에 대한 데이터 처리 요금은 발생할 수 있습니다.

1. 개념 이해: NACL이란 무엇인가?

NACL은 Subnet (서브넷, VPC 내의 논리적인 네트워크 분할 영역) 수준에서 인바운드 및 아웃바운드 트래픽을 제어하는 Stateless (상태 비저장) 기반의 핵심 네트워크 방화벽입니다.

AWS 클라우드 인프라스트럭처에서 NACL은 서브넷 단위로 들어오고 나가는 트래픽을 제어하는 방화벽 역할을 합니다. 쉽게 비유하자면, SG (Security Group, 보안 그룹)가 개별 집(인스턴스)의 현관문 자물쇠라면, NACL은 아파트 단지(서브넷)의 정문 경비실과 같습니다.

2. 핵심 특성: Stateless (상태 비저장)

NACL을 이해하는 데 있어 가장 중요한 핵심은 Stateless (상태 비저장)이라는 점입니다. 이는 트래픽의 왕복(요청과 응답) 상태를 기억하지 않는다는 의미입니다.

Stateful (상태 저장) - 예: 보안 그룹

사용자가 웹 서버에 80포트로 요청을 보냈다면, 방화벽이 이 '상태'를 기억합니다. 따라서 서버가 사용자에게 응답을 돌려보낼 때(Outbound) 별도의 허용 규칙이 없어도 응답 트래픽이 통과됩니다.

Stateless (상태 비저장) - 예: NACL

요청이 들어온(Inbound) 사실을 기억하지 못합니다. 따라서 들어온 요청에 대한 응답을 밖으로 내보내려면(Outbound) 반드시 아웃바운드 규칙에 해당 트래픽을 허용하는 명시적인 규칙이 존재해야만 합니다.

NACL vs 보안 그룹 (Security Group) 비교

구분 NACL (Network Access Control List) SG (Security Group, 보안 그룹)
적용 범위 서브넷 (Subnet) 단위 인스턴스 (ENI - 탄력적 네트워크 인터페이스) 단위
상태 저장 여부 Stateless (상태 비저장): 반환 트래픽 명시적 허용 필요 Stateful (상태 저장): 반환 트래픽 자동 허용
규칙 평가 방식 규칙 번호 순서대로 평가 (낮은 번호 우선) 모든 규칙 평가 후 가장 관대한 규칙 적용
허용/거부 Allow (허용) 및 Deny (거부) 모두 가능 Allow (허용) 규칙만 작성 가능
기본 설정 기본 NACL: 모두 허용 / 사용자 생성: 모두 거부 인바운드: 모두 거부 / 아웃바운드: 모두 허용

⚠️ 아키텍트의 경고: 너무 복잡한 NACL은 독이 됩니다.

실무에서 흔히 하는 실수 중 하나는 보안 그룹(SG)에서 해야 할 세밀한 포트 제어를 NACL에서 하려는 것입니다. NACL은 서브넷 전체에 적용되므로 관리 복잡도가 급격히 상승합니다. NACL은 블랙리스트(특정 악의적 IP 차단) 형태나, 서브넷 간의 굵직한 라우팅 격리용으로만 사용하고, 세부적인 애플리케이션 포트 제어는 보안 그룹에 위임하는 것이 베스트 프랙티스입니다.

3. NACL 구성 요소와 작동 원리

NACL은 낮은 규칙 번호부터 순차적으로 평가되며, 원활한 양방향 통신을 위해 임시 포트(Ephemeral Ports)의 개념을 반드시 이해하고 아웃바운드 규칙에 적용해야 합니다.

1. Rule Number (규칙 번호)

NACL의 규칙은 1부터 32766 사이의 번호를 가집니다. 트래픽이 들어오거나 나갈 때, 가장 낮은 번호의 규칙부터 순차적으로 평가됩니다. 트래픽이 특정 규칙에 일치하면, 그 이후의 규칙은 평가하지 않고 즉시 허용(Allow) 또는 거부(Deny)를 결정합니다.

💡 실전 팁: 규칙 번호 할당 전략

규칙 번호를 1, 2, 3과 같이 연속으로 부여하지 마십시오. 나중에 규칙 사이에 새로운 규칙을 삽입해야 할 때 번호를 모두 밀어내야 하는 대참사가 발생합니다. 실무에서는 보통 100 단위 (100, 200, 300...) 또는 최소 10 단위로 여유를 두고 규칙 번호를 할당합니다.

2. Ephemeral Ports (임시 포트)의 이해

클라이언트가 서버에 연결할 때, 클라이언트는 1024 이상의 무작위 포트 번호를 자신의 통신 포트로 엽니다. 이것을 임시 포트라고 부릅니다.

NACL은 Stateless(상태 비저장)이므로, 웹 서버가 응답을 관리자 PC로 돌려보낼 때 아웃바운드 규칙에서 대상 포트(클라이언트의 임시 포트)를 허용해주어야 합니다.

💻 운영체제별 임시 포트 범위

  • Windows Server 2008 이상: 49152 - 65535
  • Linux (Amazon Linux, Ubuntu): 32768 - 60999
  • Elastic Load Balancer (ELB): 1024 - 65535
  • 실무 권장: 1024 - 65535 전체 개방

4. 기본 웹 서버 서브넷 NACL 설정

일반적인 퍼블릭 서브넷(웹 서버 위치)에 적용되는 기본적인 NACL 규칙 예시입니다. 인터넷으로부터의 HTTP/HTTPS 트래픽과 SSH 관리 트래픽을 허용합니다.

110HTTPS (TCP)4430.0.0.0/0Allow

5. 실무 사례 (Real-world Use Cases)

사례 1: 특정 악성 IP 차단 (Blacklist)

보안 그룹(SG)은 '거부' 규칙을 만들 수 없습니다. 특정 IP 대역에서 DDoS 공격 등이 발생할 경우 NACL의 낮은 번호 규칙으로 Deny를 설정하여 차단합니다.

# Inbound NACL
50 | ALL Traffic | 198.51.100.0/24 | DENY (공격자 차단)
100 | HTTP (80) | 0.0.0.0/0 | ALLOW (정상 트래픽)

사례 2: 3-Tier 아키텍처의 엄격한 격리

데이터베이스(Private Subnet)는 오직 WAS 서브넷하고만 통신해야 하며, 웹 서버(Web Subnet)의 직접 접근은 차단해야 합니다.

사례 3: Bastion Host (배스천 호스트) 통제

내부망 접속용 서버인 Bastion Host는 회사 VPN이나 특정 IP에서만 접근 가능하도록 극도로 제한해야 합니다.

🚀 최신 트렌드: AWS Systems Manager (SSM) Session Manager

최근에는 22번 포트를 아예 열지 않고도 인스턴스에 접속할 수 있는 SSM을 사용하는 것이 보안상 더 권장됩니다.

6. 자주 하는 실수 및 트러블슈팅 가이드

1. Stateless 특성 망각 (Timeout 오류)

인바운드는 열었는데 아웃바운드의 임시 포트(1024-65535)를 열지 않아 발생하는 접속 무한 대기 현상입니다.

✔ 해결책: 항상 양방향(Inbound/Outbound) 규칙 쌍을 설계하십시오.

2. 규칙 우선순위 역전

전체 허용(0.0.0.0/0)을 100번에 두고, 차단 규칙을 200번에 둔 경우입니다. 100번에서 평가가 끝나버려 차단이 작동하지 않습니다.

✔ 해결책: 구체적인 규칙일수록 낮은 번호에 배치하십시오.

3. 내부 서브넷 간 통신 차단

커스텀 NACL을 만들면서 VPC 내부 대역(예: 10.0.0.0/16) 간의 허용 규칙을 누락하여 DB 통신이 끊기는 경우입니다.

✔ 해결책: VPC CIDR 대역에 대한 허용 규칙이 있는지 반드시 점검하십시오.
