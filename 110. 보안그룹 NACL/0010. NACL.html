AWS NACL 완벽 이해 및 실무 활용

💰
과금 안내: NACL (Network Access Control List, 네트워크 접근 제어 목록) 기능 자체는 무료(추가 요금 없음)입니다. VPC (Virtual Private Cloud, 가상 사설 클라우드)를 생성할 때 기본적으로 제공되며, 규칙을 추가하거나 수정하는 데 비용이 발생하지 않습니다. 단, NACL을 통과하는 트래픽이 NAT 게이트웨이 등 유료 데이터 전송 서비스를 거칠 경우 해당 서비스에 대한 데이터 처리 요금은 발생할 수 있습니다.

개념 이해: NACL이란 무엇인가?

NACL은 서브넷(Subnet) 수준에서 인바운드 및 아웃바운드 트래픽을 제어하는 상태 비저장(Stateless) 기반의 핵심 네트워크 방화벽입니다.

AWS 클라우드 인프라스트럭처에서 NACL (Network Access Control List, 네트워크 접근 제어 목록)은 Subnet (서브넷, VPC 내의 논리적인 네트워크 분할 영역) 단위로 들어오고 나가는 트래픽을 제어하는 방화벽 역할을 합니다. 쉽게 비유하자면, SG (Security Group, 보안 그룹)가 개별 집(인스턴스)의 현관문 자물쇠라면, NACL은 아파트 단지(서브넷)의 정문 경비실과 같습니다.

핵심 특성: Stateless (상태 비저장)

NACL을 이해하는 데 있어 가장 중요한 핵심은 Stateless (상태 비저장)이라는 점입니다. 이는 트래픽의 왕복(요청과 응답) 상태를 기억하지 않는다는 의미입니다.

* 주석: Stateless (상태 비저장) vs Stateful (상태 저장)
- Stateful (예: 보안 그룹): 사용자가 웹 서버에 80포트로 요청을 보냈다면, 방화벽이 이 '상태'를 기억합니다. 따라서 서버가 사용자에게 응답을 돌려보낼 때(Outbound) 별도의 허용 규칙이 없어도 응답 트래픽이 통과됩니다.
- Stateless (예: NACL): 요청이 들어온(Inbound) 사실을 기억하지 못합니다. 따라서 들어온 요청에 대한 응답을 밖으로 내보내려면(Outbound) 반드시 아웃바운드 규칙에 해당 트래픽을 허용하는 명시적인 규칙이 존재해야만 합니다.

NACL과 보안 그룹(Security Group)의 비교

보안 아키텍처를 설계할 때 NACL과 보안 그룹을 적절히 혼합하여 심층 방어(Defense in Depth)를 구현해야 합니다. 두 기능의 차이를 명확히 인지하는 것이 중요합니다.

구분 NACL (Network Access Control List) SG (Security Group, 보안 그룹)
적용 범위 서브넷 (Subnet) 단위 인스턴스 (ENI - 탄력적 네트워크 인터페이스) 단위
상태 저장 여부 Stateless (상태 비저장): 반환 트래픽을 명시적으로 허용해야 함 Stateful (상태 저장): 인바운드 허용 시 반환 트래픽 자동 허용
규칙 평가 방식 규칙 번호 순서대로 평가 (가장 낮은 번호부터 우선 적용) 모든 규칙을 평가하여 가장 관대한 규칙 적용
허용/거부 Allow (허용) 및 Deny (거부) 규칙 모두 작성 가능 Allow (허용) 규칙만 작성 가능 (기본적으로 모두 거부)
기본 설정 기본 NACL은 모든 트래픽 허용 / 사용자 지정 생성 시 모두 거부 기본 보안 그룹은 인바운드 모두 거부, 아웃바운드 모두 허용
아키텍트의 경고: 너무 복잡한 NACL은 독이 됩니다.

실무에서 흔히 하는 실수 중 하나는 보안 그룹(SG)에서 해야 할 세밀한 포트 제어를 NACL에서 하려는 것입니다. NACL은 서브넷 전체에 적용되므로 관리 복잡도가 급격히 상승합니다. NACL은 블랙리스트(특정 악의적 IP 차단) 형태나, 서브넷 간의 굵직한 라우팅 격리용으로만 사용하고, 세부적인 애플리케이션 포트 제어는 보안 그룹에 위임하는 것이 베스트 프랙티스입니다.

NACL 구성 요소와 작동 원리

NACL은 낮은 규칙 번호부터 순차적으로 평가되며, 원활한 양방향 통신을 위해 임시 포트(Ephemeral Ports)의 개념을 반드시 이해하고 아웃바운드 규칙에 적용해야 합니다.

1. Rule Number (규칙 번호)

NACL의 규칙은 1부터 32766 사이의 번호를 가집니다. 트래픽이 들어오거나 나갈 때, 가장 낮은 번호의 규칙부터 순차적으로 평가됩니다. 트래픽이 특정 규칙에 일치하면, 그 이후의 규칙은 평가하지 않고 즉시 허용(Allow) 또는 거부(Deny)를 결정합니다.

실전 팁: 규칙 번호 할당 전략

규칙 번호를 1, 2, 3과 같이 연속으로 부여하지 마십시오. 나중에 규칙 사이에 새로운 규칙을 삽입해야 할 때 번호를 모두 밀어내야 하는 대참사가 발생합니다. 실무에서는 보통 100 단위 (100, 200, 300...) 또는 최소 10 단위로 여유를 두고 규칙 번호를 할당합니다.

2. Ephemeral Ports (임시 포트)의 이해

NACL 설정 시 수강생과 주니어 엔지니어들이 가장 많이 겪는 장애 원인이 바로 Ephemeral Ports (임시 포트)입니다. 클라이언트가 서버에 연결할 때, 클라이언트는 1024 이상의 무작위 포트 번호를 자신의 통신 포트로 엽니다.

예를 들어, 관리자가 집에서 VPC 내의 웹 서버(80번 포트)에 접속할 때, 관리자의 PC는 임의의 포트(예: 54321)를 열고 웹 서버의 80번 포트로 요청을 보냅니다. NACL은 Stateless(상태 비저장)이므로, 웹 서버가 응답을 관리자 PC로 돌려보낼 때 아웃바운드 규칙에서 대상 포트 54321을 허용해주어야 합니다. 이를 위해 클라이언트 운영체제에 맞는 임시 포트 범위(보통 1024-65535)를 아웃바운드 규칙에 허용해야 통신이 성립됩니다.

* 운영체제별 임시 포트 범위:
- Windows Server 2008 이상, Windows 10/11: 49152 - 65535
- Amazon Linux, Ubuntu 및 대부분의 최신 리눅스: 32768 - 60999
- Elastic Load Balancer (ELB): 1024 - 65535
* 실무에서는 호환성을 위해 보통 1024 - 65535 범위를 전체 개방하는 경우가 많습니다.

간단한 예: 기본 웹 서버 서브넷 NACL 설정

일반적인 웹 서비스 환경에서는 HTTP/HTTPS 포트와 관리용 SSH 포트를 개방하며, 서버 응답을 위한 임시 포트(1024-65535) 아웃바운드 허용이 필수적으로 동반되어야 합니다.

일반적인 퍼블릭 서브넷(웹 서버 위치)에 적용되는 기본적인 NACL 규칙 예시입니다. 인터넷으로부터의 HTTP/HTTPS 트래픽과 SSH 관리 트래픽을 허용합니다.

인바운드 규칙 (Inbound Rules)

규칙 번호 유형 프로토콜 포트 범위 소스 (Source) 허용/거부
100 HTTP TCP 80 0.0.0.0/0 (인터넷 전체) Allow
110 HTTPS TCP 443 0.0.0.0/0 Allow
120 SSH TCP 22 203.0.113.50/32 (회사 오피스 IP) Allow
130 사용자 지정 TCP TCP 1024 - 65535 0.0.0.0/0 Allow
* 모든 트래픽 모든 프로토콜 전체 0.0.0.0/0 Deny (기본값)

※ 규칙 번호 130번은 내부 웹 서버가 외부 API나 패치 서버에 요청을 보낸 후 응답을 받을 때 필요한 임시 포트 인바운드 허용입니다.

아웃바운드 규칙 (Outbound Rules)

규칙 번호 유형 프로토콜 포트 범위 대상 (Destination) 허용/거부
100 HTTP TCP 80 0.0.0.0/0 Allow
110 HTTPS TCP 443 0.0.0.0/0 Allow
120 사용자 지정 TCP TCP 1024 - 65535 0.0.0.0/0 Allow
* 모든 트래픽 모든 프로토콜 전체 0.0.0.0/0 Deny (기본값)

※ 규칙 번호 120번은 외부 클라이언트가 80/443 포트로 접속한 후, 서버가 응답을 돌려보낼 때 클라이언트의 임시 포트를 향해 나가는 아웃바운드 트래픽을 허용합니다.

실무 사례 (Real-world Use Cases)

실제 클라우드 엔터프라이즈 환경에서는 악성 IP의 원천 차단(블랙리스트), 데이터베이스 서브넷의 엄격한 격리, 배스천 호스트(Bastion Host)의 접근 통제 등에 NACL을 심층 방어 용도로 활용합니다.

실무 사례 1: 특정 악성 IP 차단 (블랙리스트 기반 방어)

보안 그룹(SG)은 기본적으로 'Allow(허용)' 규칙만 생성할 수 있습니다. 특정 IP 대역에서 DDoS (Distributed Denial of Service, 분산 서비스 거부) 공격이나 악의적인 스캐닝 시도가 탐지되었을 때, 이를 원천적으로 서브넷 단에서 드롭(Drop)시키기 위해 NACL의 Deny(거부) 규칙을 활용합니다.

[Inbound NACL 설정]
규칙 번호: 50
유형: 모든 트래픽
소스: 198.51.100.0/24 (공격자 IP 대역)
작업: Deny

규칙 번호: 100
유형: HTTP
소스: 0.0.0.0/0
작업: Allow

해설: 규칙 번호 50번이 100번보다 먼저 평가되므로, 정상적인 인터넷 트래픽(0.0.0.0/0)은 100번에 의해 허용되지만, 악성 IP 대역에서 오는 트래픽은 50번에 걸려 서브넷에 진입조차 하지 못하고 차단됩니다.

실무 사례 2: 3-Tier 아키텍처의 엄격한 네트워크 격리

기업의 핵심 데이터를 보관하는 데이터베이스 서브넷(Private Subnet)은 외부 인터넷은 물론, 프론트엔드 웹 서버 서브넷으로부터의 직접 접근도 철저히 차단되어야 합니다. 데이터베이스는 오직 WAS (Web Application Server, 웹 애플리케이션 서버) 서브넷하고만 통신해야 합니다.

DB 서브넷 인바운드 NACL 설정 (MySQL 기준):

규칙 번호 유형 포트 소스 (Source) 허용/거부 설명
100 MySQL/Aurora 3306 10.0.2.0/24 (WAS 서브넷 CIDR) Allow WAS 서버의 접근만 허용
900 모든 트래픽 전체 10.0.1.0/24 (WEB 서브넷 CIDR) Deny 웹 서버 서브넷에서의 직접 접근 원천 차단
* 모든 트래픽 전체 0.0.0.0/0 Deny 기타 모든 트래픽 차단

실무 사례 3: Bastion Host (배스천 호스트) 서브넷의 접근 통제

프라이빗 서브넷에 있는 서버들을 관리하기 위해 퍼블릭 서브넷에 Bastion Host (배스천 호스트, 내부망 접속을 위한 징검다리 서버)를 둡니다. 이 배스천 호스트가 위치한 서브넷의 NACL은 회사 본사의 VPN IP 또는 특정 관리자 IP에서만 접근할 수 있도록 극도로 제한해야 합니다.

[Bastion Subnet Inbound]
100 | SSH (22) | 203.0.113.0/28 (회사 지정 IP 대역) | Allow
* | 모든 트래픽 | 0.0.0.0/0 | Deny

[Bastion Subnet Outbound]
100 | SSH (22) | 10.0.0.0/16 (VPC 전체 프라이빗 대역) | Allow
110 | 사용자 지정 TCP (1024-65535) | 203.0.113.0/28 (회사 지정 IP 대역 반환 트래픽) | Allow
* | 모든 트래픽 | 0.0.0.0/0 | Deny
아키텍트의 조언:

최근 실무에서는 보안 강화를 위해 Bastion Host의 22번 포트를 퍼블릭하게 열어두는 대신, AWS Systems Manager (SSM) Session Manager를 사용하는 추세입니다. SSM을 사용하면 인바운드 포트(22번)를 전혀 열지 않고도 프라이빗 인스턴스에 안전하게 접속할 수 있습니다.

자주 하는 실수 및 트러블슈팅 가이드

NACL 적용 시 가장 흔히 발생하는 장애는 상태 비저장(Stateless) 특성 망각으로 인한 반환 트래픽 누락과, 규칙 번호의 우선순위 역전으로 인한 의도치 않은 트래픽 차단입니다.

현업 엔지니어들이 NACL을 다루며 가장 자주 겪는 장애 패턴들입니다.

  1. 상태 비저장(Stateless) 간과로 인한 통신 단절 (Timeout 오류):

    가장 빈번한 실수입니다. 인바운드에서 웹 트래픽(80)을 허용해 놓고, 아웃바운드에서 클라이언트로 돌아갈 임시 포트(1024-65535)를 열어주지 않아 접속이 무한 대기(Timeout) 상태에 빠지는 현상입니다. 해결책: 항상 양방향(Inbound/Outbound) 규칙 쌍을 고려하여 설계하십시오.

  2. 규칙 우선순위 역전:

    Deny 규칙을 200번에 넣고, 전체 허용(0.0.0.0/0 Allow)을 100번에 넣은 경우입니다. NACL은 100번에서 트래픽을 허용해버리고 평가를 종료하므로, 200번의 차단 규칙은 영원히 실행되지 않습니다. 해결책: 구체적인 규칙(차단 또는 특정 IP)일수록 번호를 낮게, 포괄적인 규칙일수록 번호를 높게 배치해야 합니다.

  3. 내부 서브넷 간 통신 차단:

    NACL을 기본 상태(모두 허용)에서 Custom 상태로 변경하면서, VPC 내부 대역(예: 10.0.0.0/16) 간의 통신을 명시적으로 허용하는 규칙을 누락하는 경우입니다. 이로 인해 동일 VPC 내의 웹 서버와 DB 서버 간 통신이 끊어집니다. 해결책: NACL을 강화할 때는 항상 VPC CIDR에 대한 Inbound/Outbound 허용 규칙이 포함되어 있는지 점검하십시오.
