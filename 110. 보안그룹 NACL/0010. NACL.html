AWS NACL 완벽 이해 및 실무 활용

Network Access Control List (네트워크 접근 제어 목록)

💰

과금 안내 (Billing Information)

NACL 기능 자체는 무료(추가 요금 없음)입니다. VPC (Virtual Private Cloud, 가상 사설 클라우드)를 생성할 때 기본적으로 제공되며, 규칙을 추가하거나 수정하는 데 비용이 발생하지 않습니다. 단, NACL을 통과하는 트래픽이 NAT 게이트웨이 등 유료 데이터 전송 서비스를 거칠 경우 해당 서비스에 대한 데이터 처리 요금은 발생할 수 있습니다.

개념 이해: NACL이란 무엇인가?

NACL은 Subnet (서브넷, VPC 내의 논리적인 네트워크 분할 영역) 수준에서 인바운드 및 아웃바운드 트래픽을 제어하는 Stateless (상태 비저장) 기반의 핵심 네트워크 방화벽입니다.

AWS 클라우드 인프라스트럭처에서 NACL은 서브넷 단위로 들어오고 나가는 트래픽을 제어하는 방화벽 역할을 합니다. 쉽게 비유하자면, SG (Security Group, 보안 그룹)가 개별 집(인스턴스)의 현관문 자물쇠라면, NACL은 아파트 단지(서브넷)의 정문 경비실과 같습니다.

🖥️ 콘솔 접근 경로

1 AWS Management Console 접속 → 2 VPC 서비스 검색 및 선택 → 3 왼쪽 사이드바 [보안] 메뉴 → 4 네트워크 ACL 클릭

핵심 특성: Stateless (상태 비저장)

NACL을 이해하는 데 있어 가장 중요한 핵심은 Stateless (상태 비저장)이라는 점입니다. 이는 트래픽의 왕복(요청과 응답) 상태를 기억하지 않는다는 의미입니다.

Stateful (상태 저장) - 예: 보안 그룹

사용자가 웹 서버에 80포트로 요청을 보냈다면, 방화벽이 이 '상태'를 기억합니다. 따라서 서버가 사용자에게 응답을 돌려보낼 때(Outbound) 별도의 허용 규칙이 없어도 응답 트래픽이 통과됩니다.

Stateless (상태 비저장) - 예: NACL

요청이 들어온(Inbound) 사실을 기억하지 못합니다. 따라서 들어온 요청에 대한 응답을 밖으로 내보내려면(Outbound) 반드시 아웃바운드 규칙에 해당 트래픽을 허용하는 명시적인 규칙이 존재해야만 합니다.

NACL vs 보안 그룹 (Security Group) 비교

구분 NACL (Network Access Control List) SG (Security Group, 보안 그룹)
적용 범위 서브넷 (Subnet) 단위 인스턴스 (ENI - 탄력적 네트워크 인터페이스) 단위
상태 저장 여부 Stateless (상태 비저장) Stateful (상태 저장)
규칙 평가 방식 규칙 번호 순서대로 평가 가장 관대한 규칙 적용
허용/거부 Allow 및 Deny 가능 Allow 규칙만 작성 가능

⚠️ 아키텍트의 경고: 너무 복잡한 NACL은 독이 됩니다.

실무에서 흔히 하는 실수 중 하나는 보안 그룹(SG)에서 해야 할 세밀한 포트 제어를 NACL에서 하려는 것입니다. NACL은 블랙리스트(특정 악의적 IP 차단) 형태나, 서브넷 간의 굵직한 라우팅 격리용으로만 사용하고, 세부적인 애플리케이션 포트 제어는 보안 그룹에 위임하는 것이 베스트 프랙티스입니다.

NACL 구성 요소와 작동 원리

1. Rule Number (규칙 번호)

가장 낮은 번호의 규칙부터 순차적으로 평가됩니다. 트래픽이 특정 규칙에 일치하면, 그 이후의 규칙은 평가하지 않고 즉시 결과를 결정합니다.

🛠️ 커스텀 NACL 생성 및 규칙 추가 방법

1 [네트워크 ACL 생성] 클릭 → 이름 입력 및 대상 VPC 선택

2 목록에서 생성된 NACL 선택 → 하단 [인바운드 규칙] 탭 클릭 → [인바운드 규칙 편집]

3 [새 규칙 추가] 클릭 → 규칙 번호(예: 100) 입력 → 유형/포트/소스 설정 → [변경 사항 저장]

2. Ephemeral Ports (임시 포트)의 이해

클라이언트가 서버에 연결할 때, 클라이언트는 1024 이상의 무작위 포트 번호를 자신의 통신 포트로 엽니다. NACL은 Stateless이므로, 서버의 응답이 이 임시 포트로 나갈 수 있도록 아웃바운드 규칙에서 반드시 허용해주어야 합니다.

💻 운영체제별 임시 포트 범위

  • 리눅스 (Amazon Linux, Ubuntu): 32768 - 60999
  • Windows Server 2008 이상: 49152 - 65535
  • ELB (Elastic Load Balancer): 1024 - 65535
  • 실무 팁: 보통 1024 - 65535 범위를 전체 개방하여 호환성을 유지합니다.

기본 웹 서버 서브넷 NACL 설정

⛓️ 서브넷에 NACL 연결하기

1 [네트워크 ACL] 목록에서 설정할 NACL 선택
2 하단 [서브넷 연결] 탭 클릭 → [서브넷 연결 편집] 클릭
3 해당 NACL을 적용할 서브넷 체크 → [변경 사항 저장]

실무 사례 (Real-world Use Cases)

사례 1: 특정 악성 IP 차단 (Blacklist)

🚫 차단 규칙(Deny) 적용하기

1 인바운드 규칙 편집 접속 → 2 가장 낮은 번호(예: 10)로 새 규칙 추가 → 3 소스에 공격자 IP 입력 → 4 작업을 거부(Deny)로 설정

# 우선순위 평가 원리
10 | 모든 트래픽 | 1.2.3.4/32 | DENY (최우선 차단)
100 | HTTP (80) | 0.0.0.0/0 | ALLOW (나머지 허용)

사례 2: 3-Tier 아키텍처의 엄격한 격리

데이터베이스(DB) 서브넷의 NACL을 설정하여 웹 서버에서의 직접 접근을 차단하고 오직 WAS 서버의 접근만 허용합니다.

🔒 DB 서브넷 보호 설정

1 WAS 서브넷의 CIDR 확인 (예: 10.0.2.0/24)
2 DB NACL 인바운드 규칙 100번에 WAS CIDR에 대한 3306 포트 Allow 추가
3 나머지 모든 트래픽에 대해 Deny 확인

자주 하는 실수 및 트러블슈팅 가이드

1. Stateless 특성 망각 (Timeout 오류)

인바운드만 허용하고 아웃바운드의 임시 포트를 열지 않아 통신이 끊기는 경우입니다.

✔ 해결책: 아웃바운드 규칙에 1024-65535 포트 허용이 있는지 확인하십시오.

2. 규칙 우선순위 역전

범위가 넓은 허용 규칙이 차단 규칙보다 번호가 낮을 때 발생합니다.

✔ 해결책: 차단(Deny) 규칙은 항상 가장 낮은 번호를 부여하십시오.
