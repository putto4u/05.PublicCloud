Auto Scaling: 탄력적 인프라의 핵심

인프라스트럭처의 Elasticity (탄력성, 부하에 따른 유연한 자원 조절 능력)은 현대 클라우드 아키텍처의 성패를 가르는 기준입니다. AS (Auto Scaling, 오토 스케일링)은 단순히 서버를 늘리는 것을 넘어, 가용성 보장과 비용 최적화라는 두 마리 토끼를 잡는 지능형 자동화 서비스입니다.

고가용성 아키텍처의 기본 도식

그림 1-1: 다중 가용 영역(Multi-AZ) 기반의 인스턴스 자동 확장 구조

위 도식은 ASG (Auto Scaling Group, 오토 스케일링 그룹)가 여러 가용 영역에 걸쳐 인스턴스를 균형 있게 배치하는 모습을 보여줍니다. 특정 지역의 데이터 센터에 장애가 발생하더라도 다른 영역에서 자원을 즉시 보충하여 서비스의 연속성을 보장합니다.

시작 템플릿 가이드와 구성 전략

LT (Launch Template, 시작 템플릿)는 인스턴스의 유전 정보를 담고 있는 설계도입니다. 인프라 구축 시 마주하게 되는 설정들의 실질적인 의미를 분석합니다.

PRO-TIP: "Auto Scaling 지침"의 진실

콘솔 UI의 "Auto Scaling 지침(Auto Scaling Guidance)" 체크박스는 실제 리소스의 동작 방식을 바꾸지 않습니다. 이는 오토 스케일링에 적합하지 않은 구형 인스턴스나 복잡한 네트워크 구성을 사용자가 실수로 선택하지 않도록 필터링해주는 '안전장치'입니다. 숙련된 아키텍트라면 이를 해제하여 더 세밀한 사용자 정의 구성을 할 수 있습니다.

필수 구성 요소

  • • AMI (Amazon Machine Image, 머신 이미지): OS 및 앱 패키지
  • • Instance Type (인스턴스 유형): CPU 및 메모리 사양
  • • Security Group (보안 그룹): 방화벽 규칙
  • • Key Pair (키 페어): 서버 접속 인증 키

고급 구성 요소

  • • User Data (사용자 데이터): 부팅 시 실행할 스크립트
  • • IAM Role (IAM 역할): 서버에 부여할 AWS 권한
  • • Storage (스토리지): EBS 볼륨 및 스냅샷 설정

FINOPS ALERT: 과금 및 자원 관리

오토 스케일링 그룹 자체는 무료지만, 생성된 EC2 인스턴스와 EBS (Elastic Block Store, 가상 디스크)는 유료입니다. 공격적인 확장 정책은 비용 폭탄의 원인이 될 수 있습니다. 반드시 Cooldown (쿨다운, 확장 후 대기 시간)과 Warm-up (웜업, 준비 시간) 기간을 설정하여 불필요한 자원 요동(Thrashing)을 방지하십시오.

구현 가이드: Auto Scaling 관리 콘솔 (GUI) 절차

아키텍처 설계도를 바탕으로 AWS Management Console 환경에서 인프라를 직접 프로비저닝(자원 할당)하는 단계별 표준 실무 절차입니다.

Phase 1. 시작 템플릿 (Launch Template) 생성

PATH EC2 콘솔 > 인스턴스 > 시작 템플릿 > [시작 템플릿 생성]
  • 시작 템플릿 이름 및 설명: 템플릿 이름에 web-app-template-v1을 입력하고, 버전 설명을 기록합니다.
  • Auto Scaling 지침 (선택 사항): Auto Scaling 지침 설정 체크박스를 선택하여 호환되지 않는 옵션을 필터링합니다.
  • AMI (Amazon Machine Image): 최근 안정화 버전인 Amazon Linux 2023 AMI 또는 비즈니스 요건에 맞는 OS를 선택합니다.
  • 인스턴스 유형 (Instance Type) 과금: 부하 테스트 기준에 맞춰 t3.micro 또는 t3.small을 선택합니다.
  • 키 페어 (Key pair): 서버 접근을 위한 기존 키 페어를 선택하거나 새 키 페어를 생성합니다.
  • 네트워크 설정 (Network settings): VPC를 지정하지 않고(ASG에서 덮어씀), 기존에 생성한 웹 서버용 보안 그룹 (Security Group)(예: 80, 443 포트 허용)을 선택합니다.
  • 고급 세부 정보 (Advanced details) > 사용자 데이터 (User data): 서버 부팅 시 웹 서버(Nginx/Apache)를 자동 설치하고 구동하는 Bash 스크립트를 입력합니다.
  • 하단의 [시작 템플릿 생성] 버튼을 클릭하여 설계를 완료합니다.

Phase 2. Auto Scaling 그룹 (ASG) 생성 및 네트워크 구성

PATH EC2 콘솔 > Auto Scaling > Auto Scaling 그룹 > [Auto Scaling 그룹 생성]
  • 이름 지정 및 템플릿 선택: 그룹 이름에 web-app-asg-prod를 입력하고, 방금 생성한 web-app-template-v1을 선택 후 [다음]을 클릭합니다.
  • 네트워크 구성: 대상 VPC를 선택하고, 고가용성을 위해 트래픽을 분산할 가용 영역 및 서브넷(Subnets)을 2개 이상 (예: ap-northeast-2a, ap-northeast-2c의 퍼블릭 또는 프라이빗 서브넷) 다중 선택합니다.
  • 고급 옵션 구성 (로드 밸런싱): 기존 로드 밸런서에 연결을 선택하고 사전에 생성한 ALB (Application Load Balancer)의 대상 그룹(Target Group)을 지정합니다.
  • 상태 확인 (Health Checks): 기본 EC2 외에 Elastic Load Balancing 상태 확인 켜기를 체크하여, 앱이 죽었을 때도 인스턴스를 교체하도록 설정합니다.
  • 그룹 크기 및 조정 정책:
    • 원하는 용량 (Desired capacity): 2
    • 최소 용량 (Minimum capacity): 2
    • 최대 용량 (Maximum capacity): 4 (트래픽 폭주 시 최대 확장 한계선)
  • 조정 정책 (Scaling Policy): 대상 추적 크기 조정 정책 (Target tracking)을 선택하고, 지표 유형을 평균 CPU 사용률로, 대상 값을 50(%)으로 설정합니다.
  • 검토 후 [Auto Scaling 그룹 생성]을 클릭하여 인프라 자동 확장을 개시합니다.

CloudFront: 전역 콘텐츠 전송 가속화

CloudFront (클라우드프론트)는 AWS의 고성능 CDN (Content Delivery Network, 콘텐츠 전송 네트워크) 서비스입니다. 이는 지리적 한계를 극복하고 전 세계 사용자에게 동일한 수준의 고속 서비스를 제공하는 아키텍처의 필수 요소입니다.

CloudFront 전역 인프라 도식

그림 4-1: 오리진(Origin)과 전역 엣지 로케이션(Edge Locations) 간의 배포 구조

CloudFront는 전 세계 450개 이상의 Edge Location (엣지 로케이션, 콘텐츠 전송 지점)을 보유하고 있습니다. 사용자의 요청은 가장 가까운 엣지로 전달되며, 데이터가 없을 경우에만 AWS 전용 네트워크를 통해 오리진에서 데이터를 가져와 캐싱합니다.

엣지 기술 심화: 캐싱 정책과 보안

효율적인 캐싱과 철저한 보안은 CloudFront 아키텍처의 완성도를 결정합니다.

캐싱 최적화 (Caching Policy)

TTL (Time To Live, 콘텐츠 유지 시간) 설정을 통해 데이터의 신선도를 조절합니다. 정적 이미지(JPG, PNG)는 길게, 자주 변하는 API 데이터는 짧게 설정하는 것이 핵심입니다.

보안 강화 (Edge Security)

OAC (Origin Access Control, 오리진 액세스 제어)를 적용하면 오리진(S3 등)의 퍼블릭 접근을 완전히 차단하고 오직 CloudFront 서명 요청만 허용하여 데이터 유출을 방지합니다.

전문가 기술 지침: Invalidation (무효화) 월 1,000회 초과 시 과금

배포된 웹사이트의 코드를 수정했을 때 사용자가 즉각 변화를 보지 못한다면 엣지에 옛날 데이터가 캐싱되어 있기 때문입니다. 이때 Invalidation (무효화, 캐시 강제 삭제) 기능을 사용하십시오. 하지만 무분별한 전체 경로(/*) 삭제는 대규모 추가 비용을 발생시킬 수 있으므로 특정 파일 경로(예: /images/logo.png)만 지정하는 것이 경제적입니다.

구현 가이드: CloudFront 관리 콘솔 (GUI) 절차

S3 (Simple Storage Service, 객체 스토리지) 버킷을 오리진으로 하는 안전한 정적 웹사이트 호스팅 및 CDN 배포 구성 절차입니다.

Phase 1. 배포 (Distribution) 생성 및 오리진 바인딩

PATH CloudFront 대시보드 > 배포 (Distributions) > [배포 생성 (Create distribution)]
  • 오리진 도메인 (Origin domain): 드롭다운 목록에서 사전 생성해 둔 S3 버킷 (예: corp-assets.s3.ap-northeast-2.amazonaws.com)을 선택합니다.
  • 오리진 액세스 (Origin access): 보안을 위해 반드시 원본 액세스 제어 설정(권장) (Origin access control settings)을 선택합니다.
  • 제어 설정 생성 (Create control setting): [제어 설정 생성]을 클릭하고 기본값을 유지한 채 생성합니다.
    (※ 중요: 배포 생성 완료 후, 제공되는 S3 버킷 정책(Bucket Policy)을 복사하여 해당 S3 버킷 권한 탭에 반드시 붙여넣어야 접근이 허용됩니다.)

Phase 2. 기본 캐시 동작 (Default Cache Behavior) 설정

엣지 로케이션에서 클라이언트의 요청을 어떻게 처리할지 정의합니다.

  • 뷰어 프로토콜 정책 (Viewer protocol policy): 보안 강화를 위해 HTTP를 HTTPS로 리디렉션 (Redirect HTTP to HTTPS)을 선택합니다.
  • 허용된 HTTP 메서드 (Allowed HTTP methods): 정적 콘텐츠 조회용이므로 GET, HEAD를 선택합니다.
  • 캐시 키 및 오리진 요청 (Cache key and origin requests): Cache policy and origin request policy (캐시 정책 및 오리진 요청 정책)를 선택합니다.
  • 캐시 정책 (Cache policy): AWS 관리형 정책인 CachingOptimized를 선택하여 기본 TTL 및 압축 옵션을 적용합니다.

Phase 3. 글로벌 설정 및 배포 완료

  • 가격 등급 (Price class): 글로벌 서비스가 아닌 국내 전용 서비스일 경우, 비용 절감을 위해 북미, 유럽, 아시아, 중동 및 아프리카 사용 (Use only North America, Europe, Asia, Middle East, and Africa) 수준으로 제한합니다.
  • AWS WAF 웹 ACL (Web Application Firewall) 과금 요인: 추가 보안이 불필요한 기본 실습의 경우 보안 보호 비활성화 (Do not enable security protections)를 선택합니다. 활성화 시 별도의 WAF 과금이 발생합니다.
  • 대체 도메인 이름 (CNAME) / 사용자 정의 SSL 인증서 (선택 사항): 자체 도메인(예: www.mycorp.com)을 연결하려면 도메인을 입력하고 ACM (AWS Certificate Manager, 인증서 관리자)에서 발급받은 인증서를 연결합니다.
  • 기본 루트 객체 (Default root object): 도메인 최상단 접속 시 띄울 파일명인 index.html을 입력합니다.
  • 설정 검토 후 맨 하단의 [배포 생성 (Create distribution)]을 클릭합니다. 배포 상태가 'Deploying'에서 완료되기까지 약 3~5분이 소요됩니다.

구현 가이드: Terragrunt (IaC) 자동화

실제 운영 환경에서는 IaC (Infrastructure as Code, 코드형 인프라) 도구를 통해 인프라를 형상 관리합니다. 아래는 Terragrunt (테라그런트)를 사용하여 표준화된 CloudFront 배포를 정의하는 코드입니다. 

# terragrunt.hcl (CloudFront 배포 자동화 모듈)
include "root" {
  path = find_in_parent_folders()
}

terraform {
  source = "tfr:///terraform-aws-modules/cloudfront/aws//?version=3.4.0"
}

inputs = {
  comment             = "엔터프라이즈 미디어 전송용 CDN"
  enabled             = true
  is_ipv6_enabled     = true
  price_class         = "PriceClass_200" # 비용 최적화 (북미/유럽/아시아 엣지 중심)

  # S3 오리진 구성 (OAC 적용)
  origin = {
    s3_origin = {
      domain_name = "corp-data-production.s3.amazonaws.com"
      origin_access_control_id = "E1A2B3C4D5E6"
    }
  }

  default_cache_behavior = {
    target_origin_id       = "s3_origin"
    viewer_protocol_policy = "redirect-to-https"
    allowed_methods        = ["GET", "HEAD"]
    cached_methods         = ["GET", "HEAD"]
    compress               = true
    query_string           = false
  }
}

* 테라폼 모듈을 활용하여 재사용성을 극대화한 인프라 정의 예시

학습 마무리

본 교재를 통해 Auto Scaling의 동적 유연성과 CloudFront의 전역 전송 최적화의 본질을 이해하고, 관리 콘솔을 통한 정교한 실무 프로비저닝 기법을 습득하였습니다. 고성능 인프라는 기술의 나열이 아닌, 사용자 경험과 비용 사이의 정교한 Trade-off (트레이드오프, 균형 맞추기)를 통해 완성됩니다.
