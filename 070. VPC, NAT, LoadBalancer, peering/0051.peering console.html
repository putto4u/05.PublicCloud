DNS 기초와 Route 53의 정의

DNS (Domain Name System, 도메인 이름 시스템)은 사람이 읽을 수 있는 도메인 이름(예: example.com)을 컴퓨터가 이해할 수 있는 IP 주소(예: 192.0.2.1)로 변환해주는 '인터넷의 전화번호부'와 같습니다.

Route 53 (Route 53, 53번 포트 경로)은 AWS에서 제공하는 가용성과 확장성이 뛰어난 클라우드 DNS 웹 서비스입니다. 이름의 유래는 DNS 쿼리가 표준적으로 사용하는 **TCP/UDP 53번 포트**와 미국 전역을 횡단하는 전설적인 도로인 **Route 66**의 오마주가 결합된 것입니다.

DNS 쿼리 흐름도 (Recursive vs Authoritative)

도메인 해석 요청(Recursive)과 최종 답변 제공(Authoritative)의 상호작용

전문 용어 주석

  • • Recursive Resolver (재귀적 해석기): 사용자의 요청을 받아 여러 네임서버를 대신 찾아다니며 최종 IP를 알아내는 서버 (예: ISP DNS, Google 8.8.8.8).
  • • Authoritative Name Server (권한 있는 네임서버): 특정 도메인에 대한 최종 답변(IP 주소 등)을 공식적으로 보유하고 있는 서버 (Route 53의 역할).

기능 및 역할 비교 분석

성공적인 하이브리드 클라우드 구축을 위해서는 전통적인 On-premise (온프레미스, 사내 구축형) DNS와 TLD (Top-Level Domain, 최상위 도메인) 서버, 그리고 Route 53의 차이를 명확히 이해해야 합니다.

비교 항목 온프레미스 DNS (예: BIND) 상위/TLD DNS (예: .com) Route 53 (Managed DNS)
관리 주체 사내 인프라 팀 (직접 관리) ICANN 및 각국 등록기관 AWS (Managed Service)
주요 기능 내부망 호스트 이름 해석 하위 네임서버 위치 안내 (포인팅) 전역 트래픽 제어 및 상태 확인
가용성 서버 이중화 필요 (H/W 의존) 극도로 높음 (인터넷 근간) 100% SLA 보장 (Anycast 활용)
하이브리드 지원 수동 전달(Forwarding) 설정 해당 없음 R53 Resolver를 통한 자동화

Architect's Insight: Anycast 네트워크

Route 53이 100% 가용성을 보장할 수 있는 이유는 **Anycast (애니캐스트, 전 세계 어디서든 동일한 IP로 가장 가까운 노드에 접속하는 기술)**를 사용하기 때문입니다. 전 세계 수백 개의 엣지 로케이션에 동일한 IP 주소를 할당하여, 특정 지역에 장애가 발생해도 시스템이 자동으로 다른 최적의 경로로 쿼리를 라우팅합니다.

Route 53 독보적 핵심 기능

다양한 라우팅 정책 (Routing Policies)

  • • Weighted (가중치 기반): 트래픽의 %를 나누어 전송 (카나리 배포 용도).
  • • Latency (지연 시간 기반): 사용자에게 응답 속도가 가장 빠른 리전으로 전달.
  • • Failover (장애 조치): 기본 서버 장애 시 예비 서버로 자동 전환.
  • • Geolocation (지리적 위치): 사용자의 국가/대륙별로 다른 IP 제공.

상태 확인 및 모니터링 (Health Checks)

단순히 IP를 알려주는 것을 넘어, 대상 서버의 상태를 주기적으로 체크합니다. 서버가 응답하지 않으면 DNS 수준에서 해당 레코드를 즉시 제거하여 사용자 경험 저하를 막습니다.

Alias Record (별칭 레코드)

AWS 리소스(ELB, S3 등) 전용 기능입니다. 일반적인 CNAME과 달리 **추가적인 DNS 쿼리 비용이 발생하지 않으며**, Zone Apex(루트 도메인, 예: example.com)에 대해서도 별칭을 지정할 수 있는 혁신적인 기능입니다.

하이브리드 리졸버 (Resolver)

온프레미스 DNS와 AWS VPC 내부 DNS 사이의 가교 역할을 합니다. **Inbound/Outbound Endpoint**를 통해 VPN이나 Direct Connect를 넘어 양방향 도메인 질의를 가능케 합니다.

실무 구축 가이드: AWS Console (GUI)

1

Hosted Zone (호스팅 영역) 생성

구매한 도메인(예: expert-cloud.com)을 등록하는 단계입니다.

  • [Create hosted zone] 버튼 클릭
  • [Domain name] 입력 및 [Type] 선택 (Public: 인터넷용, Private: 내부망용)
2

Record (레코드) 생성

실제 트래픽을 어디로 보낼지 정의합니다.

  • [Create record] 클릭
  • [Record type] 선택 (A: IPv4 주소, CNAME: 다른 도메인, MX: 이메일 서버 등)
  • [Alias] 스위치 활성화: AWS ELB나 S3 엔드포인트를 선택할 때 권장

⚠️ 과금 정책 안내 (Pricing)

• **Hosted Zone**: 영역당 **월 $0.50**의 기본 고정비가 발생합니다.
• **Query Fee**: 100만 건당 **$0.40** (표준 쿼리 기준)의 사용료가 부과됩니다.
• **Health Checks**: 리소스당 **월 $0.50~$0.75** (AWS 내/외부 여부에 따라 다름)가 추가됩니다.

Architect's Checklist: 주의사항 및 팁

필독 주의사항

  • ! TTL (Time To Live, 유지 시간) 관리: 변경 사항이 생길 예정이라면 사전에 TTL 값을 낮춰두어야 합니다. 너무 높으면 전 세계 DNS 캐시에 옛 정보가 남아 장애 복구가 늦어집니다.
  • ! NS (Name Server, 네임서버) 위임 확인: 도메인 등록처(가비아, 후이즈 등)에서 Route 53이 부여한 4개의 네임서버 주소를 정확히 업데이트해야 활성화됩니다.
  • ! CNAME vs Alias: 루트 도메인(example.com)에는 RFC 표준상 CNAME을 쓸 수 없습니다. 반드시 **Route 53 Alias**를 사용하십시오.

실무 전문가 팁

  • ★ Blue/Green 배포 활용: Weighted Routing을 사용하여 신규 버전으로 트래픽을 5%, 10%, 50% 점진적으로 넘기며 안정성을 테스트하십시오.
  • ★ CloudWatch 통합: Health Check 상태를 CloudWatch 경보와 연동하여 관리자가 즉시 인지하도록 자동화하십시오.
  • ★ 로그 감사: 쿼리 로그(Query Logs)를 활성화하여 어떤 IP에서 비정상적인 요청이 오는지 분석하고 보안에 대비하십시오.

IaC 자동화: Terragrunt 구현

Terragrunt (테라그룬트)를 사용하면 여러 계정이나 리전의 DNS 설정을 표준화된 코드로 관리할 수 있습니다. 

# terragrunt.hcl (Route 53 DNS 설정 자동화)
include "root" {
  path = find_in_parent_folders()
}

terraform {
  source = "tfr:///terraform-aws-modules/route53/aws//modules/zones?version=2.10.2"
}

inputs = {
  zones = {
    "expert-infra.com" = {
      comment = "전문가 인프라 교육용 공식 도메인"
      tags = {
        Environment = "Production"
        Owner       = "SeniorArchitect"
      }
    }
  }

  records = [
    {
      name = "www"
      type = "A"
      alias = {
        name    = "dualstack.my-alb-12345.ap-northeast-2.elb.amazonaws.com"
        zone_id = "Z23456789ABCDEF" # 해당 리전의 ELB 호스팅 영역 ID
      }
    },
    {
      name = "api"
      type = "A"
      ttl  = 300
      records = ["13.124.1.100", "13.124.1.101"] # 라운드 로빈 다중 IP
    }
  ]
}

구축 결과 확인 및 검증 방법

1. 명령줄 도구 활용 (CLI 검증)

$ dig www.expert-infra.com

→ ANSWER SECTION에서 설정한 IP/Alias 확인

$ nslookup api.expert-infra.com

→ 설정된 다중 IP가 라운드 로빈으로 나오는지 확인

2. 온라인 도구 활용 (전역 전파 확인)

전 세계의 DNS 서버에 캐시가 갱신되었는지 확인하기 위해 WhatsMyDNS나 DNSChecker 서비스를 활용하십시오. 지도상에 초록색 체크 표시가 나타나면 정상 전파된 것입니다.

강의 요약 및 핵심 정리

DNS는 서비스의 관문입니다. Route 53은 단순한 주소 변환기가 아닌, **전역 로드 밸런싱(Global Server Load Balancing, GSLB)**의 정수입니다. 테라그룬트와 같은 도구로 자동화하고, 상태 확인을 통해 자가 치유(Self-healing) 인프라를 구축하는 것이 현대 아키텍트의 숙명입니다.
