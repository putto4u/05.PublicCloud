Network Interconnectivity (Expert Practical)

프라이빗 네트워크의 직접 연결:
VPC Peering 심층 분석

VPC Peering(VPC 피어링)은 서로 다른 두 개의 VPC 간에 인터넷을 거치지 않고 AWS의 고속 프라이빗 백본(Backbone) 네트워크를 통해 트래픽을 라우팅할 수 있게 해주는 가장 기본적이고 강력한 네트워크 연결 서비스입니다.

VPC Peering의 3대 핵심 원칙

No Overlapping CIDR

피어링을 맺으려는 두 VPC의 IP 주소 대역(CIDR 블록)이 단 1개의 IP라도 겹치면 연결 생성이 원천적으로 불가능합니다. 사전 IP 주소 체계 설계가 필수입니다.

Non-Transitive Routing

전이적 라우팅 불가. A와 B가 연결되고, B와 C가 연결되었다고 해서 A와 C가 통신할 수 없습니다. A와 C가 통신하려면 별도의 1:1 피어링(A-C)을 맺어야 합니다.

Cross-Region / Account

같은 AWS 계정이나 리전뿐만 아니라, 완전히 다른 물리적 국가(예: 서울-도쿄)에 있거나 전혀 다른 회사의 AWS 계정과도 안전하게 피어링을 맺을 수 있습니다.

실무 예제 1: 기본 1:1 피어링 (동일 계정, 동일 리전)

가장 흔하게 사용되는 패턴입니다. 개발 서버가 위치한 App VPC와 공용 데이터베이스가 위치한 DB VPC를 연결합니다. 하나의 테라폼 코드에서 요청(Request)과 수락(Accept)을 동시에 처리할 수 있습니다.

App VPC (Requester)
10.1.0.0/16
RT: 10.2.0.0/16 → pcx-01
Peering: pcx-01
DB VPC (Accepter)
10.2.0.0/16
RT: 10.1.0.0/16 → pcx-01

Terraform: 1:1 Peering

HCL 
# ==========================================
# 1. VPC Peering 연결 생성 (동일 계정/리전이므로 auto_accept 사용 가능)
# ==========================================
resource "aws_vpc_peering_connection" "app_to_db" {
  vpc_id      = aws_vpc.app_vpc.id    # 요청자 (Requester)
  peer_vpc_id = aws_vpc.db_vpc.id     # 수락자 (Accepter)
  auto_accept = true                  # 피어링 자동 수락 활성화

  tags = { Name = "Peering-App-to-DB" }
}

# ==========================================
# 2. App VPC 라우팅 테이블 업데이트 (DB로 가는 길 추가)
# ==========================================
resource "aws_route" "app_to_db_route" {
  route_table_id            = aws_route_table.app_rt.id
  destination_cidr_block    = aws_vpc.db_vpc.cidr_block # 10.2.0.0/16
  vpc_peering_connection_id = aws_vpc_peering_connection.app_to_db.id
}

# ==========================================
# 3. DB VPC 라우팅 테이블 업데이트 (App으로 돌아가는 길 추가)
# ==========================================
resource "aws_route" "db_to_app_route" {
  route_table_id            = aws_route_table.db_rt.id
  destination_cidr_block    = aws_vpc.app_vpc.cidr_block # 10.1.0.0/16
  vpc_peering_connection_id = aws_vpc_peering_connection.app_to_db.id
}

실무 예제 2: 교차 리전(Cross-Region) 피어링

글로벌 서비스 운영이나 재해 복구(DR)를 위해 서울 리전(ap-northeast-2)과 도쿄 리전(ap-northeast-1)의 VPC를 연결합니다. 서로 다른 리전을 제어하기 위해 테라폼에서 Provider Alias(공급자 별칭)를 사용하여 명시적으로 권한을 분리해야 합니다. (이 경우 auto_accept = true는 작동하지 않으므로 수락자 리소스를 별도로 작성해야 합니다.)

Terraform: Cross-Region Peering

HCL 
# ==========================================
# 0. Provider 별칭(Alias) 설정: 테라폼이 두 리전을 동시에 제어하도록 선언
# ==========================================
provider "aws" {
  region = "ap-northeast-2" # 기본 프로바이더 (서울)
}

provider "aws" {
  alias  = "tokyo"
  region = "ap-northeast-1" # 별칭 프로바이더 (도쿄)
}

# ==========================================
# 1. 서울 리전에서 피어링 요청 (Requester)
# ==========================================
resource "aws_vpc_peering_connection" "seoul_req" {
  vpc_id        = aws_vpc.seoul_vpc.id
  peer_vpc_id   = aws_vpc.tokyo_vpc.id
  peer_region   = "ap-northeast-1" # 타겟 리전 명시
  
  # 주의: 교차 리전에서는 auto_accept가 불가능함
}

# ==========================================
# 2. 도쿄 리전에서 피어링 수락 (Accepter)
# ==========================================
resource "aws_vpc_peering_connection_accepter" "tokyo_acc" {
  provider                  = aws.tokyo # 별칭 프로바이더 사용 필수!
  vpc_peering_connection_id = aws_vpc_peering_connection.seoul_req.id
  auto_accept               = true    # 수락자 측에서 승인
}

# (라우팅 테이블 추가 코드는 예제 1과 동일하므로 생략)

실무 예제 3: 전이적 라우팅 불가(Non-Transitive)의 이해

가장 많이 겪는 아키텍처 설계 오류입니다. 중앙 관리용 VPC(A)를 두고 여러 서비스 VPC(B, C)를 피어링으로 연결한 가상 허브 앤 스포크 모델을 만들었다고 가정해 봅시다. A와 B, A와 C는 통신이 되지만, B와 C는 A를 거쳐서 통신할 수 없습니다. B와 C가 통신하려면 B-C 간에 직접 피어링을 맺거나, AWS Transit Gateway(전송 게이트웨이)로 아키텍처를 업그레이드해야 합니다.

VPC A
(Mgmt)
VPC B
(Service 1)
VPC C
(Service 2)

수석 아키텍트의 실전 설계 팁: 확장성 한계 극복

VPC 개수가 4~5개를 넘어가 서로 모두 통신해야 하는 상황(Full-Mesh)이 되면 피어링 연결 개수는 기하급수적으로 늘어납니다 (N*(N-1)/2 공식). 즉, 10개의 VPC를 연결하려면 45개의 피어링과 라우팅 테이블 관리가 필요합니다. 이 시점부터는 VPC Peering을 폐기하고 중앙 집중형 라우터인 AWS Transit Gateway로 아키텍처를 즉시 마이그레이션해야 운영 오버헤드를 막을 수 있습니다.

피어링 구성 시 과금 및 보안 주의사항 (필독)

데이터 전송(Data Transfer) 요금

VPC Peering 연결 자체를 생성하고 유지하는 것은 무료입니다. 하지만 피어링을 통과하는 트래픽에는 요금이 발생합니다. 같은 리전 내의 서로 다른 AZ 간 통신이라면 양방향으로 데이터 전송 요금(GB당 $0.01)이 부과되며, 교차 리전(Cross-Region) 피어링의 경우 해당 리전 간의 표준 데이터 전송 요금(더 비쌈)이 적용됩니다. 대규모 데이터 백업 시 요금 폭탄에 주의하십시오.

보안 그룹(Security Group) 상호 참조 제약

동일 리전 내의 피어링된 VPC 간에는 상대방 VPC의 인스턴스 보안 그룹 ID(예: sg-1234abcd)를 내 보안 그룹의 '인바운드 소스'로 직접 지정하여 정밀한 방화벽 규칙을 세울 수 있습니다. 하지만, 교차 리전 피어링의 경우에는 보안 그룹 ID 참조가 불가능하므로 상대방의 IP 대역(CIDR) 전체를 열어주어야 하는 보안 취약점이 발생합니다.

주석 및 용어 설명

  1. Provider Alias (공급자 별칭): 테라폼에서 동일한 클라우드 프로바이더(예: AWS)를 사용하지만, 접속할 리전(Region)이나 계정(Account)이 다를 때 이를 구분하기 위해 부여하는 가상의 이름입니다. 글로벌 인프라 배포 시 필수적인 문법입니다.
  2. CIDR (Classless Inter-Domain Routing): 10.1.0.0/16과 같이 네트워크의 크기(사용 가능한 IP 주소의 수)를 정의하는 방식입니다. 인프라 설계 시 서로 다른 VPC 간에 이 범위가 겹치지 않도록 사전에 IP 설계(IP Addressing Plan)를 수립하는 것이 매우 중요합니다.
  3. Full-Mesh (풀 메시 토폴로지): 네트워크 내의 모든 노드(여기서는 VPC)가 다른 모든 노드와 직접 1:1로 촘촘하게 연결된 형태를 말합니다. 내결함성은 최고 수준이지만 확장이 매우 어렵습니다.
