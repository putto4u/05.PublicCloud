중앙 집중형 인바운드 라우팅: Hub ALB에서 TGW를 거쳐 Spoke NLB로 연결하는 아키텍처

기업의 클라우드 네트워크가 확장됨에 따라, 보안과 트래픽 관리를 중앙 집중화하기 위해 Hub-Spoke (허브 앤 스포크) 아키텍처를 채택하는 경우가 많습니다. 본 장에서는 중앙의 Hub VPC (Virtual Private Cloud, 가상 사설 클라우드)에 위치한 ALB (Application Load Balancer, 애플리케이션 로드 밸런서)가 트래픽을 수신하고, 이를 TGW (Transit Gateway, 트랜짓 게이트웨이)를 거쳐 대상 서비스가 있는 Spoke VPC의 NLB (Network Load Balancer, 네트워크 로드 밸런서)로 안전하게 전달하는 고급 라우팅 구성 방법을 단계별로 살펴봅니다.

이 구성은 L7 (Layer 7) 수준의 라우팅과 AWS WAF (Web Application Firewall, 웹 애플리케이션 방화벽) 보호를 중앙 Hub에서 처리하고, 개별 Spoke 서비스는 L4 (Layer 4) 수준의 NLB를 통해 확장성을 확보하는 하이브리드 로드 밸런싱 패턴입니다.

도식 1. Hub ALB에서 TGW를 경유하여 Spoke NLB로 향하는 중앙 집중형 트래픽 흐름

⚠️ 비용 주의 사항

본 아키텍처를 구현할 때 사용되는 ALB [과금 발생], NLB [과금 발생], TGW [과금 발생]는 모두 생성 시점부터 시간당 과금 및 처리된 데이터 용량당 과금이 발생하는 유료 서비스입니다. 인프라 실습 후에는 반드시 리소스를 삭제하여 불필요한 비용을 방지해야 합니다.

1. 사전 준비 및 아키텍처 이해

본 구성 요소를 배포하기 전에 트래픽의 정확한 흐름을 이해해야 합니다. 외부 클라이언트의 요청은 Hub VPC의 퍼블릭 서브넷에 있는 ALB로 인입되며, ALB는 TGW를 통해 Spoke VPC 내 프라이빗 서브넷에 위치한 NLB의 ENI (Elastic Network Interface, 탄력적 네트워크 인터페이스) IP (Internet Protocol, 인터넷 프로토콜) 주소로 트래픽을 전달합니다.

2. Spoke VPC 구성: 빈 대상 그룹 및 NLB 생성

Spoke VPC에는 실제 애플리케이션 서버가 위치하게 되지만, 초기 인프라 프로비저닝 단계에서는 아직 대상 서버 (EC2 (Elastic Compute Cloud, 엘라스틱 컴퓨트 클라우드) 또는 컨테이너)가 준비되지 않았을 수 있습니다. 따라서 대상을 등록하지 않은 빈 TG (Target Group, 대상 그룹)를 먼저 생성하고 NLB를 연결합니다.

Step 2-1. 빈 대상 그룹 생성

  • Amazon EC2 콘솔의 로드 밸런싱 섹션에서 대상 그룹 메뉴로 이동합니다.
  • 대상 유형 선택: 향후 연결할 백엔드 리소스에 맞춰 인스턴스(Instance) 또는 IP 주소(IP addresses)를 선택합니다.
  • 프로토콜 및 포트: TCP (Transmission Control Protocol, 전송 제어 프로토콜) 프로토콜과 서비스할 포트(예: 80 또는 443)를 지정합니다.
  • VPC 선택: 인프라가 배포될 Spoke VPC를 선택합니다.
  • 대상 등록 단계 생략: 대상을 등록하는 화면에서 어떠한 IP나 인스턴스도 추가하지 않고 '빈 그룹' 상태로 생성을 완료합니다.

Step 2-2. Spoke VPC에 NLB 생성 [과금 발생]

  • 로드 밸런서 메뉴에서 Network Load Balancer를 생성합니다.
  • 체계(Scheme): 인터넷 트래픽을 직접 받지 않고 내부 통신만 수행하므로 반드시 내부(Internal)로 설정합니다.
  • 네트워크 매핑: Spoke VPC와 NLB가 위치할 AZ (Availability Zone, 가용 영역) 및 프라이빗 서브넷을 선택합니다.
  • 리스너 및 라우팅: 방금 생성한 빈 대상 그룹을 선택하여 리스너 규칙에 연결합니다.
  • NLB IP 주소 확보: NLB 생성이 완료되면, 네트워크 인터페이스(ENI) 콘솔 메뉴에서 해당 NLB가 각 서브넷에 할당받은 프라이빗 IP 주소들을 반드시 기록해 둡니다. (이 IP들이 이후 Hub ALB의 최종 라우팅 대상이 됩니다.)

3. TGW 생성 및 네트워크 연결 [과금 발생]

VPC 간의 패킷 전달을 위해 중앙 집중형 라우터 역할을 수행하는 TGW를 구성합니다.

Step 3-1. TGW 생성 및 연결 (Attachment)

  • VPC 콘솔에서 TGW를 생성합니다. (기본 설정 유지)
  • TGW Attachment (트랜짓 게이트웨이 연결)를 생성하여 Hub VPC와 Spoke VPC를 각각 TGW에 연결합니다.
  • 이때, 각 VPC 내에서 TGW 트래픽 전용으로 할당된 라우팅 서브넷(/28 대역 등)을 지정하는 것이 엔터프라이즈 아키텍처의 모범 사례입니다.

Step 3-2. TGW 및 VPC 라우팅 테이블 설정

  • TGW 라우팅 테이블: Hub VPC에서 출발한 트래픽이 Spoke VPC의 CIDR (Classless Inter-Domain Routing, 클래스 없는 도메인 간 라우팅)로 향할 수 있도록 경로를 검증 및 추가합니다. Spoke에서 Hub로 향하는 반환 경로도 구성되어야 합니다.
  • Hub VPC 라우팅 테이블: 대상(Destination)이 Spoke VPC CIDR (예: 10.1.0.0/16)인 트래픽을 TGW로 보내도록 규칙을 추가합니다.
  • Spoke VPC 라우팅 테이블: 대상이 Hub VPC CIDR (예: 10.0.0.0/16)인 트래픽을 TGW로 보내도록 반환 규칙을 추가합니다.

4. Hub VPC 구성: ALB 생성 및 로드 밸런싱 설정

Hub VPC의 ALB는 외부에서 들어오는 클라이언트 트래픽을 수신하여, TGW를 거쳐 Spoke VPC의 NLB IP로 전달해야 합니다. ALB는 다른 VPC에 있는 NLB 객체 자체를 식별할 수 없으므로, NLB의 프라이빗 IP 주소를 직접 대상으로 지정하는 방식을 사용합니다.

Step 4-1. Hub VPC용 IP 대상 그룹 생성

  • Hub VPC 콘솔에서 새로운 대상 그룹을 생성합니다.
  • 대상 유형 선택: 반드시 IP 주소(IP addresses)를 선택해야 합니다.
  • 네트워크 설정: 라우팅 네트워크를 선택하고, 직접 Spoke VPC CIDR 대역을 입력하거나 '기타 프라이빗 IP 주소(Other private IP addresses)'를 지정합니다.
  • 대상 등록: 앞서 'Step 2-2'에서 기록해 둔 Spoke NLB의 ENI 프라이빗 IP 주소들을 모두 입력하여 등록합니다.

Step 4-2. Hub VPC에 ALB 생성 [과금 발생]

  • 로드 밸런서 메뉴에서 Application Load Balancer를 생성합니다.
  • 체계(Scheme): 클라이언트 통신용이므로 인터넷 연결(Internet-facing)을 선택합니다.
  • 네트워크 매핑: 외부 통신이 가능한 Hub VPC의 퍼블릭 서브넷 2개 이상을 선택합니다.
  • 리스너 및 라우팅: 생성해 둔 Hub VPC용 IP 대상 그룹을 리스너로 연결합니다.

5. 실전 팁 및 자주 오해하거나 실수하는 부분 💡

현업에서 위 아키텍처를 구성할 때 발생하는 가장 흔한 장애 원인과 아키텍트로서 반드시 고려해야 할 실무 팁입니다.

① 빈 대상 그룹과 ALB 상태 검사 (Health Check) 실패 문제

Spoke NLB에 대상을 등록하지 않고 '빈 그룹'으로 두었을 때, Hub ALB에서 이 NLB의 IP로 보내는 상태 검사는 반드시 실패(Unhealthy)하게 됩니다. NLB는 뒤에 활성화된 타겟(EC2 등)이 없으면 수신된 TCP 연결 요청에 대해 연결을 맺어주지 않고 RST (Reset) 패킷을 반환하거나 연결을 타임아웃 시키기 때문입니다.

결론: 라우팅 및 인프라 네트워크 세팅이 완벽하더라도, Spoke NLB에 실제 애플리케이션 IP가 등록되고 서비스가 정상 구동되기 전까지는 Hub ALB가 해당 경로로 트래픽을 넘기지 않습니다. 이는 정상적인 작동 방식입니다.

② 보안 그룹 (Security Group) 체인 설정

최근 AWS 업데이트로 NLB에도 보안 그룹(SG) 적용이 가능해졌습니다. 트래픽 흐름을 완성하기 위해 다음 사항을 확인하십시오.

  • Hub ALB 보안 그룹: 아웃바운드로 Spoke NLB의 포트(예: 80, 443)를 허용해야 합니다.
  • Spoke NLB 보안 그룹: 인바운드 규칙에 반드시 Hub VPC의 CIDR 또는 ALB 서브넷 대역을 허용해야 합니다.

③ 상태 비저장 (Stateless) 비대칭 라우팅 오류 방지

TGW를 통과하는 트래픽이 서로 다른 가용 영역(AZ)으로 인입 및 반환될 경우 패킷이 드롭되는 비대칭 라우팅 문제가 발생할 수 있습니다. 본 구성에서는 ALB가 프록시 역할을 수행하여 패킷의 출발지 IP가 ALB 자신의 IP로 SNAT (Source Network 시Address Translation, 출발지 네트워크 주소 변환)되므로 발생 확률이 낮습니다. 하지만 방화벽 어플라이언스를 통과하는 등 복잡한 라우팅이 추가될 경우 TGW 연결 설정에서 어플라이언스 모드(Appliance Mode) 활성화 여부를 검토해야 합니다.

용어 주석 설명

Hub-Spoke 아키텍처: 자전거 바퀴의 중심(Hub)과 바퀴살(Spoke)의 형태에서 유래한 네트워크 디자인입니다. 모든 트래픽이 중앙 허브를 거치도록 설계하여 방화벽이나 로드 밸런서 같은 공통 보안/네트워크 장비를 일원화할 수 있는 강력한 엔터프라이즈 환경 모델입니다.

SNAT (Source Network Address Translation, 출발지 네트워크 주소 변환): 패킷이 네트워크 장비(여기서는 ALB)를 통과할 때, 응답 패킷이 다시 자신에게 돌아올 수 있도록 패킷의 원래 출발지 IP 주소를 장비 자신의 내부 IP 주소로 변환하는 기술입니다.

ENI (Elastic Network Interface, 탄력적 네트워크 인터페이스): AWS VPC 내에서 인스턴스나 관리형 서비스(NLB 등)가 네트워크 통신을 하기 위해 할당받는 가상의 네트워크 카드입니다. 고정적인 프라이빗 IP를 가집니다.
