IGW vs NAT Gateway

AWS 네트워크 관문 서비스의 개념과 설계 차이점 분석

1. IGW (Internet Gateway, 인터넷 게이트웨이)

**IGW**(Internet Gateway, 인터넷 게이트웨이)는 **VPC**(Virtual Private Cloud, 가상 사설 클라우드)와 인터넷 간의 통신을 가능하게 하는 수평 확장 가능하고 중복성이 확보된 고가용성 **VPC** 구성 요소입니다. 이는 네트워크 트래픽에 대한 가용성 위험이나 대역폭 제약 없이 원활한 통신을 제공합니다.

주요 기능 및 특징

  • **양방향 통신 지원:** 외부에서 내부로(**Inbound**, 인바운드) 및 내부에서 외부로(**Outbound**, 아웃바운드)의 통신이 모두 가능합니다.
  • **대상:** **Public IP**(Public Internet Protocol, 공인 인터넷 규약 주소)를 가진 인스턴스에 직접 연결됩니다.
  • **논리적 객체:** 물리적 장비가 아닌 소프트웨어 정의 네트워크로 구현되어 성능 한계가 거의 없습니다.
[주석]
* **Inbound/Outbound (인바운드/아웃바운드):** 네트워크 경계를 기준으로 들어오는 트래픽(Inbound)과 나가는 트래픽(Outbound)을 의미합니다.
* **Public IP (공인 IP):** 인터넷 전체에서 유일하게 식별될 수 있는 주소로, 외부 망에서 해당 서버를 찾기 위한 필수 주소입니다.

[그림: IGW와 NAT Gateway의 논리적 위치 및 흐름도]

2. NAT Gateway (Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이)

**NAT Gateway**(Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이)는 프라이빗 서브넷의 인스턴스가 인터넷 또는 다른 **AWS** 서비스에 연결될 수 있게 하지만, 외부 인터넷에서 해당 인스턴스로의 직접적인 연결은 차단하는 고성능 매니지드 서비스입니다.

주요 기능 및 특징

  • **단방향 통신 특화:** 내부에서 밖으로 나가는 **Outbound**(아웃바운드) 요청만 가능합니다.
  • **보안 강화:** 실제 서버의 사설 IP를 숨기고 게이트웨이의 공인 IP로 통신합니다.
  • **AZ 종속성:** 가용 영역별로 생성하여 장애 격리를 유지합니다.

부재 시 발생하는 문제점

  • **패치/업데이트 불가:** OS 보안 업데이트 및 패키지 설치가 차단됩니다.
  • **API 연동 실패:** 결제 모듈이나 외부 데이터 API 서버와 통신이 끊깁니다.
  • **설계적 모순:** 외부 통신을 위해 보안이 중요한 자원을 퍼블릭으로 노출하게 됩니다.

2.1 NAT 부재 시의 상세 영향 분석

가. 소프트웨어 생명주기 관리 마비

리눅스의 `yum install`이나 `apt-get`, 윈도우의 **Windows Update**(윈도우 업데이트) 서비스는 외부 리포지토리에 접속해야 합니다. **NAT**가 없으면 서버는 최신 보안 패치를 받을 수 없으며, 이는 장기적으로 심각한 보안 취약점을 초래합니다.

나. 외부 비즈니스 서비스 연동 불가

현대의 애플리케이션은 **SaaS**(Software as a Service, 서비스형 소프트웨어) 형태의 외부 API(예: 결제 게이트웨이, 공공 데이터 포털)를 빈번하게 호출합니다. **Outbound**(아웃바운드) 경로가 없으면 이러한 비즈니스 로직이 모두 작동하지 않습니다.

[주석]
* **SaaS (Software as a Service):** 클라우드 기반의 소프트웨어 제공 모델로, 사용자가 설치 없이 웹을 통해 소프트웨어를 이용하는 방식입니다.
* **Repository (리포지토리, 저장소):** 소프트웨어 패키지들이 저장되어 있는 중앙 서버로, 업데이트 시 해당 위치에서 파일을 다운로드합니다.

3. 온프레미스 통합 장비에서 AWS 독립 서비스로의 진화 이유

전통적인 온프레미스(On-premises, 사내 구축형) 환경에서는 하나의 **L3 Switch**(Layer 3 Switch, 3계층 스위치)나 **Firewall**(파이어월, 방화벽) 장비가 모든 게이트웨이 역할을 통합 수행했습니다. 하지만 **AWS**가 이를 분리한 이유는 클라우드만의 **고가용성**과 **무한 확장**을 달성하기 위함입니다.

1) 가용 영역(AZ)의 완전한 격리와 독립성

온프레미스 장비는 해당 장비가 고장 나면 전체 네트워크가 마비되는 **SPOF**(Single Point of Failure, 단일 장애점)가 됩니다. **AWS**는 **NAT Gateway**를 **AZ**(Availability Zone, 가용 영역)별로 독립 생성하게 하여, 한 데이터 센터의 장애가 다른 영역의 외부 통신에 영향을 주지 않도록 설계되었습니다.

2) SDN 기반의 무한 확장성 확보

통합 장비는 물리적 하드웨어의 성능(CPU, 메모리) 한계가 뚜렷합니다. **AWS**는 **IGW**를 소프트웨어로 정의된 논리적 계층(**SDN**, 소프트웨어 정의 네트워크)으로 분리하여 병목 현상 없이 수조 개의 패킷을 처리하게 하고, **NAT Gateway**는 별도의 매니지드 성능을 제공하여 각각의 역할에 최적화된 성능을 보장합니다.

3) 보안 계층화 (Defense in Depth)

게이트웨이를 분리하면 공격 표면이 줄어듭니다. **IGW**는 라우팅과 매핑에만 집중하고, **NAT Gateway**는 주소 변환과 단방향 통신 제어에 집중합니다. 이로써 한 서비스의 설정 오류가 전체 네트워크 보안을 붕괴시키는 리스크를 최소화합니다.

[주석]
* **SPOF (Single Point of Failure):** 시스템 구성 요소 중, 동작하지 않으면 시스템 전체가 중단되는 핵심 지점을 의미합니다.
* **Defense in Depth (심층 방어):** 하나의 방어선이 뚫리더라도 다음 방어선이 위협을 차단할 수 있도록 다중의 보안 계층을 두는 전략입니다.

4. IGW vs NAT Gateway 차이점 비교

항목 IGW (인터넷 게이트웨이) NAT Gateway (NAT 게이트웨이)
통신 방향 양방향 (Inbound / Outbound) 단방향 (Outbound Only)
연결 대상 퍼블릭 IP 보유 인스턴스 프라이빗 IP 보유 인스턴스
확장성 자동 무한 확장 (SDN 기반) 최대 45Gbps까지 자동 확장
설치 위치 VPC 전체 (논리적) 특정 서브넷 내 (물리적 AZ 종속)

⚠ 중요: 과금 체계 안내 (Cost Management)

IGW 비용

**무료.** 생성 및 유지에 따른 비용이 없습니다. (단, 인터넷 데이터 전송료 별도)

NAT Gateway 비용 (유료)

**시간당 요금 + 데이터 처리 요금.**
- 시간당 약 $0.045 (서울 리전 기준)
- 데이터 처리 GB당 약 $0.045

* NAT Gateway는 인스턴스가 꺼져 있어도 게이트웨이가 생성되어 있으면 지속적으로 과금되므로 미사용 시 즉시 삭제가 권장됩니다.
