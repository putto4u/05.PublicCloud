허브 앤 스포크 네트워크 환경의 스포크 VPC 내 EKS 구축 실무

허브 앤 스포크(Hub and Spoke) 네트워크 토폴로지가 완성된 상태에서, 특정 스포크 VPC (Virtual Private Cloud, 가상 사설 클라우드)에 EKS (Elastic Kubernetes Service, 완전 관리형 쿠버네티스 서비스)를 안전하고 효율적으로 배포하기 위한 구성 가이드입니다.

* 스포크 VPC (Spoke VPC): 허브 앤 스포크 네트워크 구조에서 외부 인터넷과 직접 연결되지 않고, 중앙의 허브 VPC를 통해서만 트래픽이 제어되고 라우팅되는 격리된 사설 네트워크 단위를 의미합니다.
그림 1: TGW(Transit Gateway) 기반 허브 앤 스포크 환경에서의 EKS 아키텍처
과금 주의 안내 (Billing Warning)

EKS 클러스터는 생성 시점부터 제어 평면(Control Plane)에 대해 시간당 $0.10의 비용이 지속적으로 발생합니다. 또한 워커 노드로 사용되는 EC2 (Elastic Compute Cloud, 탄력적 클라우드 컴퓨팅) 인스턴스 비용, 노드 간 통신에 따른 데이터 전송 비용, 그리고 허브 VPC의 NAT (Network Address Translation, 네트워크 주소 변환) 게이트웨이 데이터 처리 비용이 별도로 청구됩니다.

1. 사전 준비: IAM 역할(Role) 생성 및 권한 부여

EKS 클러스터 제어 평면과 워커 노드가 AWS의 다른 리소스에 접근할 수 있도록 IAM (Identity and Access Management, 신원 및 접근 관리) 역할을 사전에 생성해야 합니다. 이 과정이 누락되거나 권한이 부족하면 클러스터 생성 자체가 실패하거나 워커 노드가 클러스터에 조인(Join)하지 못하는 치명적인 오류가 발생합니다.

그림 2: EKS 제어 평면과 워커 노드의 IAM 역할 기반 권한 위임 구조

1.1 클러스터 역할 (Cluster Role) 생성

EKS 제어 평면이 AWS 리소스(예: 보안 그룹, 탄력적 네트워크 인터페이스 등)를 생성하고 관리할 수 있도록 권한을 위임합니다.

경로: AWS IAM 콘솔 ➔ [역할(Roles)] ➔ [역할 생성(Create role)]
  1. 신뢰할 수 있는 엔터티 선택:
    • 엔터티 유형: AWS 서비스(AWS service) 선택
    • 사용 사례: 검색창에 EKS 입력 후 나타나는 목록에서 EKS - Cluster 선택 ➔ [다음]
  2. 권한 추가: AmazonEKSClusterPolicy가 자동으로 추가되어 있는지 확인 ➔ [다음]
  3. 이름 지정, 검토 및 생성:
    • 역할 이름: eks-spoke-cluster-role 등 직관적인 이름 입력 ➔ [역할 생성]

1.2 노드 역할 (Node Role) 생성

워커 노드(EC2)가 EKS 제어 평면과 안정적으로 통신하고, ECR (Elastic Container Registry, 탄력적 컨테이너 레지스트리)에서 컨테이너 이미지를 다운로드하기 위해 필요합니다.

경로: AWS IAM 콘솔 ➔ [역할(Roles)] ➔ [역할 생성(Create role)]
  1. 신뢰할 수 있는 엔터티 선택:
    • 엔터티 유형: AWS 서비스(AWS service) 선택
    • 사용 사례: 검색창에 EC2 입력 후 EC2 선택 ➔ [다음]
  2. 권한 추가: 다음 3가지 정책을 검색하여 체크박스를 선택합니다.
    • AmazonEKSWorkerNodePolicy (노드 통신 권한)
    • AmazonEC2ContainerRegistryReadOnly (이미지 다운로드 권한)
    • AmazonEKS_CNI_Policy (네트워크 인터페이스 구성 권한)
  3. 이름 지정, 검토 및 생성:
    • 역할 이름: eks-spoke-node-role 등 입력 ➔ [역할 생성]
실전 팁 및 자주 실수하는 부분

노드 역할을 생성할 때, '사용 사례(Use case)'를 EKS가 아닌 반드시 EC2로 선택해야 합니다. 워커 노드의 실체는 EC2 인스턴스이므로, EC2가 해당 역할을 인스턴스 프로파일(Instance Profile) 형태로 부여받아야만 정상적으로 권한이 행사됩니다.

2. EKS 클러스터 생성 (제어 평면 구성)

IAM 준비가 완료되었다면 실제 클러스터의 제어 평면을 배포합니다. 이 단계에서 정의하는 네트워크 아키텍처가 보안의 핵심입니다.

경로: AWS EKS 콘솔 ➔ [클러스터(Clusters)] ➔ [클러스터 추가(Add cluster)] ➔ [생성(Create)]

2.1 클러스터 구성 (Configure cluster)

  • 이름: 클러스터의 고유 식별자 입력.
  • Kubernetes 버전: 가급적 최신 안정화 버전(Stable Version) 선택.
  • 클러스터 서비스 역할: 1.1단계에서 생성한 eks-spoke-cluster-role 선택.

2.2 네트워킹 지정 (Specify networking)

허브 앤 스포크 아키텍처의 설계 사상을 반영하는 가장 중요한 섹션입니다.

  • VPC: 워크로드가 배치될 스포크 VPC를 선택합니다.
  • 서브넷 (Subnets): 스포크 VPC 내의 프라이빗 서브넷(Private Subnet)을 최소 2개 이상의 가용 영역(Availability Zone)에 걸쳐 선택합니다.
  • 보안 그룹 (Security groups): 제어 평면과 워커 노드 간의 통신을 제어합니다. (특별한 요건이 없다면 EKS가 자동 생성하는 기본값 사용 권장)
  • 클러스터 엔드포인트 액세스 (Cluster endpoint access):
    • 퍼블릭(Public) 및 퍼블릭 및 프라이빗(Public and private)
    • 프라이빗(Private): 반드시 이것을 선택합니다.
* 클러스터 엔드포인트 프라이빗 (Private Endpoint): 이를 선택하면 스포크 VPC 내부, 또는 TGW (Transit Gateway, 전송 게이트웨이)로 연결된 허브 VPC/온프레미스 망에서만 쿠버네티스 API(제어 평면)에 접근할 수 있습니다. 엔터프라이즈 환경에서 외부의 악의적인 공격(예: 포트 스캐닝, API 탈취)을 원천 차단하기 위한 필수 보안 설정입니다.
그림 3: 허브 배스천을 통한 EKS 프라이빗 엔드포인트(Private Endpoint) 접근 흐름도

2.3 로깅 구성 (Configure logging)

  • 필요에 따라 API 서버, 감사(Audit), 인증자(Authenticator) 등의 로그 활성화 여부를 선택합니다.
  • 과금 주의: 로깅을 켜면 CloudWatch Logs로 데이터가 전송되며, 데이터 수집 및 보관에 따른 상당한 추가 비용이 발생할 수 있으므로 디버깅 목적 외에는 신중히 켜야 합니다.

3. 관리형 노드 그룹 (Managed Node Group) 추가

클러스터 상태가 Active로 변경(약 10~15분 소요)된 후, 실제 파드(Pod) 형태의 컨테이너 어플리케이션이 구동될 워커 노드를 생성합니다.

경로: EKS 클러스터 상세 페이지 ➔ [컴퓨팅(Compute)] 탭 ➔ [노드 그룹 추가(Add node group)]

3.1 노드 그룹 및 컴퓨팅 구성

  • 노드 IAM 역할: 1.2단계에서 생성한 eks-spoke-node-role 선택.
  • AMI 유형: Amazon Linux 2 또는 컨테이너에 최적화된 경량 OS인 Bottlerocket (보틀로켓) 중 선택.
  • 인스턴스 유형: 워크로드 특성(CPU/Memory 집약적 등)에 맞게 m5.large, c5.xlarge 등 선택.
    (*과금 주의: 선택한 인스턴스 스펙과 수량 단위로 요금이 청구됩니다.)
  • 노드 그룹 크기 조정: 오토스케일링(Auto Scaling)을 위한 최소, 최대, 원하는(Desired) 노드 수 지정.

3.2 네트워킹 지정 (Specify networking)

  • 서브넷: 외부 노출을 막기 위해 반드시 스포크 VPC 내의 프라이빗 서브넷만 지정합니다. (퍼블릭 서브넷은 선택 해제)
라우팅 테이블 확인 필수! (가장 잦은 장애 원인)

프라이빗 서브넷에 배치된 노드가 부팅되면서 EKS 제어 평면과 통신하거나 컨테이너 이미지를 받아오려면 아웃바운드 인터넷 연결이 필수적입니다. 따라서 스포크 VPC 프라이빗 서브넷의 라우팅 테이블(Route Table)에 0.0.0.0/0 트래픽이 허브 VPC의 NAT 게이트웨이(TGW를 경유)를 향하도록 올바르게 설정되어 있는지 반드시 확인하십시오. 이 라우팅이 막혀있으면 노드 생성은 완료되지만 클러스터에 합류하지 못해 영원히 NotReady 상태에 머물게 됩니다.

4. 필수 애드온(Add-ons) 내부 요소 설명

클러스터 및 노드 구성이 완료되면 EKS 상세 페이지의 [애드온(Add-ons)] 탭에서 다음 3가지 핵심 애드온이 정상적으로 작동하는지 확인해야 합니다.

그림 4: 워커 노드 내부의 필수 애드온 구성 및 상호작용 요소
  1. VPC CNI (Virtual Private Cloud Container Network Interface, 가상 사설 클라우드 컨테이너 네트워크 인터페이스)

    AWS 네트워크 환경과 쿠버네티스를 결합하는 핵심 요소입니다. 파드(Pod)에 가상 IP가 아닌 스포크 VPC 대역의 실제 프라이빗 IP 주소를 직접 할당합니다. 덕분에 파드는 VPC 내부의 일반적인 EC2 인스턴스처럼 취급되며, TGW 라우팅을 통해 다른 스포크 VPC의 리소스와 별도의 NAT 없이 오버레이 네트워크 오버헤드 없이 직접 통신할 수 있습니다.

  2. CoreDNS (코어디엔에스, 쿠버네티스용 내부 DNS 서버)

    클러스터 내부의 서비스 디스커버리(Service Discovery) 및 네임 리졸루션(Name Resolution)을 담당합니다. 파드가 my-service.default.svc.cluster.local과 같은 도메인 이름으로 다른 서비스를 찾을 수 있게 해줍니다.

  3. kube-proxy (큐브 프록시)

    각 워커 노드에서 데몬셋(DaemonSet) 형태로 실행됩니다. 쿠버네티스 '서비스(Service)' 개념을 구현하기 위해 노드의 네트워크 라우팅 규칙(일반적으로 iptables 또는 IPVS)을 관리하여, 서비스 IP로 들어온 트래픽을 백엔드의 실제 파드들로 로드밸런싱합니다.
