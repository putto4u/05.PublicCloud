중앙 집중식 보안 아키텍처 구축 가이드

본 가이드는 Nginx → VPC → TGW → NAT → ALB → NFW → IGW로 이어지는 복합 경로를 통해 외부 브라우저 접속이 가능한 고보안 웹 서비스를 구축하는 전문 직업훈련 과정의 핵심 교안입니다.

최고 수석 아키텍트 설계 보안 컴플라이언스 준수 엔터프라이즈 표준

1. 시스템 구성도 및 트래픽 흐름 (Traffic Flow)

외부 브라우저
▶
IGW
(인터넷 게이트웨이)
▶
NFW
(네트워크 방화벽)
▶
ALB
(로드 밸런서)
▶
TGW
(전송 게이트웨이)
▶
Nginx
(웹 서버)

[전문 아키텍트 분석]: 위 경로는 외부 사용자가 접속하는 '인바운드(Inbound)' 흐름을 기준으로 설명합니다. NAT (Network Address Translation, 네트워크 주소 변환) 게이트웨이는 인스턴스가 패치나 업데이트를 위해 인터넷으로 나가는 '아웃바운드(Outbound)' 전용이므로, 서비스 응답 트래픽의 복귀 경로 혹은 프라이빗 서버 자체의 외부 통신 시에 경유하게 됩니다.

2. 상세 구축 절차 (Step-by-Step)

단계 1: VPC 및 기본 네트워크 설계

VPC 자체는 무료

서비스가 운영될 Service-VPC와 보안/인터넷 접점이 될 Hub VPC (Inspection)를 각각 생성합니다.

콘솔 메뉴: VPC 콘솔 > VPC 생성 > 이름: Service-VPC / CIDR: 10.1.0.0/16
콘솔 메뉴: VPC 콘솔 > VPC 생성 > 이름: Hub-VPC / CIDR: 10.0.0.0/16

단계 2: TGW(전송 게이트웨이) 통합

시간당 점유 비용 + 데이터 처리 비용 발생

두 VPC를 연결하는 중앙 허브를 구축합니다.

콘솔 메뉴: VPC 콘솔 > Transit Gateways > Transit Gateway 생성
콘솔 메뉴: VPC 콘솔 > Transit Gateway Attachments > Attachment 생성 (Service-VPC 및 Hub-VPC 연결)

단계 3: ALB(애플리케이션 로드 밸런서) 및 Nginx 구성

ALB LCU 시간당 비용 발생

Hub VPC에 인터넷 가시성을 가진 `ALB`를 생성하고, `Service-VPC`의 프라이빗 서브넷에 있는 `Nginx` 서버를 대상 그룹으로 연결합니다.

콘솔 메뉴: EC 구 콘솔 > 로드 밸런서 > ALB 생성 > Internet-facing 선택
콘솔 메뉴: EC2 콘솔 > 대상 그룹 > Target Group 생성 > Nginx 인스턴스 등록

단계 4: NAT 게이트웨이 및 아웃바운드 라우팅

시간당 비용 + 데이터 처리 비용 발생

서버의 보안 업데이트 및 내부 통신을 위해 `NAT` 게이트웨이를 `Hub-VPC`의 지정된 서브넷에 배치합니다.

콘솔 메뉴: VPC 콘솔 > NAT Gateway > 생성 (탄력적 IP 할당 필요)

3. 실전 심화: 기존 환경에 NFW 투명하게 삽입하기

기존에 IGW → ALB로 구성되어 있던 인프라에 NFW (Network Firewall, 네트워크 방화벽)를 무중단에 가깝게 중간에 끼워 넣는(Bump-in-the-wire) 고도화된 VPC Ingress Routing (VPC 인그레스 라우팅) 기법을 설명합니다.

Step 3.1: Hub VPC 서브넷 구성 및 NFW 배포

시간당 방화벽 엔드포인트 비용 발생

NFW, ALB, NAT는 각각 분리된 전용 퍼블릭 서브넷에 위치해야 라우팅 제어가 가능합니다.

GUI 콘솔 작업:

  1. VPC 콘솔 > 서브넷 > 서브넷 생성을 클릭합니다.
  2. Hub VPC (10.0.0.0/16) 내에 다음 3개의 서브넷을 생성합니다:
    • NFW 전용 서브넷: 10.0.10.0/24
    • ALB 전용 서브넷: 10.0.20.0/24
    • NAT 전용 서브넷: 10.0.30.0/24
  3. VPC 콘솔 > AWS Network Firewall > 방화벽 생성으로 이동합니다.
  4. 방화벽 이름을 지정하고, Hub VPC와 방금 생성한 NFW 전용 서브넷(10.0.10.0/24)을 선택하여 방화벽을 프로비저닝합니다.
  5. 생성 완료 후, 방화벽 세부 정보에서 생성된 방화벽 엔드포인트 ID (VPCE ID, 예: vpce-01234abcd)를 반드시 메모합니다.

Step 3.2: NFW 보안 정책 (Firewall Policy) 및 규칙 설정

웹 서비스(80 포트) 통과 및 내부 인스턴스의 OS 업데이트(Outbound HTTPS)를 허용하는 상태 저장(Stateful) 규칙을 생성합니다.

GUI 콘솔 작업:

  1. Network Firewall > 방화벽 정책 > 방화벽 정책 생성에서 새 정책을 만듭니다.
  2. 상태 저장 규칙 그룹(Stateful Rule Group)을 생성합니다. (기본 Action은 Drop established 등 환경에 맞게 설정하되, 보통 명시적 Allow 규칙을 만듭니다.)
  3. 웹 서비스 Inbound 규칙 (5-Tuple):
    - 프로토콜: TCP / 소스: Any (0.0.0.0/0) / 소스 포트: Any
    - 대상: Internal VPC CIDRs (예: 10.0.0.0/16, 10.1.0.0/16) / 대상 포트: 80 / 작업: Pass (통과)
  4. 인스턴스 업데이트 Outbound 규칙 (Domain List):
    - 도메인 목록을 사용하여 .amazonaws.com, .ubuntu.com (운영체제별 상이) 등에 대한 HTTP/HTTPS 트래픽을 Pass 처리합니다.

Step 3.3: IGW 엣지 라우팅 연결 (Edge Association) - 핵심 기술

외부에서 들어오는 인터넷 트래픽이 ALB로 가기 전에 강제로 NFW를 거치도록 IGW 자체에 라우팅 테이블을 연결합니다.

GUI 콘솔 작업:

  1. VPC 콘솔 > 라우팅 테이블 > 라우팅 테이블 생성 (이름: rt-igw-edge).
  2. 생성한 라우팅 테이블의 라우팅 편집(Edit routes)을 클릭합니다.
  3. ALB 서브넷 보호: 대상(Destination)에 ALB 서브넷 CIDR (10.0.20.0/24), 타겟(Target)에 Step 3.1에서 메모한 Gateway Load Balancer Endpoint (NFW 엔드포인트)를 지정합니다.
  4. NAT 서브넷 보호 (선택): 외부에서 NAT로 직접 들어오는 것을 막거나 검사하려면 대상에 NAT 서브넷 CIDR (10.0.30.0/24), 타겟에 NFW 엔드포인트를 지정합니다.
  5. 저장 후, 엣지 연결(Edge Associations) 탭을 클릭하여 rt-igw-edge 테이블을 Internet Gateway (IGW)에 연결(Attach)합니다.

Step 3.4: 내부 서브넷별 상세 라우팅 테이블 재조정

트래픽이 루프(Loop)를 돌거나 끊기지 않도록 각 서브넷의 기본 라우팅(0.0.0.0/0)을 정교하게 제어합니다.

① NFW 서브넷 라우팅

검사가 끝난 트래픽을 인터넷으로 보내거나(Outbound), 목적지 서브넷으로 보냅니다.

  • 0.0.0.0/0 → IGW
  • ※ VPC 내부 대역(10.0.0.0/16)은 Local로 자동 라우팅됨

② ALB 서브넷 라우팅

응답 트래픽(Outbound)이 다시 NFW를 거쳐 나가도록 강제합니다.

  • 0.0.0.0/0 → NFW VPCE
  • Service VPC (10.1.0.0/16) → TGW

③ NAT 서브넷 라우팅

내부 인스턴스가 NAT를 타고 나갈 때도 NFW의 검사를 받도록 설정합니다.

  • 0.0.0.0/0 → NFW VPCE

Step 3.5: 보안 그룹 (Security Group) 최종 점검

NFW는 Layer 3/4에서 투명하게 동작하므로, IP 주소를 변경(NAT)하지 않습니다. 따라서 ALB 입장에서 클라이언트의 원본 IP가 그대로 들어옵니다.

  1. ALB 보안 그룹 (ALB-SG): 인바운드 규칙에 TCP 80 포트를 0.0.0.0/0 (Anywhere)에 대해 허용 유지합니다. (NFW에서 1차 차단을 수행하므로 SG는 개방해 두어야 원활히 수신됩니다.)
  2. Nginx 인스턴스 보안 그룹 (Web-SG): 인바운드 규칙에 TCP 80 포트를 ALB-SG의 보안 그룹 ID에 대해서만 허용합니다. (직접 접근 원천 차단)

4. 외부 접속 확인 (Verification)

  • 브라우저에 ALB의 DNS 이름을 입력하여 Nginx 초기 화면이 뜨는지 확인합니다.
  • NFW 로그(CloudWatch Logs)에서 해당 접속 IP의 패킷이 통과(Pass)되었는지 검증합니다.
  • Nginx 서버 내부 보안 그룹(Security Group)에서 ALB의 보안 그룹으로부터의 80포트 인입만 허용했는지 확인합니다.
