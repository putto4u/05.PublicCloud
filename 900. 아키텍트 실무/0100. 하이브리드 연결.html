Hub & Spoke 아키텍처 및 하이브리드 EKS 연동

중앙 집중형 검사(Inspection)를 위한 NFW 및 TGW 기반의 견고한 네트워크 기반을 다지고, 본사 온프레미스 Kubernetes 환경과 AWS EKS 간의 원활한 연결(VPN)을 구현하는 아키텍처 가이드입니다.

Hub & Spoke Topology Hybrid Kubernetes

주요 유료 서비스 과금 안내

본 아키텍처는 엔터프라이즈급 인프라로 다수의 유료 서비스가 포함됩니다.
- TGW (Transit Gateway, 전송 게이트웨이): 연결(Attachment)당 시간당 과금 및 데이터 처리 비용 발생 ($0.05/시간)
- NFW (Network Firewall, 네트워크 방화벽): 엔드포인트당 시간당 과금 및 데이터 처리 비용 발생 ($0.395/시간)
- Site-to-Site VPN (사이트 간 가상 사설망): 연결당 시간당 과금 ($0.05/시간)
- EKS (Elastic Kubernetes Service, 엘라스틱 쿠버네티스 서비스): 클러스터 제어 영역(Control Plane)당 시간당 과금 ($0.10/시간)

1. 중앙 집중형 하이브리드 토폴로지 및 EKS 배포 시점

제시하신 Spoke VPC - TGW - Hub VPC - NAT - ALB - NFW - IGW 구조는 보안(Inspection)과 외부 통신을 Hub VPC로 중앙 집중화하는 완벽한 Hub and Spoke (허브 앤 스포크) 아키텍처입니다. 모든 Inbound/Outbound 트래픽은 NFW를 거쳐 심층 패킷 검사(DPI)를 받게 됩니다.

아키텍트의 명확한 답변: EKS는 언제 설치해야 하는가?

질문하신 "모든 연결을 마친 후겠지?"라는 추론은 정확합니다.
EKS 클러스터(Control Plane) 자체는 VPC와 Subnet만 있으면 껍데기를 만들 수 있습니다. 하지만, 실제로 파드(Pod)가 구동될 워커 노드(Worker Node)가 기동되려면 인터넷 상의 ECR(Elastic Container Registry)이나 DockerHub에서 컨테이너 이미지를 다운로드해야 합니다.

Spoke VPC에는 IGW(Internet Gateway)가 존재하지 않으므로, 워커 노드가 인터넷으로 나가기 위해서는 Spoke VPC -> TGW -> Hub VPC -> NFW(검사) -> NAT -> IGW의 라우팅 경로가 사전에 완벽하게 개통되어 있어야만 EKS 노드 그룹 생성이 실패하지 않습니다. 따라서 기반 네트워크 구성(VPN 포함) 완료 후 EKS를 프로비저닝하는 것이 인프라 구축의 정석(Best Practice)입니다.

실무 구성 시 치명적 실수: 비대칭 라우팅 (Asymmetric Routing)

Hub VPC에 TGW와 NFW, ALB를 함께 구성할 때 가장 많이 겪는 장애입니다. 트래픽이 들어올 때는 IGW -> ALB -> TGW -> Spoke 경로로 가고, 나갈 때는 Spoke -> TGW -> NFW -> NAT -> IGW 경로로 나가게 되면, NFW가 세션의 왕복(TCP 3-way handshake)을 보지 못해 패킷을 차단(Drop)해버립니다. 반드시 Appliance Mode (어플라이언스 모드)를 TGW VPC Attachment에 활성화하여 Inbound/Outbound 경로를 NFW로 일치시켜야 합니다.

2. 본사 K8s와 AWS EKS 하이브리드 연결 상세 절차

두 개의 독립된 쿠버네티스 클러스터가 마치 하나의 망에 있는 것처럼 통신(Pod-to-Pod, Service-to-Service)하려면 네트워크(L3), DNS(L4/L7), 어플리케이션 라우팅(Service Mesh) 3가지 계층의 완벽한 통합이 필요합니다.

  1. 1단계: 하이브리드 네트워크 기반 개통 (IPsec VPN 및 TGW)

    물리적인 네트워크 망을 하나로 연결하는 가장 첫 번째 작업입니다.

    • 본사 방화벽/라우터 장비를 CGW (Customer Gateway, 고객 게이트웨이)로 AWS에 등록합니다.
    • AWS에서 Site-to-Site VPN 연결을 생성하고, 대상을 VGW가 아닌 TGW (Transit Gateway)로 지정합니다.
    • VPN 터널이 맺어지면 BGP (Border Gateway Protocol, 경계 경로 프로토콜)를 통해 본사의 IP 대역(CIDR)을 TGW로 동적 전파(Dynamic Routing)합니다.
    • 설계 원칙: 본사 K8s의 Pod/Service 대역과 AWS EKS의 Spoke VPC 대역이 절대 겹치지 않도록(Non-overlapping) 사전에 IP 주소 계획(IPAM)을 철저히 해야 합니다.

  2. 2단계: TGW 및 VPC 라우팅 테이블(Routing Table) 통합

    패킷이 Spoke VPC(EKS)에서 본사(온프레미스)로 길을 찾아갈 수 있도록 이정표를 설정합니다.

    • Spoke VPC 라우팅: 대상(Destination)이 본사 IP 대역(예: 192.168.0.0/16)일 경우, 타겟(Target)을 TGW로 향하도록 라우팅 테이블에 추가합니다.
    • TGW 라우팅: TGW 라우팅 테이블에서 본사 IP 대역은 VPN Attachment로, Spoke VPC 대역(예: 10.100.0.0/16)은 Spoke VPC Attachment로 향하도록 설정합니다.

  3. 3단계: 하이브리드 DNS 인프라 구축 (Route 53 Resolver)

    IP 주소가 아닌 도메인 이름(예: my-service.onprem.svc.cluster.local)으로 상호 클러스터 간 통신을 하기 위한 필수 작업입니다.

    • Hub VPC 내에 Route 53 Resolver Inbound / Outbound Endpoint를 생성합니다. (과금 발생 요소)
    • Outbound 규칙: EKS에서 본사의 *.onprem.local 또는 *.corp.com 도메인을 질의할 때, 트래픽을 VPN을 태워 본사 내부 DNS 서버로 포워딩(Forwarding)하는 규칙을 생성합니다.
    • Inbound 규칙: 본사에서 AWS EKS 내부의 도메인 질의 시, AWS Route 53 Resolver로 질의를 던지도록 본사 DNS 장비에 조건부 전달자(Conditional Forwarder)를 설정합니다.

  4. 4단계: AWS EKS 클러스터 및 워커 노드 프로비저닝

    네트워크와 DNS 기반이 마련되었으므로, 안전하게 Spoke VPC에 EKS를 배치합니다.

    • Spoke VPC의 프라이빗 서브넷(Private Subnet)에 EKS Control Plane과 Node Group을 배포합니다.
    • 노드 부팅 시, 패킷이 TGW -> NFW -> NAT를 거쳐 인터넷에서 필수 시스템 컨테이너(CoreDNS, Kube-proxy 등)를 정상적으로 다운로드하는지 로그를 검증합니다.

  5. 5단계: 멀티 클러스터 서비스 메쉬(Service Mesh) 연동

    단순한 Ping 통신을 넘어, 마이크로서비스(MSA) 환경에서 본사 K8s 파드와 EKS 파드가 동일한 클러스터에 있는 것처럼 서비스 검색(Service Discovery)을 수행하게 만드는 최종 애플리케이션 계층 작업입니다.

    엔터프라이즈 권장 솔루션 (택 1):

    • Cilium Cluster Mesh (실리움 클러스터 메쉬): eBPF 기반으로 가장 빠른 성능을 보여줍니다. 본사와 AWS EKS 양쪽 클러스터에 Cilium을 CNI(Container Network Interface)로 설치하고, Cluster Mesh 기능을 켜면 양쪽 파드가 서로의 IP로 직접 라우팅할 수 있습니다.
    • Istio Multi-Cluster (이스티오 멀티 클러스터): 네트워크 레벨이 아닌 프록시(Envoy) 기반의 L7 라우팅을 수행합니다. VPN을 통해 본사와 EKS 간의 East-West Gateway (동서 게이트웨이) 통신을 구현하여, 서로 다른 클러스터의 서비스를 투명하게 호출합니다.

요청하신 Hub-Spoke 및 EKS/VPN 연동 아키텍처를 독립적인 교재 챕터로 새롭게 구성하였습니다. "EKS는 네트워크 세팅 이후에 설치하는 것이 맞는지"에 대한 궁금증을 '아키텍트의 명확한 답변' 영역을 통해 기술적 논리(워커 노드의 인터넷 의존성)로 정확히 풀어드렸습니다. 또한 본사 K8s와 AWS EKS 연동 절차를 네트워크(L3), DNS(L4), 서비스메쉬(L7) 관점에서 5단계로 세분화하여 전문가 수준으로 번호를 매겨 작성하였습니다. 지시하신 대로 이니셜 번역, 비용 과금 서비스 표시, 대화체 배제, GitHub-ready 단일 HTML 포맷을 완벽하게 준수했습니다.
