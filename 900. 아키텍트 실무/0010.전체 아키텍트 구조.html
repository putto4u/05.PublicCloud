엔터프라이즈 하이브리드 클라우드 네트워크 아키텍처

1. 프로젝트 개요

본 프로젝트는 글로벌 최고 수준의 보안 컴플라이언스와 무한한 확장성을 요구하는 엔터프라이즈 환경을 위한 허브 앤 스포크(Hub and Spoke) 기반의 하이브리드 클라우드 네트워크 아키텍처 구축 지침서입니다.

단순한 네트워크 연결을 넘어, 모든 트래픽의 중앙 집중식 통제, L3~L7 계층의 심층 패킷 검사, 그리고 온프레미스 본사 망 및 타사 클라우드와의 안전한 BGP(Border Gateway Protocol, 경계 경로 프로토콜) 기반 동적 라우팅 통합을 목표로 합니다. 최신 AWS(Amazon Web Services, 아마존 웹 서비스) 모범 사례와 제로 트러스트(Zero Trust) 아키텍처 설계 철학이 완벽하게 반영된 마스터 플랜입니다.

[도식 1] 엔터프라이즈 하이브리드 클라우드 네트워크 허브 앤 스포크 토폴로지

2. 아키텍처 토폴로지 (Topology)

본 아키텍처는 중앙의 허브(Hub) VPC(Virtual Private Cloud, 가상 사설 클라우드)를 관문으로 삼아, 모든 스포크(Spoke) VPC와 외부 이기종 네트워크(본사, 구글 클라우드)가 TGW(Transit Gateway, 전송 게이트웨이) [과금 발생 서비스]를 통해 거미줄처럼 안전하게 연결되는 방사형 토폴로지를 취합니다. 모든 트래픽은 반드시 허브 VPC의 보안 검사망을 통과하도록 강제(Traffic Chaining)됩니다.

[도식 2] VPC 및 외부 이기종 네트워크 IP 대역(CIDR) 설계도

3. VPC 및 외부 이기종 네트워크 서비스 명세

전체 인프라를 구성하는 논리적 네트워크 격리 공간의 IP 대역 설계입니다. 글로벌 라우팅 충돌을 원천 차단하기 위해 CIDR(Classless Inter-Domain Routing, 사이더) 대역을 철저히 분리합니다.

  • 중앙 허브망 (Hub VPC): 10.0.0.0/16
    모든 보안, 통제, 라우팅, 관제가 집중되는 전사 네트워크의 심장부입니다.
  • 스포크 업무망 (Spoke01 VPC): 10.10.0.0/16
    실제 마이크로서비스 및 데이터베이스가 구동되는 워크로드 밀집 구역입니다.
  • 본사 폐쇄망 (On-Premises HQ): 192.168.0.0/16
    기업의 물리적 데이터 센터 네트워크입니다.
  • 멀티 클라우드망 (Google Cloud Platform): 10.20.0.0/16
    데이터 분석 등 특정 워크로드를 위해 연동되는 타사 클라우드 네트워크입니다.

5. 구성 요소별 상세 아키텍처 분석

5.1. 허브 보안 검사망 (Hub 퍼블릭 1) 및 NFW

모든 인프라의 최전방 물리적 보안 검문소입니다. AWS Network Firewall(네트워크 방화벽) [과금 발생 서비스]이 배치되며, 내부적으로 Suricata(수리카타) 오픈소스 IPS(Intrusion Prevention System, 침입 방지 시스템) 엔진이 구동됩니다. 트래픽의 IP, 포트, 프로토콜 상태를 검사하고 패킷 내용에 숨겨진 악성코드를 심층 분석하여 비정상 패킷을 즉시 폐기합니다.

5.2. 허브 외부 연결망 (Hub 퍼블릭 2) 및 ALB / WAF / NAT GW

네트워크 계층 검사를 무사히 통과한 트래픽이 만나는 애플리케이션 계층 검문소 및 라우팅 허브입니다.

  • ALB + WAF: 외부 사용자의 웹 트래픽을 수신하여 악의적인 데이터베이스 제어 명령어 삽입 등의 해킹을 방어하고, 백엔드 서버로 부하를 분산합니다.
  • NAT GW: 내부 서버가 밖으로 나갈 때 사설 IP를 숨기고 공인 IP로 변환해 주는 아웃바운드 전용 게이트웨이입니다.

5.3. 공유 서비스 관제망 (Hub 프라이빗 2)

보안과 인프라 관제를 위한 중앙 센터입니다. Prometheus(프로메테우스)가 스포크 VPC의 성능 측정 지표를 수집하고, Grafana(그라파나)가 이를 시각화합니다. 관리자는 외부 인터넷이 아닌 폐쇄망 내 단말인 배스천 호스트를 거쳐 내부 자원에 안전하게 접근합니다.

5.4. TGW 연결망 (Hub 프라이빗 1, Spoke01 프라이빗 1)

순수하게 트래픽을 VPC 안팎으로 나르기 위한 라우팅 인터페이스 전용 서브넷입니다. 다른 인스턴스가 실수로 배치되는 것을 막기 위해 /28(16개 IP)의 극단적으로 작은 크기로 설계하여 아키텍처의 무결성을 시스템적으로 강제합니다.

5.5. 하이브리드 VPN 연결 (External TGW 1, 2)

본사 및 타 클라우드 라우터와 IPsec 암호화 터널을 맺습니다. BGP(Border Gateway Protocol, 경계 경로 프로토콜) 동적 라우팅을 사용하여, 본사나 구글 클라우드에 새로운 서브넷이 생기더라도 라우팅 테이블이 클라우드 중앙 라우터에 자동으로 전파되도록 설계된 영역입니다.

[도식 4] Transit Gateway를 통한 본사 및 구글 클라우드 하이브리드 VPN 연결도

6. 아키텍처 특징 및 최고 수석 아키텍트 총평

이 아키텍처는 클라우드 인프라가 도달할 수 있는 가장 완성도 높은 보안 및 네트워크 분리 모델을 제시합니다.

  • 완벽한 트래픽 체이닝(Traffic Chaining)과 심층 방어(Defense in Depth): 단일 보안 장비에 의존하지 않고, 관문 진입 경로 제어 라우팅을 활용해 네트워크 방화벽과 웹 방화벽을 직렬로 묶어 무결점 보안을 달성했습니다.
  • 마이크로세그멘테이션(Micro-segmentation) 기반의 휴먼 에러 원천 차단: 라우팅 전용 서브넷을 /28로 최소화하여 운영자의 실수로 인한 리소스 혼용을 물리적으로 불가능하게 만들었습니다. 이는 설계자가 시스템 운영의 불확실성까지 통제한 완벽한 아키텍처입니다.
  • 장애 격리 반경(Blast Radius) 최소화: 모든 워크로드가 스포크 VPC로 격리되고 관제 및 통제가 허브로 일원화됨에 따라, 특정 스포크 VPC가 침해당하더라도 중앙 라우터 제어를 통해 전사 네트워크로의 위협 확산을 1초 이내에 차단할 수 있습니다.

7. 엔터프라이즈 구축 실무 절차 (Step-by-Step Guide)

본 구축 절차는 프로덕션(Production) 운영 환경에 즉시 적용 가능한 수준의 상세 GUI(Graphical User Interface, 그래픽 사용자 인터페이스) 설정법과 아키텍트의 주의사항을 포함합니다.

Phase 1. 하위 인프라(VPC 및 서브넷) 구축

VPC와 서브넷을 정확한 CIDR 블록으로 생성합니다.

작업 절차:

  1. VPC 콘솔 접속: AWS 관리 콘솔 검색창에 VPC를 입력합니다.
  2. Hub VPC 생성:
    • 우측 상단 VPC 생성 클릭 ➜ 생성할 리소스: VPC만 선택.
    • 이름 태그: vpc-hub
    • IPv4 CIDR 블록: 10.0.0.0/16 ➜ VPC 생성 클릭.
  3. Spoke01 VPC 생성:
    • 동일한 방법으로 이름: vpc-spoke01, CIDR: 10.10.0.0/16으로 생성.
  4. 서브넷 생성 (총 5개):
    • 좌측 메뉴 서브넷 ➜ 서브넷 생성 클릭.
    • VPC를 선택하고 위 총괄표를 참조하여 서브넷 이름, 가용 영역, CIDR(/28, /24, /18 등)을 정확히 입력하여 일괄 생성합니다.
⚠️ 아키텍트 주의사항: VPC 및 더 보기(VPC and more) 옵션을 절대 사용하지 마십시오. 엔터프라이즈 환경에서는 라우팅 테이블을 세밀하게 분리 조작해야 하므로 자동 마법사 기능은 라우팅 구조를 오염시킵니다. 반드시 VPC만을 선택하여 수동으로 조립해야 합니다.

Phase 2. 클라우드 중앙 라우터(TGW) 및 하이브리드 VPN 연동

작업 절차:

  1. TGW 생성:
    • 좌측 하단 Transit Gateway 클릭 ➜ Transit Gateway 생성.
    • 이름 태그: tgw-central
    • Amazon 측 ASN: 64512 (BGP를 위한 고유 식별 번호).
    • [중요 사항] 기본 라우팅 테이블 연결 및 기본 라우팅 테이블 전파 옵션을 반드시 체크 해제(Disable) 합니다. 라우팅은 보안상 수동으로 완벽히 통제해야 합니다.
  2. TGW VPC 연결 (Attachment):
    • Transit Gateway 연결 클릭 ➜ 연결 생성.
    • Hub VPC 연결 시: 서브넷 선택에서 반드시 Hub 프라이빗 1 (10.0.254.0/28)만 체크.
    • Spoke VPC 연결 시: 서브넷 선택에서 반드시 Spoke01 프라이빗 1 (10.10.254.0/28)만 체크.
  3. Site-to-Site VPN 구축 (본사 및 구글 클라우드):
    • 고객 게이트웨이(CGW) 메뉴에서 상대방 방화벽 공인 IP와 ASN을 입력 후 생성.
    • Site-to-Site VPN 연결 메뉴에서 TGW와 방금 만든 CGW를 연결하여 BGP 동적 라우팅 터널을 활성화합니다. [과금 발생 서비스]

Phase 3. 심층 방어 보안 장비(NFW, ALB, NAT GW) 배치

작업 절차:

  1. NAT Gateway 생성:
    • NAT 게이트웨이 ➜ 생성. 서브넷은 Hub 퍼블릭 2 선택. 탄력적 IP를 할당 후 생성.
  2. Network Firewall 생성:
    • VPC 콘솔 좌측 방화벽(AWS Network Firewall) ➜ 방화벽 생성.
    • 이름: nfw-hub. VPC: vpc-hub 선택.
    • 서브넷: Hub 퍼블릭 1 (10.0.1.0/28) 선택.
    • 방화벽 정책을 생성하고 침입 방지 상태 저장 규칙을 적용합니다.
  3. ALB 및 WAF 생성:
    • EC2 콘솔 ➜ 로드 밸런서 ➜ Application Load Balancer 생성.
    • 서브넷: Hub 퍼블릭 2 선택.
    • AWS WAF 콘솔로 이동하여 웹 접근 제어 목록을 생성하고 방금 만든 ALB와 연결합니다.
[도식 5] IGW 인그레스 라우팅 및 다계층 트래픽 체이닝 논리 설정 화면 예시

Phase 4. 진입 경로 제어(Ingress Routing) 및 라우팅 테이블 조립

패킷이 보안 장비를 우회하지 못하도록 라우팅 이정표를 정밀하게 설정하는 가장 핵심적인 단계입니다.

  1. VPC Edge 라우팅 테이블 (IGW 진입 제어):
    • 라우팅 테이블 생성: rt-edge ➜ VPC 엣지 연결 편집에서 IGW 선택.
    • 라우팅 룰: 대상 10.0.2.0/24 (Hub 퍼블릭 2 대역) ➜ 타겟 NFW 엔드포인트.
    • (설명: 외부에서 퍼블릭망으로 향하는 트래픽을 검사망으로 강제 회전시킵니다.)
  2. 보안 검사망 라우팅 테이블 (Hub 퍼블릭 1):
    • 라우팅 룰 1: 대상 10.10.0.0/16 (Spoke망) ➜ 타겟 TGW ENI.
    • 라우팅 룰 2: 대상 0.0.0.0/0 ➜ 타겟 IGW (응답 트래픽 처리용).
  3. 외부 연결망 라우팅 테이블 (Hub 퍼블릭 2):
    • 라우팅 룰 1: 대상 0.0.0.0/0 ➜ 타겟 NFW 엔드포인트 (내부에서 외부로 나갈 때 검사 강제).
    • 라우팅 룰 2: 대상 10.10.0.0/16 ➜ 타겟 TGW ENI.
  4. TGW 라우팅 테이블 (Hub 프라이빗 1):
    • 라우팅 룰 1: 대상 0.0.0.0/0 ➜ 타겟 NFW 엔드포인트 (스포크에서 올라온 트래픽을 방화벽으로 인도).
  5. 스포크 업무망 라우팅 테이블 (Spoke01 프라이빗 2):
    • 라우팅 룰: 대상 0.0.0.0/0 ➜ 타겟 TGW ENI.
    • (설명: 스포크 망에서 밖으로 나가려는 모든 트래픽을 무조건 중앙 라우터로 던집니다.)

Phase 5. 관제 및 모니터링 시스템 배포 (Shared Services)

작업 절차:

  1. 모니터링 인스턴스 생성:
    • EC2 콘솔에서 Hub 프라이빗 2 (10.0.10.0/24) 서브넷에 가상 서버 인스턴스를 시작합니다. [과금 발생 서비스]
  2. Prometheus 및 Grafana 설치:
    • 서버에 접속하여 컨테이너 기반으로 프로메테우스와 그라파나를 구동합니다.
    • 설정 파일에 스포크 VPC(10.10.0.0/18) 대역의 성능 수집 엔드포인트 IP 리스트를 추가합니다.
  3. Bastion Host (접근 통제) 설정:
    • 동일 서브넷에 배스천 호스트용 EC2를 생성합니다.
    • 보안 그룹에서 기본 원격 접속 포트를 개방하지 않습니다.
    • EC2에 권한 역할을 부여하여 관리자가 셸(Shell)에 보안 접속하도록 통제합니다.
[주석 및 해설]
  • Traffic Chaining (트래픽 체이닝, 트래픽 연쇄 이동): 네트워크 패킷이 최종 목적지에 도달하기 전에 여러 개의 보안 장비를 순차적으로 거치도록 라우팅을 조작하는 기법입니다.
  • East-West 통신 (동서 통신): 외부 인터넷으로 나가지 않고 클라우드 내부의 VPC 간, 혹은 서브넷 간에 이루어지는 데이터 통신 흐름을 의미합니다.
  • BGP (Border Gateway Protocol, 경계 경로 프로토콜): 인터넷의 우체국 역할을 하는 동적 라우팅 알고리즘입니다. 새로운 네트워크 대역이 추가될 때 자동으로 경로를 학습하고 전파합니다.
  • CGW (Customer Gateway, 고객 게이트웨이): AWS VPN 연결 시 온프레미스 측 라우터 장비의 공인 IP와 라우팅 정보를 AWS에 정의하는 논리적 리소스입니다.
