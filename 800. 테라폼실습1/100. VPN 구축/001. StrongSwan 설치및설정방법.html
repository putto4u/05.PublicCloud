단일 온프레미스 VPN 기반
다중 AWS EKS 클러스터 연결 아키텍처

초격차 인프라 설계를 위한 라우팅 전략 및 클라이언트 구성 가이드

단일 VPN 게이트웨이 다중 클러스터 연결 및 라우팅 전략

온프레미스(On-Premises, 자체 보유 서버 및 네트워크 환경)에 단일 VPN 게이트웨이(VPN Gateway, 가상 사설망 연결 장비 - 예: strongSwan 등)를 구축하고, 내부 네트워크의 여러 PC가 독립적인 여러 AWS 환경의 K8s(Kubernetes, 컨테이너 오케스트레이션 시스템) 클러스터에 접속하기 위해서는 고도화된 네트워크 라우팅(Routing, 데이터 패킷의 경로 설정) 설계가 필요합니다.

결론적으로 각 PC의 기본 게이트웨이(Default Gateway, 외부 네트워크로 나가는 기본 출입구)를 변경할 필요는 없으며, 다음 방식을 통해 트래픽을 제어해야 합니다.

옵션 A: 메인 라우터 정적 라우트 (권장)

내부 네트워크의 L3 스위치(Layer 3 Switch, 라우팅 기능이 포함된 네트워크 스위치) 라우팅 테이블(Routing Table, 경로 지정 표)에 AWS 클라우드 대역으로 가는 패킷의 다음 경로(Next Hop)를 온프레미스 VPN 서버로 지정합니다.

  • 개별 PC 설정 불필요, 중앙 집중식 관리.
  • 타겟 대역 전체(예: 10.0.0.0/8) 포워딩.

옵션 B: 각 PC 정적 라우트 개별 추가

메인 장비 제어가 불가능할 경우, 각 PC의 OS(Operating System, 운영체제)에 목적지 대역으로 가는 경로만 VPN으로 지정하는 분할 터널링(Split Tunneling)을 구현합니다.

C:\> route add 10.1.0.0 mask 255.255.0.0 <VPN_IP> -p

* 기본 게이트웨이를 VPN으로 통째로 변경하는 방식(옵션 C)은 일반 인터넷 트래픽까지 터널을 거치게 되어 대역폭 저하 및 NAT(Network Address Translation, 네트워크 주소 변환) 구성 복잡성을 초래하므로 지양합니다.

CIDR 설계 및 네트워크 대역 중복(Routing Conflict) 방지

다수의 접속 환경이 동일한 10.0.0.0/16 대역의 VPC(Virtual Private Cloud, 가상 사설 클라우드)를 사용할 경우, 중앙 VPN 라우터는 패킷의 목적지를 식별할 수 없어 치명적인 라우팅 충돌(Routing Conflict)이 발생합니다. 서브넷 크기(/16)가 아닌 주소 체계(Network ID) 자체가 달라야 합니다.

옥텟(Octet) 분할 아키텍처 원리

IPv4(Internet Protocol version 4) 주소 체계에서 두 번째 옥텟(8비트 데이터 블록)을 사용자별로 고유하게 할당하여 완벽한 격리를 구현합니다.

  • 환경 A: 10.1.0.0/16 (사용 범위: 10.1.0.0 ~ 10.1.255.255)
  • 환경 B: 10.2.0.0/16 (사용 범위: 10.2.0.0 ~ 10.2.255.255)
  • 환경 C: 10.3.0.0/16 (사용 범위: 10.3.0.0 ~ 10.3.255.255)

실전 팁 및 자주 오해하는 부분

  • 기본(Default) VPC 사용 금지: AWS 콘솔이 제공하는 172.31.0.0/16 기본 대역을 그대로 사용하면 모든 환경이 중복됩니다. 반드시 고유 CIDR(Classless Inter-Domain Routing, IP 주소 할당을 위한 블록 지정 방식)을 지닌 새 VPC를 생성해야 합니다.
  • EKS CNI 대역 충돌: AWS EKS는 VPC CNI(Container Network Interface, 컨테이너 네트워크 인터페이스)를 사용하므로 파드(Pod) IP가 VPC 실제 서브넷 IP와 동일합니다. 온프레미스 망(예: 192.168.x.x)과 클라우드 대역이 겹치지 않는지 교차 검증해야 합니다.

strongSwan Windows 전용 클라이언트 앱 다운로드 및 설정 가이드

윈도우 내장 VPN 기능 외에도, 복잡한 레지스트리 설정이나 어댑터 수정 없이 직관적으로 연결을 관리할 수 있는 strongSwan 공식 전용 클라이언트 앱(UWP, 범용 윈도우 플랫폼 애플리케이션)을 사용하는 것이 실무에서 널리 권장됩니다. 이 앱은 인증서 관리 및 IKEv2 터널링 구성을 GUI(Graphical User Interface, 그래픽 사용자 인터페이스) 환경에서 한 번에 해결해 줍니다.

1

공식 다운로드 사이트 및 설치 방법

strongSwan 앱은 윈도우 스토어를 통해 가장 안전하고 무결성 있게 설치할 수 있습니다.

  • 접속 방법 A (가장 권장): Windows 작업 표시줄에서 'Microsoft Store'를 실행하고 검색창에 strongSwan VPN Client를 검색하여 '다운로드(Get)' 버튼을 클릭합니다.
  • 접속 방법 B (웹 URL 연결): 웹 브라우저에서 공식 앱스토어 링크인 https://apps.microsoft.com/store/detail/strongswan/9WZDNCRDFBBG 에 접속하여 스토어 앱을 호출합니다.
  • 접속 방법 C (MSI 패키지 수동 다운로드): 폐쇄망 등 스토어 접속이 불가능한 경우, 공식 릴리즈 사이트인 https://download.strongswan.org/Windows/ 에 접속하여 최신 버전의 .msi (Microsoft Windows Installer, 마이크로소프트 윈도우 설치 패키지) 파일을 직접 다운로드 후 더블 클릭하여 설치합니다.
2

프로필 설정 및 메뉴 선택 절차

설치된 strongSwan 앱을 실행하고, 인프라 관리자가 제공한 접속 정보와 인증서를 바탕으로 연결 프로필(Connection Profile)을 생성합니다.

  1. 앱 실행 및 프로필 추가: 바탕화면이나 시작 메뉴에서 strongSwan 앱을 실행한 후, 메인 화면 우측 상단의 Add connection (연결 추가) 버튼을 클릭합니다.
  2. 서버 정보 입력 (Server): 연결 대상 VPN 게이트웨이의 주소를 입력합니다.
    • ▶ Server (서버): vpn.domain.com 또는 관리자가 할당한 공인 IP 주소 입력
  3. 인증 방식 선택 (VPN Type): 서버에 구축된 보안 정책에 맞춰 VPN 유형을 드롭다운 메뉴에서 선택합니다.
    • ▶ VPN Type (VPN 유형): 아이디/패스워드 방식일 경우 IKEv2 EAP (Username/Password) 선택. 인증서 전용 접속일 경우 IKEv2 Certificate 선택.
  4. 로그인 자격 증명 (Credentials): 앞서 선택한 EAP 방식에 따라 계정 정보를 기입합니다.
    • ▶ Username (사용자 이름): 부여받은 접속 ID 입력
    • ▶ Password (비밀번호): 부여받은 접속 비밀번호 입력
  5. 서버 인증서 지정 (CA Certificate) [매우 중요]: 서버의 신뢰성을 검증하기 위한 단계입니다.
    • ▶ CA certificate (인증 기관 인증서): 기본값은 'Select automatically (자동 선택)'이지만, 사설 인증서를 사용하는 실무 환경에서는 Select (선택) 버튼을 클릭하여 로컬 PC에 저장해둔 ca-cert.pem 파일을 직접 찾아 지정해주어야 합니다. (이 과정을 거치면 윈도우 OS 인증서 저장소에 수동으로 설치하는 복잡한 과정이 생략됩니다.)
  6. 프로필 저장 및 연결:
    • ▶ Profile name (프로필 이름): AWS EKS 실습망 등 식별하기 쉬운 이름을 기입합니다.
    • ▶ 우측 상단의 Save (저장) 버튼을 눌러 설정을 완료하고, 생성된 프로필을 클릭하여 접속을 개시합니다.

* 주석: strongSwan 전용 클라이언트 앱은 윈도우 내장 어댑터에서는 설정하기 까다로운 강력한 ESP(Encapsulating Security Payload, 캡슐화 보안 페이로드) 알고리즘(예: AES-GCM)을 자동으로 매칭하여 협상하므로, 엔터프라이즈 환경에서 압도적인 보안성과 편의성을 자랑합니다.

참고: Windows 내장 기능(Agile VPN)을 활용한 수동 구성 방법

만약 사내 보안 정책 등의 이유로 Microsoft Store 앱 및 외부 프로그램 설치가 전면 차단되어 있다면, Windows OS에 내장된 Agile VPN(애자일 VPN, 마이크로소프트 내장 네트워크 모듈)을 통해 IKEv2 터널을 수동으로 구성할 수 있습니다.

1

보안 인증서 수동 등록

ca-cert.pem 실행 ➔ '인증서 설치' ➔ 저장소 위치를 반드시 '로컬 컴퓨터'로 지정 ➔ '신뢰할 수 있는 루트 인증 기관' 폴더에 저장.

2

어댑터 보안 격상 설정

네트워크 어댑터 속성 ➔ 보안 탭 ➔ 데이터 암호화를 최대 강도 암호화로 변경 ➔ EAP-MSCHAP v2 지정 필수.

클라우드 인프라 과금 (Billing) 및 유료 전환 주의 사항

본 아키텍처를 구현할 때 다음 서비스들에서 과금이 발생하거나 비용이 증폭될 수 있으므로 철저한 예산 통제가 필요합니다.

  • AWS Data Transfer Out (데이터 송신 비용): PC에서 VPN 터널을 통해 K8s 파드의 대용량 로그나 데이터를 다운로드할 경우, 외부 인터넷 아웃바운드와 동일하게 GB당 과금이 발생합니다.
  • NAT Gateway & Elastic IP 유지비: 독립된 VPC마다 외부 통신을 위한 NAT 게이트웨이를 구성하면 인스턴스 시간당 요금과 트래픽 처리 요금이 이중 부과됩니다. 실습 후 삭제 시 EIP(탄력적 공인 IP)까지 반드시 반납(Release)해야 페널티 과금을 막을 수 있습니다.
  • 다중 VPN 연결 및 Transit Gateway: VPC별로 독립적인 Site-to-Site VPN 터널을 맺거나 AWS Transit Gateway(전송 게이트웨이)로 허브 앤 스포크(Hub & Spoke)를 구성할 경우, 연결된 Attachment 개수 및 처리량에 비례하여 유료 과금이 발생합니다.

본 교재는 IP 직업훈련 전문가 과정 아키텍처 설계를 위해 작성되었습니다.
