Hub & Spoke 기반
Local PC 통합 VPN 마스터

작업자의 개별 로컬 PC(또는 단일 VM)에 strongSwan을 직접 설치하여 AWS TGW(Transit Gateway, 전송 게이트웨이) 및 S2S VPN(Site-to-Site VPN, 사이트 간 가상 사설망)과 다이렉트로 연동하는 V2 아키텍처 기술 명세서입니다. 별도의 중앙 라우터 없이 단일 노드에서 프라이빗 EKS 환경을 구축합니다.

설명: Version 1이 사내망 전체를 아우르는 중앙 집중형 VPN 라우터를 구축하는 방식이었다면, Version 2는 개발자나 엔지니어의 개별 PC 자체를 VPN 엔드포인트(Endpoint, 종단점)로 구성하여 AWS 인프라와 1:1로 안전하게 통신하는 구조입니다.

2. 로컬 K8s 통신을 위한 IP 포워딩 활성화

현재 PC 내부에서 구동되는 K8s Pod(파드)들의 가상 네트워크 트래픽이 외부(AWS)로 나가는 VPN 터널을 탈 수 있도록 커널의 포워딩 기능을 반드시 활성화해야 합니다.

COPY 
# sysctl.conf 파일 수정

# 설정 즉시 적용
sudo sysctl -p

💡 아키텍트 실전 팁 - NAT Traversal (NAT-T, 네트워크 주소 변환 순회):
로컬 PC가 공유기 하단(사설 IP)에 위치하더라도, strongSwan은 UDP 4500 포트를 이용한 NAT-T 기능을 통해 AWS VPN과 성공적으로 터널을 협상할 수 있습니다. 단, 로컬 PC 측에서 먼저 통신(Ping 등)을 발생시켜 터널을 깨워야(Initiate) 방화벽을 원활하게 통과합니다.

설명: Version 1에서는 다른 PC들의 트래픽을 대신 전달하기 위해 포워딩을 켰다면, Version 2에서는 내 PC 안에 있는 컨테이너(Docker, K8s)들의 트래픽이 물리적 네트워크 인터페이스를 거쳐 가상 터널로 빠져나가게 하기 위해 포워딩(IP Forwarding)을 활성화합니다.

strongSwan 설정 파일(ipsec.conf) 구성 팁

AWS에서 다운로드한 설정 파일을 바탕으로 로컬 PC의 /etc/ipsec.conf를 편집할 때, leftsubnet 부분을 반드시 아래와 같이 확장해야 K8s 파드 통신이 가능해집니다.

COPY 
conn Tunnel1
    # ... (인증 및 암호화 설정 생략) ...
    left=192.168.1.50                 # 로컬 PC의 사설 IP
    leftid=203.0.113.50               # 로컬 PC망의 공인 IP (CGW IP)
    right=[AWS_TUNNEL_1_PUBLIC_IP]    # AWS 터널 공인 IP
    
    # [핵심] 호스트 IP와 K8s CNI 대역을 콤마(,)로 연결하여 모두 명시
    leftsubnet=192.168.1.50/32,10.244.0.0/16
    
    rightsubnet=10.251.0.0/16         # 접근할 AWS 스포크 VPC 대역
    auto=start                        # 서비스 시작 시 터널 자동 연결
정책 기반 라우팅(Policy-Based Routing): strongSwan은 리눅스 커널의 XFRM 프레임워크를 사용하여 라우팅 테이블이 아닌 보안 정책을 기반으로 패킷을 낚아챕니다. 따라서 leftsubnet과 rightsubnet에 명시된 규칙만이 암호화되어 터널로 들어갑니다.

Phase 4. 로컬 Kubernetes(K8s) 통신 검증 및 보안 그룹 오픈

로컬 PC에서 구동 중인 Minikube나 kubeadm 클러스터의 파드들이 터널을 거쳐 AWS EKS 자원(또는 프라이빗 EC2)과 통신할 수 있도록 방화벽(Security Group)을 해제합니다.

4-1 AWS 보안 그룹(Security Group) 인바운드 허용

AWS 콘솔에서 스포크 VPC 내 목적지 리소스의 보안 그룹을 수정하여, 로컬 K8s의 IP 대역을 신뢰할 수 있도록 추가합니다.

EC2 콘솔 ➔ 네트워크 및 보안 ➔ 보안 그룹(Security Groups) ➔ 인바운드 규칙 편집
유형 (Type)

10.244.0.0/16 (로컬 K8s CNI 대역)
192.168.1.50/32 (로컬 PC 호스트 IP)
설명: 클라우드 방화벽인 보안 그룹은 기본적으로 모든 인바운드(외부에서 내부로 들어오는) 트래픽을 차단하므로, 로컬 환경에서 사용하는 사설 IP 대역을 명시적으로 허용해 주어야 합니다.

4-2 로컬 K8s 파드 내에서 통신 테스트 수행

내 PC의 K8s 클러스터에 테스트 파드를 띄우고, AWS 리소스를 향해 직접 트래픽을 쏘아 터널의 건강 상태를 진단합니다.

COPY 
# 1. 터미널을 열고 테스트용 컨테이너 생성
kubectl run vpn-test-pod --image=busybox --restart=Never -- sleep 3600

# 2. 파드 내부에 진입하여 AWS 스포크 VPC의 사설 IP로 핑 발송
kubectl exec -it vpn-test-pod -- ping 10.251.10.100

# 정상 통신 결과 예시:
# 64 bytes from 10.251.10.100: seq=0 ttl=253 time=15.4 ms
# 64 bytes from 10.251.10.100: seq=1 ttl=253 time=14.8 ms
아키텍처 검증 성공: 파드 내부에서 발생한 트래픽이 호스트(내 PC)의 커널로 나와 XFRM 정책에 의해 암호화된 후, 공인망을 거쳐 AWS S2S VPN으로 인입되어 타겟 자원까지 성공적으로 라우팅되었음을 증명합니다.

Phase 5. 스포크 라우팅 구성 및 프라이빗 EKS 접근

🚨 실무 자원 회수 (Clean-up) 순서

  1. EKS 클러스터 삭제: 노드 그룹(Node Group) 및 로드밸런서(ELB, Elastic Load Balancing, 탄력적 로드 밸런싱) 선행 제거 필수.
  2. VPN 연결 삭제: 사이트 간 VPN을 삭제해야 연관된 고객 게이트웨이(CGW) 삭제 가능.
  3. TGW 연결(Attachment) 삭제: VPC 및 VPN 연결 리소스를 모두 분리.
  4. Transit Gateway 본체 삭제: 모든 연결 해제 확인 후 최종 제거.
  5. VPC 환경 삭제: NAT Gateway(Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이), Elastic IP(탄력적 IP) 정리.
