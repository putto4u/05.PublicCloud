TGW & S2S VPN 통합 마스터

AWS TGW(Transit Gateway, 전송 게이트웨이)와 S2S VPN(Site-to-Site VPN, 사이트 간 가상 사설망)을 결합하여 대규모 다중 VPC(Virtual Private Cloud, 가상 사설 클라우드) 환경을 구축하고, 10.251.0.0/16 대역의 스포크를 연동하는 전 과정에 대한 기술 명세서입니다.

Phase 0. 온프레미스 환경 준비: VPN 라우터 및 클라이언트 설정

0-1 중앙 PC (VPN 라우터) strongSwan 다운로드 및 설치

사내망 트래픽을 모아서 AWS로 보내줄 중앙 리눅스 서버(Ubuntu 기준)에 오픈소스 IPsec VPN 솔루션인 strongSwan을 설치합니다.

VPN 라우터 기본 패키지 구성 Free OpenSource

1. 패키지 업데이트 및 strongSwan 설치

COPY 
sudo apt-get update -y
sudo apt-get install strongswan strongswan-pki libcharon-extra-plugins libcharon-extauth-plugins libstrongswan-extra-plugins -y
설명: strongSwan 본체와 함께 AWS와의 원활한 IKEv2 인증 및 라우팅 교환을 위해 필요한 추가 플러그인(plugins)들을 일괄 설치합니다.

2. IP 포워딩(IP Forwarding, 패킷 전달) 활성화

중앙 PC가 일반 클라이언트가 아닌 라우터 역할을 하려면, 들어온 네트워크 패킷을 다른 곳(AWS)으로 넘겨주는 포워딩 기능이 켜져 있어야 합니다.

COPY 
# sysctl.conf 파일 수정
sudo sed -i 's/#net.ipv4.ip_forward=1/net.ipv4.ip_forward=1/' /etc/sysctl.conf

# 설정 즉시 적용
sudo sysctl -p

💡 Pro Tip: 지금은 기본 뼈대만 설치한 상태입니다. 실제 AWS와 연결하기 위한 ipsec.conf (터널 설정 파일) 및 ipsec.secrets (사전 공유 키) 구성은 Phase 2에서 AWS VPN 생성 후 제공받는 구성 파일을 바탕으로 입력하게 됩니다.

0-2 내 PC (사내망 클라이언트) 라우팅 설정 및 통신 확인

여러분의 실습용 PC(Windows 기준)에서 AWS 대역(예: 10.0.0.0/8)으로 향하는 트래픽이 사내 중앙 VPN 라우터를 거쳐가도록 정적 라우팅을 추가해야 합니다.

Windows 정적 라우팅 추가 (Static Route Add)

※ 반드시 '관리자 권한'으로 명령 프롬프트(CMD)를 실행해야 합니다.

COPY 
# 구문: route add [대상 네트워크] mask [서브넷 마스크] [게이트웨이 IP] -p
# 가정: 사내망 중앙 VPN 라우터 IP가 192.168.1.254 이고, AWS 스포크 대역이 10.251.0.0/16 인 경우

route add 10.251.0.0 mask 255.255.0.0 192.168.1.254 -p
명령어 해석: 내 PC에서 10.251.x.x 대역으로 가는 모든 패킷은, 인터넷 공유기가 아니라 사내망 중앙 VPN 라우터(192.168.1.254)로 던지라는 의미입니다. -p 옵션은 재부팅 후에도 설정이 영구적으로(persistent) 유지되게 합니다.

접속 확인 방법 (Phase 3 완료 후 테스트)

COPY 
# 1. 라우팅 테이블이 정상적으로 등록되었는지 확인
route print | findstr "10.251"

# 2. AWS 스포크 VPC 내부 자원에 대한 도달성 테스트 (ICMP Ping)
ping 10.251.10.100

# 3. 경로 추적을 통해 패킷이 사내 VPN 라우터를 거쳐가는지 확인
tracert 10.251.10.100

Phase 4. 온프레미스 Kubernetes(K8s)의 AWS 리소스 접근 및 통신 검증

학원 내 여러분의 개인 PC에 설치된 K8s 클러스터(예: Minikube, kubeadm 기반)에서 생성된 파드(Pod)가, VPN 터널을 타고 AWS 스포크 VPC 내부 자원에 접근할 수 있도록 보안을 열어주고 통신을 검증하는 매우 중요한 단계입니다.

4-1 작업 1: 온프레미스 K8s CNI 대역 확인

내부 파드들이 사용하는 가상 네트워크 IP 대역(CNI, Container Network Interface)을 알아야 AWS 방화벽에서 허용해 줄 수 있습니다.

COPY 
# 캘리코(Calico) 플러그인을 사용하는 경우 기본 대역 확인 명령어
kubectl get ippools -o wide

# 또는 플란넬(Flannel) 등을 포함한 일반적인 클러스터 정보 확인
kubectl cluster-info dump | grep -i cidr
결과 예시: IP 대역이 192.168.0.0/16 또는 10.244.0.0/16 등으로 출력됩니다. 이 값을 잘 메모해 둡니다.

4-2 작업 2: AWS 보안 그룹(Security Group) 클릭 및 작업 단위 허용 설정

AWS 콘솔에 접속하여 스포크 VPC 내에 있는 리소스(예: 프라이빗 EC2, RDS 등)의 가상 방화벽을 열어줍니다.

EC2 콘솔 ➔ 네트워크 및 보안 ➔ 보안 그룹(Security Groups)
인바운드 규칙 편집 (Edit Inbound Rules) 단계별 가이드
  1. EC2 보안 그룹 목록에서 스포크 VPC에 속한 타겟 리소스의 [보안 그룹 ID]를 클릭합니다.
  2. 화면 하단의 탭 메뉴 중 [인바운드 규칙(Inbound rules)]을 선택합니다.
  3. 우측 상단의 [인바운드 규칙 편집(Edit inbound rules)] 버튼을 클릭합니다.
  4. 좌측 하단의 [규칙 추가(Add rule)] 버튼을 클릭합니다.

4-3 작업 3: 로컬 K8s 접속 확인 및 통신 테스트 (CLI 명령어)

실제 학원 PC의 K8s 클러스터 내부에 테스트용 컨테이너를 띄우고, 그 안에서 AWS 내부로 Ping을 날려 VPN 통과 여부를 검증합니다.

테스트용 파드 생성 및 Ping 테스트

COPY 
# 1. ping 및 네트워킹 유틸리티가 포함된 busybox 컨테이너를 백그라운드로 실행
kubectl run vpn-test-pod --image=busybox --restart=Never -- sleep 3600

# 2. 파드가 Running 상태가 될 때까지 확인
kubectl get pods

# 3. 파드 내부로 진입하여 AWS 스포크 VPC의 사설 IP(예: 프라이빗 EC2의 IP 10.251.10.100)로 ping 테스트
kubectl exec -it vpn-test-pod -- ping 10.251.10.100

# 통신 성공 시 출력 예시:
# 64 bytes from 10.251.10.100: seq=0 ttl=254 time=5.42 ms
# 64 bytes from 10.251.10.100: seq=1 ttl=254 time=4.89 ms

# 4. 테스트 완료 후 파드 삭제
kubectl delete pod vpn-test-pod
검증 완료: 응답(time=X ms)이 정상적으로 돌아온다면, 내 PC의 K8s 클러스터 네트워크가 사내망 VPN 라우터를 거쳐 AWS TGW를 통과하여 스포크 VPC까지 완벽하게 도달했음을 증명하는 것입니다.

Phase 5. 라우팅 테이블 업데이트 및 AWS EKS 접속 검증

VPC 콘솔 ➔ 라우팅 테이블(Route Tables) ➔ 스포크 VPC 라우팅 테이블 선택 ➔ 라우팅 편집

5-2 EKS 클러스터 접속 제어 설정

이제 AWS 클라우드 내부에 띄워진 EKS 클러스터를 인터넷(Public)이 아닌 오직 VPN을 통해서만(Private) 접근하도록 보안을 강화합니다.

🚨 실습 자원 회수 (Clean-up) 순서 철저 준수

  1. EKS 클러스터 삭제: 클러스터 본체 삭제 전, 반드시 연결된 노드 그룹(Node Group) 및 로드밸런서(ELB)가 먼저 제거되어야 합니다.
  2. VPN 연결 삭제: 사이트 간 VPN 연결을 먼저 해제(Delete)해야 고객 게이트웨이(CGW) 삭제가 가능해집니다.
  3. TGW 연결(Attachment) 삭제: 모든 스포크 VPC와 VPN Attachment를 완전히 끊어야 TGW 본체 삭제가 진행됩니다.
  4. Transit Gateway 본체 삭제: 모든 연결(Attachment)이 해제된 것을 확인 후 허브 본체를 제거합니다.
  5. VPC 환경 삭제: NAT Gateway, Elastic IP(탄력적 IP), 서브넷 등을 순차적으로 정리합니다.
