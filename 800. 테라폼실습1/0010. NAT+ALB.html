1. 프로젝트 소개 및 아키텍처 개요

본 과정에서는 중견 기업 이상의 엔터프라이즈 환경에서 필수적으로 요구되는 Hub & Spoke (허브 앤 스포크) 기반의 네트워크 아키텍처를 Terragrunt (테라그런트)를 활용하여 IaC (Infrastructure as Code, 코드형 인프라)로 구축합니다. 본 프로젝트의 루트 디렉터리 명칭은 NatAlb로 명명합니다.

[과금 주의 및 유료 서비스 안내]

본 실습 아키텍처를 AWS (Amazon Web Services, 아마존 웹 서비스)에 배포할 경우 프리티어(Free Tier) 범위를 초과하는 즉각적인 과금이 발생합니다. 실습 완료 후 반드시 리소스를 삭제해야 합니다.

  • TGW (Transit Gateway, 전송 게이트웨이): 시간당 연결 유지 비용 및 트래픽 처리(GB당) 비용 발생
  • NAT (Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이): 시간당 구동 비용 및 데이터 처리 비용 발생
  • ALB (Application Load Balancer, 애플리케이션 로드 밸런서): 시간당 LCU (Load Balancer Capacity Units, 로드 밸런서 용량 단위) 비용 발생
  • EC2 (Elastic Compute Cloud, 엘라스틱 컴퓨트 클라우드): 인스턴스 유형(t3.micro 등)에 따른 시간당 컴퓨팅 과금 발생

2. 하이브리드 디렉터리 구조 설계

테라그런트의 강력한 환경 분리 기능과 테라폼의 모듈화 장점을 결합하기 위해, 각 리소스 단계별 디렉터리 내에 terragrunt.hcl과 .tf 파일들을 함께 배치하는 하이브리드 모듈(Hybrid Module) 방식을 적용합니다. 이를 통해 코드의 응집도를 높이고 디버깅을 단순화합니다.

NatAlb/
├── terragrunt.hcl              # 전역 프로바이더 및 로컬 백엔드 설정
└── prod/                       # 운영 환경
    ├── env.hcl                 # 공통 환경 변수 (리전 등)
    ├── 01-hub-vpc/             
    │   ├── terragrunt.hcl      # 변수 주입 및 모듈 호출 (source = ".")
    │   └── main.tf             # Hub VPC 실제 구성 코드
    ├── 02-service-vpc/         # Service VPC 구성
    ├── 03-tgw/                 # Transit Gateway 및 라우팅 구성
    ├── 04-alb/                 # ALB 및 인바운드 보안 그룹 구성
    └── 05-ec2-nginx/           # Nginx 인스턴스 및 아웃바운드 보안 그룹 구성
설명: 디렉터리 의존성
위 구조는 인프라의 의존성 흐름(01 → 05)을 명확히 보여줍니다. 각 하위 디렉터리는 독립적인 State (상태)를 가지면서도 테라그런트의 dependency 블록을 통해 상위 리소스의 결과값을 동적으로 참조합니다.

3. 글로벌 및 환경 공통 설정

최상위 계층에서는 클라우드 인증 정보와 상태 파일이 저장될 백엔드를 정의합니다. 비용 최적화와 빠른 로컬 테스트를 위해 로컬 상태 저장소(Local State Backend)를 활용합니다.

NatAlb/terragrunt.hcl
locals {
  env_vars   = read_terragrunt_config(find_in_parent_folders("env.hcl"))
  aws_region = local.env_vars.locals.aws_region
}

generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "aws" {
  region = "$${local.aws_region}"
}
EOF
}

remote_state {
  backend = "local"
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite_terragrunt"
  }
  config = {
    path = "$${get_terragrunt_dir()}/terraform.tfstate"
  }
}
NatAlb/prod/env.hcl
locals {
  environment = "prod"
  aws_region  = "ap-northeast-2"
}
설명: DRY 원칙의 적용
read_terragrunt_config 및 find_in_parent_folders 내장 함수를 사용하여 하위 디렉터리 어디에서 실행하든 항상 최상위의 공통 설정을 상속받도록 DRY (Don't Repeat Yourself, 중복 배제 원칙) 구조를 구성했습니다. 상태 파일은 각 리소스가 위치한 하위 디렉터리에 terraform.tfstate 이름으로 자동 생성됩니다.

4. 코어 네트워크 구축 (Hub & Service VPC)

외부 통신을 전담하는 Hub VPC와 실제 서비스가 격리되는 Service VPC를 각각 구성합니다.

4.1 Hub VPC (NAT 및 IGW 포함)

NatAlb/prod/01-hub-vpc/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

terraform {
  source = "." # 현재 디렉터리의 .tf 파일들을 실행
}

inputs = {
  vpc_name = "hub-vpc"
  vpc_cidr = "10.0.0.0/16"
}
NatAlb/prod/01-hub-vpc/main.tf
variable "vpc_name" { type = string }
variable "vpc_cidr" { type = string }

module "hub_vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"

  name = var.vpc_name
  cidr = var.vpc_cidr

  azs             = ["ap-northeast-2a", "ap-northeast-2c"]
  public_subnets  = ["10.0.1.0/24", "10.0.2.0/24"]
  private_subnets = ["10.0.11.0/24", "10.0.12.0/24"]

  enable_nat_gateway = true
  single_nat_gateway = true # 실습용 단일 NAT
}

output "vpc_id" { value = module.hub_vpc.vpc_id }
output "public_subnets" { value = module.hub_vpc.public_subnets }
output "private_subnets" { value = module.hub_vpc.private_subnets }
output "private_route_table_ids" { value = module.hub_vpc.private_route_table_ids }

5. 중앙 라우팅 구축 (TGW 및 라우팅 테이블)

TGW를 생성하고, Service VPC의 아웃바운드 트래픽이 Hub VPC로 향하도록 라우팅을 강제합니다.

NatAlb/prod/03-tgw/terragrunt.hcl
include "root" { path = find_in_parent_folders() }

dependency "hub" { 
  config_path = "../01-hub-vpc" 
  mock_outputs = {
    vpc_id                  = "vpc-mock-12345"
    private_subnets         = ["subnet-mock-1", "subnet-mock-2"]
    public_subnets          = ["subnet-mock-pub1", "subnet-mock-pub2"]
    private_route_table_ids = ["rtb-mock-1", "rtb-mock-2"]
  }
  mock_outputs_allowed_terraform_commands = ["init", "plan", "validate"]
}

dependency "service" { 
  config_path = "../02-service-vpc" 
  mock_outputs = {
    vpc_id                  = "vpc-mock-67890"
    private_subnets         = ["subnet-mock-3", "subnet-mock-4"]
    private_route_table_ids = ["rtb-mock-3", "rtb-mock-4"]
  }
  mock_outputs_allowed_terraform_commands = ["init", "plan", "validate"]
}

terraform { source = "." }

inputs = {
  hub_vpc_id              = dependency.hub.outputs.vpc_id
  hub_subnet_ids          = dependency.hub.outputs.private_subnets
  service_vpc_id          = dependency.service.outputs.vpc_id
  service_subnet_ids      = dependency.service.outputs.private_subnets
  
  # 라우팅 테이블 주입
  hub_route_table_ids     = dependency.hub.outputs.private_route_table_ids
  service_route_table_ids = dependency.service.outputs.private_route_table_ids
}
실전 팁: 콜드 스타트(Cold Start) 방지와 Mock Outputs

인프라를 처음 배포할 때, 상위 리소스(예: VPC)가 아직 생성되지 않아 하위 리소스가 참조할 vpc_id 등의 출력값(Outputs)이 존재하지 않는 현상을 콜드 스타트(Cold Start)라고 합니다. Terragrunt는 init 및 plan 단계에서도 의존성을 평가하므로 출력값이 없으면 즉각적인 에러가 발생합니다.

이를 원천적으로 방지하기 위해 위 코드처럼 dependency 블록 내에 mock_outputs (모의 출력값) 속성을 정의합니다. 또한, mock_outputs_allowed_terraform_commands = ["init", "plan", "validate"]를 설정하여 실제 배포(Apply) 단계가 아닌, 초기화 및 검증 단계에서만 임시 가짜 값을 사용하도록 아키텍처를 견고하게 설계하는 것이 엔터프라이즈 환경의 핵심 노하우입니다.

NatAlb/prod/03-tgw/main.tf
variable "hub_vpc_id" { type = string }
variable "hub_subnet_ids" { type = list(string) }
variable "service_vpc_id" { type = string }
variable "service_subnet_ids" { type = list(string) }
variable "hub_route_table_ids" { type = list(string) }
variable "service_route_table_ids" { type = list(string) }

# TGW 생성
resource "aws_ec2_transit_gateway" "main" {
  description = "Hub and Spoke TGW"
  tags = { Name = "main-tgw" }
}

# Hub VPC 연결
resource "aws_ec2_transit_gateway_vpc_attachment" "hub" {
  subnet_ids         = var.hub_subnet_ids
  transit_gateway_id = aws_ec2_transit_gateway.main.id
  vpc_id             = var.hub_vpc_id
  tags = { Name = "hub-attachment" }
}

# Service VPC 연결
resource "aws_ec2_transit_gateway_vpc_attachment" "service" {
  subnet_ids         = var.service_subnet_ids
  transit_gateway_id = aws_ec2_transit_gateway.main.id
  vpc_id             = var.service_vpc_id
  tags = { Name = "service-attachment" }
}

# [핵심 라우팅] Service VPC에서 외부(0.0.0.0/0)로 가는 트래픽을 TGW로 전송
resource "aws_route" "service_to_tgw" {
  count                  = length(var.service_route_table_ids)
  route_table_id         = var.service_route_table_ids[count.index]
  destination_cidr_block = "0.0.0.0/0"
  transit_gateway_id     = aws_ec2_transit_gateway.main.id
  depends_on             = [aws_ec2_transit_gateway_vpc_attachment.service]
}

# [핵심 라우팅] Hub VPC에서 Service VPC(10.1.0.0/16)로 가는 트래픽을 TGW로 전송
resource "aws_route" "hub_to_service" {
  count                  = length(var.hub_route_table_ids)
  route_table_id         = var.hub_route_table_ids[count.index]
  destination_cidr_block = "10.1.0.0/16"
  transit_gateway_id     = aws_ec2_transit_gateway.main.id
  depends_on             = [aws_ec2_transit_gateway_vpc_attachment.hub]
}
자주 오해하거나 실수하는 부분: 비대칭 라우팅 및 누락

TGW를 생성하고 연결(Attachment)만 맺었다고 해서 통신이 이루어지지 않습니다. 반드시 VPC 내부의 Route Table (라우팅 테이블)에 목적지 대역에 대한 정적 라우팅(Static Route)을 위 코드처럼 명시적으로 추가해주어야 패킷이 TGW 방향으로 정상적으로 흘러갑니다.

6. 트래픽 분산 및 애플리케이션 계층 배포 (ALB & EC2)

보안 그룹(SG) 차단 문제를 해결한 최종 로드 밸런서와 Nginx 웹 서버를 배포합니다. 이 모듈들 역시 안정적인 프로비저닝을 위해 mock_outputs가 적용되어 있습니다.

variable "vpc_id" { type = string }
variable "public_subnets" { type = list(string) }

# ALB 전용 보안 그룹 (인바운드: 외부 허용, 아웃바운드: Service VPC 대역 허용)
resource "aws_security_group" "alb_sg" {
  name        = "hub-alb-sg"
  description = "ALB Security Group"
  vpc_id      = var.vpc_id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["10.1.0.0/16"] # Service VPC 향 트래픽 허용
  }
}

module "alb" {
  source  = "terraform-aws-modules/alb/aws"
  version = "8.7.0"

  name               = "hub-central-alb"
  load_balancer_type = "application"
  vpc_id             = var.vpc_id
  subnets            = var.public_subnets
  security_groups    = [aws_security_group.alb_sg.id]

  http_tcp_listeners = [{ port = 80, protocol = "HTTP", target_group_index = 0 }]
  target_groups = [
    {
      name             = "service-nginx-tg"
      backend_protocol = "HTTP"
      backend_port     = 80
      target_type      = "ip" # 크로스 VPC 라우팅을 위한 IP 타겟 필수 지정
    }
  ]
}

output "target_group_arns" { value = module.alb.target_group_arns }
output "alb_dns_name" { value = module.alb.lb_dns_name }

6.2 Nginx EC2 배포 (격리된 보안 그룹 및 SSM 접속 적용)

프라이빗 서브넷에 위치한 EC2 인스턴스에 안전하게 접근하기 위해, 외부 인터넷에 직접 노출되는 SSH 포트(22번)를 개방하는 대신 AWS SSM (Systems Manager, 시스템 매니저)의 Session Manager (세션 매니저)를 활용합니다. 이를 위해 IAM (Identity and Access Management, 식별 및 접근 관리) 역할과 인스턴스 프로파일을 생성하여 EC2에 부여합니다.

# --- [추가] SSM (Systems Manager) 접속을 위한 IAM 설정 ---

# 1. EC2가 역할을 수임(Assume)할 수 있도록 신뢰 정책 생성
resource "aws_iam_role" "ssm_role" {
  name = "nginx-ssm-role"
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Action = "sts:AssumeRole"
      Effect = "Allow"
      Principal = {
        Service = "ec2.amazonaws.com"
      }
    }]
  })
}

# 2. 역할에 SSM 관리형 정책(AmazonSSMManagedInstanceCore) 연결
resource "aws_iam_role_policy_attachment" "ssm_policy" {
  role       = aws_iam_role.ssm_role.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore"
}

# 3. EC2 인스턴스에 부착할 인스턴스 프로파일 생성
resource "aws_iam_instance_profile" "ssm_profile" {
  name = "nginx-ssm-profile"
  role = aws_iam_role.ssm_role.name
}

  # 생성한 SSM 인스턴스 프로파일을 EC2에 적용
  iam_instance_profile = aws_iam_instance_profile.ssm_profile.name

  user_data = <<-EOF
              #!/bin/bash
              apt-get update -y
              apt-get install -y nginx
              systemctl start nginx
              systemctl enable nginx
              echo "Welcome to Centralized Network Nginx!" > /var/www/html/index.html
              EOF
  tags = { Name = "nginx-service-instance" }
}

# ALB 타겟 그룹에 생성된 EC2의 Private IP 자동 등록 매핑
resource "aws_lb_target_group_attachment" "nginx_attach" {
  target_group_arn = var.target_group_arn
  target_id        = aws_instance.nginx.private_ip
  port             = 80
}
설명: 동적 바인딩(Dynamic Binding) 자동화 및 SSM 통합
마지막 05번 단계에서는 생성된 EC2의 Private IP (프라이빗 IP)를 앞서 생성한 ALB의 타겟 그룹에 동적으로 바인딩(묶음)하는 aws_lb_target_group_attachment 리소스를 포함시켰습니다. 또한, SSM 연동을 위한 IAM 설정이 추가되어 관리자가 프라이빗 망의 서버에 안전하게 접근할 수 있도록 진정한 완전 자동화 및 보안 아키텍처를 달성했습니다.

7. 전체 배포 및 검증 파이프라인

모든 코드가 작성되었다면 최상위 실행 디렉터리(NatAlb/prod/)로 이동하여 터미널(Terminal)에서 아래 명령어를 순서대로 실행합니다.

# 1. 의존성 트리에 맞춰 모든 하위 디렉터리 플러그인 초기화
terragrunt run-all init -upgrade

# 2. 변경 사항 미리보기 (선택 사항)
terragrunt run-all plan

# 3. 실제 프로비저닝 수행 (초기 배포 시 -out 옵션 사용 금지)
terragrunt run-all apply
경고: 초기 배포 시 Plan 파일(-out) 사용 금지

일반적인 단일 테라폼 실행에서는 plan -out=tgplan.out 방식을 권장하지만, Terragrunt의 run-all을 통한 최초 배포(Cold Start) 시에는 절대 사용해서는 안 됩니다. 모의 출력값(Mock Outputs)이 계획 파일(Plan File)에 하드코딩(Hardcoding)되어, 실제 apply 시 AWS API (Application Programming Interface, 애플리케이션 프로그래밍 인터페이스)에 가짜 ID(예: vpc-mock-12345)를 전송하여 치명적인 오류를 발생시킵니다.

실전 팁: 상태 검증 및 리소스 삭제 방법

배포 완료 후 콘솔 출력에 나타난 alb_dns_name 값을 복사하여 웹 브라우저 주소창에 붙여넣기 합니다. "Welcome to Centralized Network Nginx!" 메시지가 정상 노출된다면 인바운드와 아웃바운드(Nginx 설치를 위한 인터넷 연결), 라우팅, 방화벽(SG)이 모두 완벽하게 상호작용하고 있는 것입니다. (또한 AWS Console에서 Systems Manager > Session Manager를 통해 해당 EC2 인스턴스에 정상 접속되는지 확인합니다.)

※ 실습이 끝나면 과금을 방지하기 위해 동일한 디렉터리에서 terragrunt run-all destroy를 반드시 실행하십시오.

8. 트러블슈팅: 초기 배포(Cold Start) 에러 완벽 해결

8.1 출력값 미감지 에러 (detected no outputs)

위 7번의 배포 과정 중, 인프라를 처음 배포하기 위해 terragrunt run-all init 또는 plan을 실행할 때 아래와 같이 "detected no outputs (출력값이 감지되지 않음)" 에러가 발생할 수 있습니다.

time=2026-03-10T05:44:18+09:00 level=error msg=Module C:/.../04-alb has finished with an error: 
.../01-hub-vpc/terragrunt.hcl is a dependency of .../04-alb/terragrunt.hcl but detected no outputs.
원인 분석 및 해결 방안
하위 리소스가 참조할 진짜 값이 State File (상태 파일)에 아직 존재하지 않기 때문입니다. 이를 해결하기 위해 dependency 블록 내에 mock_outputs (모의 출력값)을 명시하여 해결합니다. 본 교재의 코드(5, 6 섹션)에는 이미 이 해결책이 모두 반영되어 있습니다.

8.2 의존성 계획(Plan) 파일 충돌 에러 (Malformed / Not Found)

만약 초기 배포 시 terragrunt run-all apply "tgplan.out"과 같이 저장된 계획 파일을 강제로 실행할 경우, 아래와 같은 에러가 다수 발생하며 인프라 생성이 중단됩니다.

Warning: Ignoring variable when applying a saved plan
The variable public_subnets cannot be overriden when applying a saved plan file...
...
Error: creating Security Group (hub-alb-sg): ... api error InvalidVpcId.Malformed: The vpc ID 'vpc-mock-12345' is malformed
...
Error: creating EC2 Transit Gateway VPC Attachment: ... api error InvalidSubnetID.NotFound: The subnet ID 'subnet-mock-2' does not exist
원인 분석 및 해결 방안
이 에러는 plan 단계에서 사용되었던 가짜 모의 값(vpc-mock-12345 등)이 .out 파일에 영구적으로 기록되었기 때문에 발생합니다. 실제 apply가 실행되면서 상위 리소스가 진짜 VPC를 생성하여 vpc-09f7b... 형태의 진짜 ID를 넘겨주려 하지만, 테라폼은 "저장된 계획 파일의 값(Mock)과 현재 주입된 값(Real)이 다르다"며 업데이트를 거부하고 AWS API에 Mock 값을 그대로 전달합니다.

해결책: 생성된 tgplan.out 파일을 삭제하고, 위 7번 섹션의 가이드에 따라 단순히 terragrunt run-all apply 명령어를 텍스트 파라미터 없이 실행하십시오. Terragrunt가 자동으로 의존성 순서에 맞춰 상위 리소스를 생성한 뒤, 진짜 출력값을 하위 리소스에 즉각 전달하여 에러 없이 배포를 완수합니다.
