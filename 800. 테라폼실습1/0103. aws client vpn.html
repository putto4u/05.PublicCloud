AWS Client VPN을 이용한 Hub-and-Spoke 통합 보안 접속 실습

IP 직업훈련 전문가과정 - 고급 하이브리드 네트워크 아키텍처 v2.0

Environment Settings

Local PC/VM: 192.168.10.x/24

VPN Pod: 172.250.0.0/16

실습 아키텍처 및 비용 안내

본 과정은 로컬 환경(Ubuntu VM)에서 AWS Client VPN (AWS 클라이언트 VPN)을 통해 클라우드 내부의 Hub VPC (중앙 허브 가상 사설 클라우드)에 접속한 후, TGW (Transit Gateway, 전송 게이트웨이)를 거쳐 폐쇄적인 Spoke/Service VPC (서비스 분기 가상 사설 클라우드) 내의 EC2 웹 서버까지 통신하는 전체 경로를 구축합니다.

[중요] 유료 서비스 과금 경고

  • Client VPN Endpoint: 시간당 약 $0.10 (엔드포인트당) + $0.05 (클라이언트 연결당)
  • Transit Gateway: 시간당 약 $0.05 + 데이터 처리 비용 ($0.02/GB)
  • NAT Gateway: 시간당 약 $0.045 + 데이터 처리 비용
  • 실습 후 반드시 모든 리소스를 삭제(Destroy) 하십시오.
STEP 01

상호 인증을 위한 인증서 생성 및 ACM 등록

AWS Client VPN은 Mutual Authentication (상호 인증) 방식을 지원합니다. 이를 위해 서버와 클라이언트 각각의 인증서가 필요합니다.

Ubuntu VM: Certificate Generation (Easy-RSA) Bash 
# 1. Easy-RSA 도구 설치 및 초기화
git clone https://github.com/OpenVPN/easy-rsa.git
cd easy-rsa/easyrsa3
./easyrsa init-pki

# 2. CA (인증 기관) 생성 (비밀번호 설정 권장)
./easyrsa build-ca nopass

# 3. 서버 인증서 및 키 생성
./easyrsa build-server-full server nopass

# 4. 클라이언트 인증서 및 키 생성
./easyrsa build-client-full client1.domain.tld nopass

# 5. 생성된 파일을 한 곳으로 복사
mkdir ~/vpn-certs/
cp pki/ca.crt pki/issued/server.crt pki/private/server.key pki/issued/client1.domain.tld.crt pki/private/client1.domain.tld.key ~/vpn-certs/

AWS 콘솔 설정: ACM (AWS Certificate Manager) 가져오기

  1. AWS Management Console 검색창에 ACM 입력 후 이동.
  2. 인증서 가져오기 (Import a certificate) 클릭.
  3. 서버 인증서 등록:
    • 인증서 본문: server.crt 내용 복사
    • 인증서 프라이빗 키: server.key 내용 복사
    • 인증서 체인: ca.crt 내용 복사
  4. 클라이언트 인증서 등록: 위 과정을 client1.domain.tld 파일들로 한 번 더 반복.
STEP 02

AWS Client VPN 엔드포인트 생성

사용자가 가상 터널을 맺을 논리적인 게이트웨이를 생성합니다.

엔드포인트 세부 구성 정보

Name Tag

LAB-Client-VPN

Client IPv4 CIDR (VPN 대역)

172.250.0.0/16

* 주의: 기존 VPC 대역(10.250, 10.251)과 겹치면 안 됨

Server certificate ARN

ACM에서 가져온 server 인증서 선택

Authentication Options

Mutual authentication (상호 인증) 사용

Client certificate ARN

ACM에서 가져온 client 인증서 선택

Connection Log options

아니요 (비용 절감 목적)

중요: Split-tunnel (분할 터널) 설정

Split-tunnel 항목을 활성화 (Yes) 하십시오. 이를 활성화하지 않으면 VPN 접속 시 로컬 PC의 모든 인터넷 트래픽이 AWS로 우회되어 로컬 인터넷 연결이 끊길 수 있습니다.

STEP 03

네트워크 연결 및 권한 설정 (Target & Authorization)

생성된 엔드포인트를 실제 VPC 서브넷에 연결하고, 접근 가능한 네트워크 대역을 허용합니다.

1. 대상 네트워크 연결 (Target Network Association)

VPN 트래픽이 진입할 통로를 지정합니다.

  • Target Network Associations 탭 선택 -> Associate target network 클릭.
  • VPC: HUB VPC 선택.
  • Subnet: Hub Private Subnet A (10.250.101.0/24) 선택.
  • * 연결 시 약 5~10분의 시간이 소요되며, 상태가 'Available'이 되어야 다음 단계가 가능합니다.

2. 권한 부여 규칙 (Authorization Rules)

VPN 클라이언트가 통신할 수 있는 목적지 대역을 허용하는 방화벽 규칙입니다.

대상 네트워크 (Destination) 설명 (Description) 권한 (Access)
10.250.0.0/16 Hub VPC 내부 접근 허용 Allow all users
10.251.0.0/16 Service VPC (Spoke) 접근 허용 Allow all users
STEP 04

Hub and Spoke 통합 라우팅 구성 (Route Table)

VPN 엔드포인트 자체 라우팅 테이블과 Hub VPC의 라우팅 테이블에 Spoke VPC로 가는 경로를 추가해야 합니다.

1. VPN 엔드포인트 라우팅 추가

"VPN 서버야, 10.251(Service)로 가는 패킷은 네가 연결된 서브넷으로 던져라"

  1. Route Table 탭 선택.
  2. Create Route 클릭.
  3. Destination CIDR: 10.251.0.0/16
  4. Target Network: 기존 연결된 Hub 서브넷 ID 선택.

2. VPC 라우팅 테이블(Hub Private) 수정

"서브넷아, 10.251로 가는 건 TGW(Transit Gateway)로 보내라"

  1. VPC 콘솔 -> 라우팅 테이블 이동.
  2. Hub Private Subnet의 라우팅 테이블 선택.
  3. Destination: 10.251.0.0/16
  4. Target: Transit Gateway (tgw-xxxx)
STEP 05

Ubuntu 클라이언트 접속 설정

설정 파일을 다운로드하고, 생성한 클라이언트 인증서를 결합하여 터널을 생성합니다.

구성 파일 다운로드 및 수정

  1. VPN 엔드포인트 페이지 우측 상단 Download client configuration 클릭.
  2. 다운로드된 downloaded-client-config.ovpn 파일을 텍스트 편집기로 엽니다.
  3. 파일 하단에 아래와 같이 인증서 경로를 절대 경로로 추가하거나 파일 내용을 직접 삽입합니다.
client-config.ovpn (수정 예시) Config 
# ... 기존 설정 내용 ...

# 하단에 추가 (파일 경로 입력 방식)
cert /home/ubuntu/vpn-certs/client1.domain.tld.crt
key /home/ubuntu/vpn-certs/client1.domain.tld.key

# 또는 <cert>, <key> 태그를 사용해 파일 내용을 직접 붙여넣기 가능
Ubuntu VM: Connection Command Bash 
# OpenVPN 패키지 설치
sudo apt update && sudo apt install openvpn -y

# VPN 접속 실행 (백그라운드 실행 시 --daemon 옵션 추가)
sudo openvpn --config downloaded-client-config.ovpn
STEP 06

최종 연결 확인 (Connectivity Test)

Hub를 거쳐 Spoke의 EC2까지 통신이 되는지 확인합니다.

1. 인터페이스 확인

VPN이 연결되면 tun0 인터페이스가 생성되고 172.250.x.x IP가 할당됩니다.

$ ip addr show tun0

2. 라우팅 테이블 확인

OS 라우팅 테이블에 AWS 대역이 tun0를 향하고 있는지 확인합니다.

$ netstat -rn

3. Spoke VPC EC2 접속 테스트

Service VPC(10.251)에 있는 Nginx 서버의 Private IP로 직접 접속을 시도합니다.

# ICMP 통신 확인 (보안그룹에서 허용된 경우)
$ ping 10.251.101.10

# HTTP 서비스 응답 확인
$ curl http://10.251.101.10

* 성공 시: "Welcome to Centralized Network Nginx!" 메시지가 출력됩니다.
