Hub & Spoke 기반
AWS Client VPN 실무 기술 명세

개별 사용자가 TLS(Transport Layer Security, 전송 계층 보안) 인증을 통해 AWS 내부망에 안전하게 접속할 수 있도록 지원하는 AWS Client VPN(클라이언트 가상 사설망) 환경을 구축합니다.

주의: 본 문서에서 다루는 Client VPN은 Customer Gateway(고객 게이트웨이)나 Virtual Private Gateway(VGW, 가상 프라이빗 게이트웨이)를 사용하는 Site-to-Site(사이트 간) VPN과 명확히 구분됩니다. 개별 원격 접속자를 위한 전용 서비스입니다.

Phase 0. 인증 인프라 및 ACM(AWS Certificate Manager) 등록

0-1 Mutual TLS(상호 TLS) 인증을 위한 사설 인증서 생성

AWS Client VPN 서비스가 서버와 클라이언트를 식별할 수 있도록 Easy-RSA를 사용하여 사설 CA(Certificate Authority, 인증 기관)와 인증서를 로컬 환경에서 생성합니다. 

# Easy-RSA 도구 복제 및 초기화
git clone https://github.com/OpenVPN/easy-rsa.git
cd easy-rsa/easyrsa3
./easyrsa init-pki

# 1. 사설 인증 기관(CA) 생성
./easyrsa build-ca nopass

# 2. 서버 인증서 및 키 생성 (AWS 엔드포인트용)
./easyrsa build-server-full server nopass

# 3. 클라이언트 인증서 및 키 생성 (사용자 PC용)
./easyrsa build-client-full client1.domain.tld nopass

0-2 ACM(AWS Certificate Manager)에 인증서 가져오기

생성된 파일들을 AWS 콘솔의 인증서 관리자 서비스에 등록합니다. Public(공인) 인증서가 아닌 Imported(가져온) 사설 인증서로 분류됩니다.

Console ➔ AWS Certificate Manager ➔ Import a certificate (인증서 가져오기)

Server Certificate (서버 인증서)

  • Certificate body: server.crt
  • Certificate private key: server.key
  • Certificate chain: ca.crt

Client Certificate (클라이언트 인증서)

  • Certificate body: client1.domain.tld.crt
  • Certificate private key: client1.domain.tld.key
  • Certificate chain: ca.crt
설명: ACM(AWS Certificate Manager, AWS 인증서 관리자)은 웹사이트나 VPN에 필요한 SSL/TLS 인증서를 통합 관리하는 도구입니다. 사설 인증서를 사용할 때는 반드시 Chain(체인) 부분에 CA 인증서를 포함해야 상호 검증이 가능합니다.

Phase 1. Hub 네트워크 및 Transit Gateway 구축

VPC(Virtual Private Cloud) & TGW 구성 VPC: Free / TGW: Paid
VPC ➔ Transit gateways ➔ Create transit gateway (전송 게이트웨이 생성)

TGW(Transit Gateway, 전송 게이트웨이)는 여러 VPC와 VPN을 중앙에서 연결하는 고성능 라우터입니다. ASN(자율 시스템 번호)은 기본값 64512를 사용합니다.

TGW Attachments (연결)

Hub Attachment

VPC: Hub-VPC (10.0.0.0/16)

Subnets: Private A, C

Spoke Attachment

VPC: Spoke-EKS (10.251.0.0/16)

Subnets: Private A, C

설명: TGW(Transit Gateway, 전송 게이트웨이)는 네트워크 허브 역할을 수행합니다. 각 VPC를 TGW에 Attachment(연결) 함으로써 상호 통신이 가능한 물리적 통로를 개설합니다.

네트워크 및 권한 최종 활성화 (중요 단계)

1. Target network associations

엔드포인트를 실제 VPC 서브넷에 결합합니다.

VPC: Hub-VPC / Subnet: Private Subnet

2. Authorization rules

어떤 네트워크 대역으로의 접근을 허용할지 정의합니다.

Destination network: 10.251.0.0/16 (Spoke)

3. Route table (Client VPN 전용)

엔드포인트 내부 라우팅을 설정합니다.

Route: 10.251.0.0/16 ➔ Target: Association된 Subnet ID
설명: Target network associations(대상 네트워크 연결)은 VPN 장비를 VPC의 스위치 포트에 꽂는 행위와 같습니다. 이 연결이 활성화(Available)되어야 실제 통신 경로가 생성됩니다.

Phase 3. VPC Route Table(라우팅 테이블) 구성

AWS 콘솔의 VPC ➔ Route tables(라우팅 테이블) 메뉴에서 패킷이 TGW를 타고 목적지로 갈 수 있도록 이정표를 설정합니다.

Hub VPC Route Table
DestinationTarget
10.0.0.0/16local
10.251.0.0/16tgw-xxxx (TGW)
Spoke VPC Route Table
DestinationTarget
10.251.0.0/16local
172.16.0.0/22tgw-xxxx (TGW)
설명: Route Table(라우팅 테이블)은 네트워크 패킷의 목적지 주소를 보고 어느 방향(Gateway)으로 보낼지 결정하는 지도입니다. 스포크 VPC에서 클라이언트 VPN 대역(172.16.x.x)을 타겟으로 하는 경로는 반드시 TGW로 지정해야 응답 패킷이 사용자에게 돌아갈 수 있습니다.

Phase 4. 클라이언트 접속 설정 (.ovpn 수정)

Client VPN Endpoints ➔ Download client configuration (클라이언트 구성 다운로드)

다운로드한 .ovpn 파일에는 서버 주소만 들어있습니다. 우리가 생성한 클라이언트 인증서와 키를 직접 파일 안에 삽입해야 합니다.

# .ovpn 파일 하단에  및  섹션 추가
<cert>
-----BEGIN CERTIFICATE-----
(client1.domain.tld.crt 내용 전체 붙여넣기)
-----END CERTIFICATE-----
</cert>

<key>
-----BEGIN PRIVATE KEY-----
(client1.domain.tld.key 내용 전체 붙여넣기)
-----END PRIVATE KEY-----
</key>
설명: .ovpn(OpenVPN configuration file)은 OpenVPN 클라이언트가 접속에 필요한 모든 정보를 담고 있는 설정 파일입니다. 상호 인증 방식에서는 클라이언트 고유의 Certificate(인증서)와 Private Key(개인키)가 파일 내에 포함되어야 보안 통신이 성립됩니다.

🚨 자원 관리 및 삭제 순서 (Cleanup)

과금을 방지하기 위해 사용 종료 후 다음 순서대로 삭제하십시오.

  1. Disassociate Target Network: Client VPN 엔드포인트에서 대상 네트워크 연결을 먼저 해제합니다. (완료까지 약 5분 소요)
  2. Delete Client VPN Endpoint: 엔드포인트를 삭제합니다.
  3. Delete TGW Attachments: Hub 및 Spoke VPC와의 연결을 끊습니다.
  4. Delete Transit Gateway: 모든 연결이 사라지면 TGW 본체를 삭제합니다.
  5. ACM Certificates: 필요 시 업로드한 인증서를 삭제합니다.
