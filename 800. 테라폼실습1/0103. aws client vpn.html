Hub & Spoke 기반
AWS Client VPN 구축 마스터

여러 사용자가 공유하는 NAT(Network Address Translation, 네트워크 주소 변환) 환경의 접속 충돌 문제를 근본적으로 해결하고, 1:N 원격 접속에 최적화된 AWS Client VPN(클라이언트 가상 사설망) 기반의 엔터프라이즈 원격 근무 아키텍처 기술 명세서입니다.

설명: Version 2의 S2S(Site-to-Site) VPN은 본사와 지점 같은 '네트워크 대 네트워크' 연결에 적합합니다. 반면 본 Version 3 아키텍처는 개별 개발자, 관리자가 위치에 상관없이 OpenVPN 클라이언트를 통해 AWS 인프라에 개별 접속하는 '사용자 대 네트워크(Client-to-Site)' 통신의 글로벌 표준(Best Practice)입니다.

Phase 0. 인증 인프라(PKI) 준비 및 ACM 등록

0-1 Easy-RSA를 이용한 상호 TLS(Mutual TLS) 인증서 생성

AWS Client VPN은 Active Directory(액티브 디렉토리), SAML(보안 어서션 마크업 언어), 그리고 상호 인증을 지원합니다. 인프라 실무 구성에서 가장 독립적이고 보편적인 상호 인증(Mutual Authentication, 서버와 클라이언트가 서로의 인증서를 검증하는 방식)을 위한 키를 로컬 환경(Ubuntu 등)에서 생성합니다.

인증 기관(CA) 및 인증서 발급 Local Tool
COPY 
# 1. OpenVPN easy-rsa 패키지 복제
git clone https://github.com/OpenVPN/easy-rsa.git
cd easy-rsa/easyrsa3

# 2. PKI(Public Key Infrastructure, 공개 키 기반 구조) 초기화
./easyrsa init-pki

# 3. 새로운 인증 기관(CA, Certificate Authority) 생성 (비밀번호 설정 없이 진행: nopass)
./easyrsa build-ca nopass

# 4. 서버(AWS Client VPN)용 인증서 및 키 생성
./easyrsa build-server-full server nopass

# 5. 클라이언트(사용자)용 인증서 및 키 생성
./easyrsa build-client-full client1.domain.tld nopass
설명: 생성된 파일 중 ca.crt(인증 기관), server.crt/server.key(서버용), client1.domain.tld.crt/key(클라이언트용) 파일이 연동의 핵심 자산이 됩니다.

0-2 AWS Certificate Manager (ACM, AWS 인증서 관리자)에 인증서 업로드

생성한 서버 인증서와 클라이언트 인증서를 AWS 클라우드 환경에서 사용할 수 있도록 ACM 서비스에 등록합니다.

ACM 콘솔 ➔ 인증서 ➔ 인증서 가져오기(Import a certificate)

1차 업로드: 서버 인증서

  • 본문: server.crt 내용 복사
  • 프라이빗 키: server.key 내용 복사
  • 체인: ca.crt 내용 복사

2차 업로드: 클라이언트 인증서

  • 본문: client1.domain.tld.crt 내용 복사
  • 프라이빗 키: client1.domain.tld.key 내용 복사
  • 체인: ca.crt 내용 복사

Phase 1. 허브 네트워크 및 TGW 인프라 구축

Client VPN은 자체적으로 단독 작동하지 않으며, 반드시 특정한 VPC(Virtual Private Cloud, 가상 사설 클라우드)의 서브넷과 연결(Association)되어 해당 네트워크의 라우팅 능력을 상속받아야 합니다. 이를 위해 허브 VPC를 생성합니다.

Step 1 & 2 요약: Hub VPC 및 TGW 구성

  • 1. Hub VPC 생성: CIDR 10.0.0.0/16 (이 VPC는 Client VPN이 거쳐갈 관문 역할만 수행합니다.)
  • 2. TGW(Transit Gateway) 생성: ASN 64512 지정.
  • 3. TGW Attachment (Hub): Hub VPC를 TGW에 연결(Attach)합니다.
설명: 스포크 VPC(10.251.x.x)에 Client VPN을 직접 연결할 수도 있지만, 여러 스포크가 존재하는 엔터프라이즈 환경에서는 허브 VPC 하나에만 VPN을 연결하고 TGW를 통해 트래픽을 분배하는 것이 라우팅 관리의 효율성을 극대화하는 아키텍트의 정석입니다.

4 대상 네트워크 연결 및 보안/라우팅 등록

엔드포인트가 만들어지면, 이를 허브 VPC에 꽂아 물리적인 라우팅 경로를 확보합니다.

1. 대상 네트워크 연결 (Target Network Association)

엔드포인트 선택 ➔ 대상 네트워크 연결 탭 ➔ 연결 생성

VPC: Hub VPC (10.0.0.0/16) 선택 서브넷: Hub VPC의 프라이빗 서브넷 선택

* 주의: 이 연결이 완료(Available)되어야 과금이 시작됩니다. (약 5~10분 소요)

2. 권한 부여 규칙 (Authorization Rules)

엔드포인트 선택 ➔ 권한 부여 규칙 탭 ➔ 권한 부여 규칙 추가

대상 네트워크: 10.251.0.0/16 (Spoke VPC 대역) 액세스 권한: 모든 사용자에게 액세스 허용

* 스포크 VPC로의 트래픽을 VPN 논리적으로 허가하는 방화벽 같은 역할입니다.

3. 경로 추가 (Route Table)

엔드포인트 선택 ➔ 경로(Routes) 탭 ➔ 경로 생성

경로 대상: 10.251.0.0/16 (Spoke VPC 대역) 대상 네트워크: 연결된 Hub VPC 서브넷 ID 선택

* VPN에서 스포크로 가려면 어떤 서브넷 인터페이스를 타고 나가야 할지 이정표를 세워줍니다.

Phase 3. 스포크 연동 및 라우팅 최종 구성

Client VPN 트래픽이 Hub VPC로 진입했으니, 이제 TGW를 타고 스포크 EKS 환경으로 넘어가고 반환되도록 네트워크 테이블을 완성합니다.

Routing Tables Architecture

Phase 4. 클라이언트 접속 구성 (.ovpn)

AWS 콘솔 상단에서 클라이언트 구성 다운로드(Download Client Configuration) 버튼을 클릭하여 downloaded-client-config.ovpn 파일을 받습니다. 이 파일에 로컬에서 만든 인증서 정보를 주입해야 작동합니다.

.ovpn 파일 편집 (텍스트 에디터 활용)

COPY 
# 원본 ovpn 파일 맨 아래에 다음 내용 추가

<cert>
-----BEGIN CERTIFICATE-----
(client1.domain.tld.crt 파일의 전체 내용 복사 붙여넣기)
-----END CERTIFICATE-----
</cert>

<key>
-----BEGIN PRIVATE KEY-----
(client1.domain.tld.key 파일의 전체 내용 복사 붙여넣기)
-----END PRIVATE KEY-----
</key>
설명: 이 설정 파일 하나만 있으면 AWS Client VPN Desktop Application(공식 앱)이나 범용 OpenVPN Connect 프로그램에서 파일을 임포트(Import)하여 즉시 안전하게 사내망에 연결할 수 있습니다.

Phase 5. 보안 그룹 오픈 및 프라이빗 EKS 접근

5-1 Spoke VPC 보안 그룹 인바운드 허용

Spoke VPC 내부 자원(EC2, EKS 노드 등)의 방화벽(Security Group)에서 Client VPN 사용자의 IP 대역을 신뢰할 수 있도록 추가합니다.

보안 그룹 인바운드 규칙
소스: 172.16.0.0/22 (클라이언트 VPN CIDR) / 포트: 모든 ICMP 및 필요한 TCP

5-2 EKS 클러스터 프라이빗 엔드포인트 제어

운영 과금 및 라이프사이클 관리

Client VPN은 인프라 유지 고정비와 더불어 사용자 접속 시간에 따른 변동비가 복합적으로 청구됩니다. 실습 종료 후 자원 회수를 철저히 해야 합니다.

Cost Breakdown

🚨 실무 자원 회수 (Clean-up) 순서

  1. EKS 클러스터 삭제: 노드 그룹 및 로드밸런서 선행 제거.
  2. 대상 네트워크 연결 해제 (우선순위): Client VPN 메뉴에서 대상 네트워크 연결(Target Network Association)을 먼저 해제해야 엔드포인트 삭제가 가능합니다. (시간이 오래 걸릴 수 있음)
  3. Client VPN 엔드포인트 삭제: 연결이 해제된 후 엔드포인트 본체를 삭제합니다.
  4. TGW 연결(Attachment) 삭제: Hub VPC 및 Spoke VPC와의 TGW 연결 분리.
  5. Transit Gateway 본체 삭제: 모든 연결 해제 확인 후 최종 제거.
  6. VPC 환경 삭제: 리소스들 정리.
