Hub & Spoke 기반
AWS Client VPN 엔터프라이즈 실무 명세

본 문서는 AWS 관리 콘솔의 최신 업데이트 항목을 반영하여, 개별 사용자가 Mutual TLS 인증을 통해 프라이빗 네트워크에 접속하는 전 과정을 상세히 기술합니다.

Phase 0. 인증 인프라 및 ACM(AWS Certificate Manager) 준비

0-1 Easy-RSA를 이용한 인증서(Certificate) 및 키(Key) 생성

Client VPN은 Server(서버)와 Client(클라이언트)가 서로를 검증하는 상호 인증 방식을 사용합니다. 이를 위해 사설 CA(인증 기관)를 구축합니다. 

# 1. 환경 초기화
git clone https://github.com/OpenVPN/easy-rsa.git
cd easy-rsa/easyrsa3
./easyrsa init-pki

# 2. CA(Certificate Authority, 인증 기관) 구축
./easyrsa build-ca nopass

# 3. 서버 인증서 및 개인키 생성 (AWS 콘솔 등록용)
./easyrsa build-server-full server nopass

# 4. 클라이언트 인증서 및 개인키 생성 (사용자 PC 배포용)
./easyrsa build-client-full user01.domain.tld nopass

0-2 ACM(AWS Certificate Manager, 인증서 관리자) 업로드

Console ➔ ACM ➔ Import a certificate (인증서 가져오기)

SERVER CERT

  • Body: server.crt
  • Key: server.key
  • Chain: ca.crt

CLIENT CERT

  • Body: user01.domain.tld.crt
  • Key: user01.domain.tld.key
  • Chain: ca.crt

TGW Attachment (전송 게이트웨이 연결) 구성

Hub Attachment (허브 연결)

VPC: Hub-VPC (10.0.0.0/16)

Spoke Attachment (스포크 연결)

VPC: Spoke-EKS (10.251.0.0/16)

Phase 2. Create Client VPN Endpoint (엔드포인트 생성)

VPC ➔ VPN ➔ Client VPN endpoints ➔ Create client VPN endpoint
핵심 설정 필드 (Required Fields)
클라이언트 IPv4 CIDR

필수 설정입니다. 비활성화 시 사용자의 인터넷 트래픽까지 모두 AWS로 흐릅니다.

Phase 3. 단계별 연결 확인 및 트러블슈팅

🚨 [해결] 172.x.x.x 가상 IP가 할당되지 않는 원인

제시된 콘솔 이미지 분석 결과: 대상 네트워크 연결(Association)이 누락되었습니다.

현상 분석 (Image Analysis)

  • • 상태 (Status): Pending-associate → 아직 네트워크 라인이 연결되지 않음.
  • • 대상 네트워크 연결 (Association): 탭에 데이터가 없음 → VPC 서브넷과 엔드포인트가 연결되지 않음.

해결 방법 (Action Plan)

  1. 1. AWS 콘솔에서 생성된 **클라이언트 VPN 엔드포인트** 클릭.
  2. 2. 하단 탭 메뉴에서 **[대상 네트워크 연결 (Target network associations)]** 클릭.
  3. 3. **[대상 네트워크 연결 (Associate target network)]** 버튼 클릭.
  4. 4. **VPC 선택:** Hub-VPC 선택.
  5. 5. **서브넷 선택:** 프라이빗 서브넷 중 하나 선택 후 [연결] 클릭.
  6. 6. **상태 확인:** 약 5~10분 후 상태가 **`Available`**로 변하면 그때 접속하십시오.

Step 3-1 접속 노드(사용자 PC) ➔ AWS 엔드포인트 연결 확인

정상 연결 시 확인 사항:
  • Windows: ipconfig 시 'TAP-Windows Adapter'에 172 대역 할당됨.
  • Linux/Mac: ifconfig 시 utun 인터페이스에 172 대역 할당됨.

Step 3-2 접속 노드 ➔ Hub VPC (임시 EC2) 통신 테스트 

# Hub VPC 내 프라이빗 EC2(10.0.x.x)로 통신 확인
ping 10.0.1.50
curl -I http://10.0.1.50

Step 3-3 Transit Gateway(TGW) 라우팅 체인 검증

위치 (Location) 필수 경로 (Required Route)
VPN 엔드포인트 경로 10.251.0.0/16 ➔ Target: Association Subnet
Spoke 라우팅 테이블 172.16.0.0/22 ➔ Target: tgw-xxxx (반환 경로)

Step 3-4 접속 노드 ➔ Spoke VPC (EKS 노드) 연결 확인 

# Spoke 내 워커 노드(10.251.x.x)로 SSH 또는 HTTP 접근
ssh -i key.pem ec2-user@10.251.10.20

Phase 4. Node(k8s Client) ➔ EKS 클러스터 연결 및 제어

EKS API Server Access

1. kubeconfig 업데이트

aws eks update-kubeconfig --region [REGION] --name [CLUSTER_NAME]

2. 최종 명령 테스트

kubectl get nodes -o wide
