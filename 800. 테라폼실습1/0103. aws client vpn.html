Hub & Spoke 기반
AWS Client VPN 엔터프라이즈 실무 명세

본 문서는 AWS 관리 콘솔의 최신 업데이트 항목을 반영하여, 개별 사용자가 Mutual TLS (Mutual Transport Layer Security, 상호 전송 계층 보안) 인증을 통해 프라이빗 네트워크에 접속하는 전 과정을 상세히 기술합니다.

Phase 0. 인증 인프라 및 ACM(AWS Certificate Manager) 준비

0-1 Easy-RSA를 이용한 인증서(Certificate) 및 키(Key) 생성

Client VPN은 Server(서버)와 Client(클라이언트)가 서로를 검증하는 상호 인증 방식을 사용합니다. 이를 위해 사설 CA(인증 기관)를 구축합니다. 

# 1. 환경 초기화
git clone https://github.com/OpenVPN/easy-rsa.git
cd easy-rsa/easyrsa3
./easyrsa init-pki

# 2. CA(Certificate Authority, 인증 기관) 구축
./easyrsa build-ca nopass

# 3. 서버 인증서 및 개인키 생성 (AWS 콘솔 등록용)
./easyrsa build-server-full server nopass

# 4. 클라이언트 인증서 및 개인키 생성 (사용자 PC 배포용)
./easyrsa build-client-full user01.domain.tld nopass

0-2 ACM(AWS Certificate Manager, 인증서 관리자) 업로드

Console ➔ AWS Certificate Manager ➔ Import a certificate (인증서 가져오기)

SERVER CERT

  • Body: server.crt
  • Key: server.key
  • Chain: ca.crt

CLIENT CERT

  • Body: user01.domain.tld.crt
  • Key: user01.domain.tld.key
  • Chain: ca.crt

Phase 1. Transit Gateway(TGW) 기반 허브 네트워킹

TGW Attachment (Transit Gateway 연결) 구성

Hub Attachment

VPC: Hub-VPC (10.0.0.0/16)

VPN 엔드포인트가 배치될 게이트웨이 VPC입니다.

Spoke Attachment

VPC: Spoke-EKS (10.251.0.0/16)

실제 워크로드가 실행되는 목적지 VPC입니다.

참고: Transit Gateway(TGW, 전송 게이트웨이)는 리전 내의 여러 VPC를 중앙 집중식으로 연결하는 서비스로, 시간당 $0.05 및 데이터 처리당 비용이 발생합니다.

  • 현재 내 PC의 로컬 IP(사설 IP)가 아닙니다.
  • VPC의 CIDR 대역(10.0.x.x)과 절대 겹치면 안 됩니다.
  • EKS 포드(Pod) IP 대역과도 겹치면 안 됩니다.
  • 기존에 사용하지 않는 제3의 대역을 새로 생성하여 입력합니다. (최소 /22 권장)

Phase 3. 단계별 연결 확인 및 테스트 가이드

Step 3-1 접속 노드(사용자 PC) ➔ AWS 엔드포인트 연결 확인

가장 먼저 사용자 PC가 AWS Client VPN 엔드포인트와 TLS 터널을 정상적으로 생성했는지 확인합니다.

확인 방법:
  • 인터페이스 확인: ifconfig (mac/linux) 또는 ipconfig (windows) 명령어로 172.16.x.x 대역의 IP가 할당된 가상 어댑터가 생겼는지 확인합니다.
  • 라우팅 테이블 확인: netstat -rn 명령어로 10.x.x.x 대역의 게이트웨이가 VPN 어댑터로 지정되었는지 확인합니다.

Step 3-2 접속 노드 ➔ Hub VPC (임시 EC2) 통신 테스트

TGW를 타기 전, 엔드포인트가 연동된 Hub VPC 내의 자원과 통신이 되는지 먼저 점검합니다.

# Hub VPC 내 프라이빗 EC2(예: 10.0.1.50)로 핑 테스트
ping 10.0.1.50

# 보안 그룹에서 ICMP를 막아둔 경우 telnet이나 curl로 확인
curl -I http://10.0.1.50
실패 시 체크리스트:
1. Client VPN 엔드포인트의 Authorization Rules(권한 부여 규칙)에 10.0.0.0/16이 등록되었는가?
2. Hub EC2의 Security Group(보안 그룹) 인바운드 규칙에 172.16.0.0/22가 허용되었는가?

Step 3-3 Transit Gateway(TGW) 연결 및 라우팅 경로 검증

트래픽이 Hub를 넘어 Spoke로 가기 위해서는 3단계의 라우팅 체인이 완벽해야 합니다.

구분 (Section) 반드시 있어야 할 경로 (Required Route)
Client VPN Route 10.251.0.0/16 ➔ Target: Subnet-Association-ID
Hub VPC Route Table 10.251.0.0/16 ➔ Target: tgw-xxxxxx
Spoke VPC Route Table 172.16.0.0/22 ➔ Target: tgw-xxxxxx (응답용 경로)
TGW Route Table Propagation(전파) 상태 확인: Hub/Spoke 양쪽 CIDR이 등록되어 있어야 함

Step 3-4 접속 노드 ➔ Spoke VPC (EKS 노드/EC2) 연결 확인

최종 목적지인 Spoke VPC 내부 자원에 접근이 가능한지 확인합니다.

최종 L3 테스트: 
# Spoke VPC 내 EKS 워커 노드(예: 10.251.10.20) 접근 확인
ssh -i key.pem ec2-user@10.251.10.20

# 또는 HTTP 서비스가 떠 있다면 curl 테스트
curl 10.251.10.20:80

Phase 4. Node(k8s Client) ➔ EKS 클러스터 연결 및 제어

EKS API Server Access (클러스터 접속 제어)
💡 EKS 클러스터가 [Private Only] 모드인 경우 VPN 연결 없이는 kubectl 명령이 불가능합니다.

1. kubeconfig 업데이트

aws eks update-kubeconfig \
--region ap-northeast-2 \
--name spoke-cluster-master

2. 컨텍스트 확인

kubectl config current-context

최종 명령 테스트 및 검증 시나리오

1

노드 상태 조회 (L7 / API Check)

VPN을 통해 EKS 제어 평면(Control Plane)과 통신이 원활한지 확인합니다.

kubectl get nodes -o wide

✔ 성공 시: STATUS가 Ready인 노드 목록 출력

✖ 실패 시: Unable to connect to the server: dial tcp ... i/o timeout

2

포드 내부 서비스 접속 (Application Check)

프라이빗 로드밸런서(Internal ALB/NLB)를 생성하고 VPN망에서 접속되는지 확인합니다.

kubectl get svc -A

출력된 Internal DNS 주소를 로컬 브라우저에 입력하여 페이지가 뜨는지 확인합니다.

🚨 구축 및 확인 핵심 요약 (Verification Cheat Sheet)

  • 1. IP 충돌 방지: Client CIDR(172.16.x.x)이 모든 VPC 대역과 겹치지 않는지 재확인하십시오.
  • 2. 보안 그룹: Spoke VPC의 인스턴스는 172.16.x.x 대역을 신뢰할 수 있는 소스로 등록해야 합니다.
  • 3. 라우팅 전파: TGW에서 Hub와 Spoke가 서로의 라우팅을 학습(Propagation)했는지 TGW Route Table 콘솔에서 반드시 확인하십시오.
  • 4. 반환 경로(Return Path): Spoke VPC 라우팅 테이블에 172.16.0.0/22 대역이 TGW를 타겟으로 등록되어 있지 않으면 나가는 패킷은 성공해도 들어오는 패킷이 없어 통신이 단절됩니다.
