프로젝트 1: Terragrunt 기반 중앙 집중형 네트워크 아키텍처 구축

1. 프로젝트 소개

본 과정에서는 중견 기업 이상의 엔터프라이즈 환경에서 필수적으로 요구되는 Hub & Spoke (허브 앤 스포크) 기반의 네트워크 아키텍처를 Terragrunt (테라그런트, 인프라스트럭처 프로비저닝 래퍼 도구)를 활용하여 코드, 즉 IaC (Infrastructure as Code, 코드로서의 인프라)로 구축합니다.

외부에서 들어오는 모든 인바운드 트래픽과 내부에서 외부로 나가는 모든 아웃바운드 트래픽을 중앙의 Hub-VPC (Virtual Private Cloud, 가상 사설 클라우드)를 통제 지점으로 삼아 통과하게 함으로써, 보안 검사, 트래픽 모니터링, 그리고 비용 최적화를 달성하는 것이 핵심 목표입니다.[1]

[과금 주의 및 유료 서비스 안내]

본 실습 아키텍처에는 다음과 같은 리소스가 포함되며, AWS (Amazon Web Services, 아마존 웹 서비스) 프리티어(Free Tier)를 초과하여 시간당 고정 비용이 발생합니다. 실습 완료 후 반드시 리소스를 삭제(Destroy)해야 합니다.

  • TGW (Transit Gateway, 트랜짓 게이트웨이): 시간당 연결(Attachment) 비용 및 데이터 처리 비용 발생
  • NAT Gateway (Network Address Translation Gateway, 네트워크 주소 변환 게이트웨이): 시간당 구동 비용 및 데이터 처리 비용 발생
  • ALB (Application Load Balancer, 애플리케이션 로드 밸런서): 시간당 LCU (Load Balancer Capacity Units, 로드 밸런서 용량 단위) 비용 발생

2. 네트워크 구성도 및 아키텍처 흐름

네트워크 트래픽의 흐름은 명확하게 인바운드와 아웃바운드로 나뉘며, TGW가 핵심 라우터 역할을 수행합니다.

2.1 인바운드 흐름 (외부 접속)

외부 브라우저 → IGW (Hub-VPC) → ALB (Hub-VPC Public Subnet) → TGW → Nginx (Service-VPC Private Subnet)

사용자의 요청은 중앙 Hub-VPC의 IGW (Internet Gateway, 인터넷 게이트웨이)를 통해 들어와 ALB에서 트래픽 부하 분산이 이루어진 후, TGW를 타고 실제 서비스가 구동 중인 Service-VPC의 사설망(Private Subnet) 내부 Nginx 서버로 전달됩니다.

2.2 아웃바운드 흐름 (업데이트 등)

Nginx (Service-VPC) → TGW → NAT Gateway (Hub-VPC Public Subnet) → IGW → 외부 인터넷

내부 Nginx 서버가 패치 업데이트 등을 위해 인터넷이 필요할 경우, Service-VPC에는 외부로 나가는 통로가 없습니다. 트래픽은 TGW를 거쳐 Hub-VPC로 전달되고, Hub-VPC에 위치한 NAT Gateway를 통해 공인 IP(Public IP)로 변환된 후 외부 인터넷으로 나갑니다.

3. Terragrunt 디렉터리 구조도

중복 코드를 제거하고 환경(Dev, Prod 등) 간의 독립성을 유지하기 위해 (DRY, Don't Repeat Yourself 원칙 적용) 다음과 같은 계층적 디렉터리 구조를 설계합니다.

Directory Tree
infrastructure/
├── terragrunt.hcl              # 최상위 설정 파일 (Provider, S3 Backend 공통 설정)
└── prod/                       # 운영 환경 디렉터리
    ├── env.hcl                 # prod 환경 공용 변수 (Region, Environment Name 등)
    ├── 01-hub-vpc/             # Hub VPC 구축 로직
    │   └── terragrunt.hcl
    ├── 02-service-vpc/         # Service VPC 구축 로직
    │   └── terragrunt.hcl
    ├── 03-tgw/                 # Transit Gateway 본체 구축 로직
    │   └── terragrunt.hcl
    ├── 03-5-tgw-routing/       # TGW와 VPC 간의 라우팅 테이블 제어 로직
    │   └── terragrunt.hcl
    ├── 04-alb/                 # Application Load Balancer 구축 로직
    │   └── terragrunt.hcl
    └── 05-ec2-nginx/           # Nginx 인스턴스 구축 로직
        └── terragrunt.hcl
[실전 팁] 디렉터리 넘버링의 중요성

인프라스트럭처는 의존성(Dependency)이 존재합니다. VPC가 있어야 TGW를 붙일 수 있고, TGW가 있어야 라우팅을 할 수 있습니다. 디렉터리명 앞에 01, 02 숫자를 붙이면 terragrunt run-all apply 실행 시 오케스트레이터가 의도한 순서대로 실행 흐름을 파악하기 쉽고, 엔지니어가 의존성 오류를 디버깅할 때 매우 직관적인 아키텍처를 유지할 수 있습니다.

4. 상세 작업 내용과 각 단계별 HCL 코드

아래는 위에서 정의한 디렉터리 구조에 맞추어 실제로 작성해야 하는 HCL (HashiCorp Configuration Language, 해시코프 설정 언어) 코드 전문입니다.

4.1 최상위 설정 및 환경 변수 구성

가장 먼저 인프라의 상태를 저장할 원격 저장소와 클라우드 공급자(Provider) 버전을 강제하는 설정을 정의합니다.

파일 경로: infrastructure/terragrunt.hcl
# 로컬 변수(Local Variables)를 통해 하위 디렉터리의 환경 설정 파일을 읽어옵니다.
locals {
  env_vars   = read_terragrunt_config(find_in_parent_folders("env.hcl"))
  aws_region = local.env_vars.locals.aws_region
}

# Provider (클라우드 공급자) 자동 생성 설정 및 버전 강제화
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "aws" {
  region = "${local.aws_region}"
}

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"  # 최신 API(cpu_core_count 등) 지원을 위해 5.0 이상 강제
    }
  }
}
EOF
}

# S3 (Simple Storage Service, 단순 스토리지 서비스)를 활용한 원격 상태 저장소(Remote State Backend) 설정
remote_state {
  backend = "s3"
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite_terragrunt"
  }
  config = {
    bucket         = "my-org-terraform-state-bucket-unique" # 고유한 버킷 이름으로 변경 필요
    key            = "${path_relative_to_include()}/terraform.tfstate"
    region         = local.aws_region
    encrypt        = true
    dynamodb_table = "terraform-locks" # 상태 파일 동시성 제어 잠금을 위한 DynamoDB 테이블
  }
}
파일 경로: infrastructure/prod/env.hcl
# 환경별(운영, 개발 등) 공통 변수를 정의합니다.
locals {
  environment = "prod"
  aws_region  = "ap-northeast-2" # 아시아 태평양 (서울) 리전
}

4.2 Hub VPC 생성 (네트워크의 중심)

외부와 직접 통신하며 NAT Gateway를 호스팅하는 중심 VPC입니다.

파일 경로: infrastructure/prod/01-hub-vpc/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

# 오픈소스 Terraform AWS VPC 모듈 사용
terraform {
  source = "tfr:///terraform-aws-modules/vpc/aws?version=5.0.0"
}

inputs = {
  name = "hub-vpc"
  cidr = "10.0.0.0/16" # Hub VPC의 전체 IP 대역
  
  azs             = ["ap-northeast-2a", "ap-northeast-2c"]
  public_subnets  = ["10.0.1.0/24", "10.0.2.0/24"]
  private_subnets = ["10.0.11.0/24", "10.0.12.0/24"]

  # NAT Gateway (네트워크 주소 변환 게이트웨이) 활성화
  enable_nat_gateway = true
  single_nat_gateway = true # 실습 비용 절감을 위해 1개만 생성 (운영 환경에서는 HA를 위해 false 권장)
  
  tags = {
    Environment = "prod"
    Role        = "hub"
  }
}

4.3 Service VPC 생성 (격리된 서비스 영역)

실제 애플리케이션(Nginx)이 배포되는 영역으로, 직접적인 인터넷 연결 통로(IGW)나 NAT가 없습니다.

파일 경로: infrastructure/prod/02-service-vpc/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

terraform {
  source = "tfr:///terraform-aws-modules/vpc/aws?version=5.0.0"
}

inputs = {
  name = "service-vpc"
  cidr = "10.1.0.0/16" # Service VPC의 전체 IP 대역 (Hub VPC와 겹치지 않아야 라우팅 가능)
  
  azs             = ["ap-northeast-2a", "ap-northeast-2c"]
  private_subnets = ["10.1.1.0/24", "10.1.2.0/24"] # Public 서브넷 없음

  # 자체 NAT를 가지지 않고 Hub VPC의 NAT를 활용하기 위해 비활성화
  enable_nat_gateway = false 

  tags = {
    Environment = "prod"
    Role        = "service"
  }
}

4.4 TGW (Transit Gateway) 생성 및 Attachment 구성

Hub VPC와 Service VPC를 연결하는 핵심 중앙 라우터입니다. 각 VPC를 연결(Attachment)합니다.

파일 경로: infrastructure/prod/03-tgw/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

# 이전 단계에서 생성된 VPC들의 출력(Outputs) 값을 가져오기 위한 의존성 선언
dependency "hub_vpc" {
  config_path = "../01-hub-vpc"
}
dependency "service_vpc" {
  config_path = "../02-service-vpc"
}

terraform {
  source = "tfr:///terraform-aws-modules/transit-gateway/aws?version=2.10.0"
}

inputs = {
  name = "main-tgw"
  
  # AWS RAM(Resource Access Manager)을 통한 타 계정 공유 비활성화 (단일 계정 실습용)
  share_tgw = false
  
  # TGW에 VPC들을 연결(Attachment)
  vpc_attachments = {
    hub = {
      vpc_id       = dependency.hub_vpc.outputs.vpc_id
      subnet_ids   = dependency.hub_vpc.outputs.private_subnets # TGW ENI가 배치될 서브넷
      transit_gateway_default_route_table_association = true
      transit_gateway_default_route_table_propagation = true
    }
    service = {
      vpc_id       = dependency.service_vpc.outputs.vpc_id
      subnet_ids   = dependency.service_vpc.outputs.private_subnets
      transit_gateway_default_route_table_association = true
      transit_gateway_default_route_table_propagation = true
    }
  }
}

4.4.1 TGW VPC 라우팅 테이블 업데이트 (의존성 분리 아키텍처)

TGW 연결이 완료되더라도, 각 VPC의 라우팅 테이블에 명시적인 경로가 없으면 트래픽은 목적지를 찾지 못하고 폐기(Drop)됩니다. VPC와 TGW 간의 순환 참조(Circular Dependency) 에러를 방지하기 위해, VPC와 TGW가 모두 생성된 직후 라우팅 테이블만 독립적으로 업데이트하는 전용 모듈을 구성합니다.

파일 경로: infrastructure/prod/03-5-tgw-routing/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

# 라우팅에 필요한 기존 리소스(VPC, TGW)의 ID 값을 가져옵니다.
dependency "hub_vpc" { config_path = "../01-hub-vpc" }
dependency "service_vpc" { config_path = "../02-service-vpc" }
dependency "tgw" { config_path = "../03-tgw" }

# 순수 Terraform 코드를 동적으로 생성하여 라우팅 규칙을 주입합니다.
generate "routes" {
  path      = "vpc_routes.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
variable "hub_public_route_tables" { type = list(string) }
variable "service_private_route_tables" { type = list(string) }
variable "tgw_id" { type = string }

# 1. Hub VPC (Public) -> Service VPC (10.1.0.0/16) 라우팅
# ALB가 Service VPC 내부의 Nginx로 트래픽을 전달하기 위한 경로입니다.
resource "aws_route" "hub_to_service" {
  count                  = length(var.hub_public_route_tables)
  route_table_id         = var.hub_public_route_tables[count.index]
  destination_cidr_block = "10.1.0.0/16"
  transit_gateway_id     = var.tgw_id
}

# 2. Service VPC (Private) -> 외부 인터넷 (0.0.0.0/0) 라우팅
# Nginx 인스턴스가 패치 업데이트를 위해 Hub VPC의 NAT Gateway로 나가기 위한 기본 경로입니다.
resource "aws_route" "service_to_hub" {
  count                  = length(var.service_private_route_tables)
  route_table_id         = var.service_private_route_tables[count.index]
  destination_cidr_block = "0.0.0.0/0"
  transit_gateway_id     = var.tgw_id
    
# 3. HUB VPC (Private) -> 외부 인터넷 (0.0.0.0/0) 라우팅
resource "aws_route" "hubpub_igw" {
  count                  = length(var.hubpub_igw_route_tables)
  route_table_id         = var.hubpub_igw_route_tables[count.index]
  destination_cidr_block = "0.0.0.0/0"
  gateway_id     = var.hub_igw_id
}

EOF
}

# 동적 생성된 Terraform 코드의 변수에 매핑할 실제 값을 주입합니다.
inputs = {
  hub_public_route_tables      = dependency.hub_vpc.outputs.public_route_table_ids
  hub_pub_igw_route_tables     = dependency.hub_vpc.outputs.public_route_table_ids
  service_private_route_tables = dependency.service_vpc.outputs.private_route_table_ids
  tgw_id                       = dependency.tgw.outputs.ec2_transit_gateway_id
  hub_igw_id                   = dependency.hub_vpc.outputs.igw_id

    

    
}
EOF
}

# 동적 생성된 Terraform 코드의 변수에 매핑할 실제 값을 주입합니다.
inputs = {
  hub_public_route_tables      = dependency.hub_vpc.outputs.public_route_table_ids
  service_private_route_tables = dependency.service_vpc.outputs.private_route_table_ids
  tgw_id                       = dependency.tgw.outputs.ec2_transit_gateway_id
}
[자주 오해하거나 실수하는 부분] 비대칭 라우팅(Asymmetric Routing)

TGW 연결 이후 각 VPC의 라우팅 테이블(Route Table)을 직접 수정하여, Service-VPC의 0.0.0.0/0 대상이 TGW를 향하도록 하고, Hub-VPC는 Service-VPC의 CIDR(10.1.0.0/16) 요청 시 TGW로 응답하도록 세밀하게 라우팅을 매핑해야 합니다. 패킷이 나간 길과 돌아오는 길이 달라지는 비대칭 라우팅이 발생하면 방화벽이나 클라우드 인프라 레벨에서 비정상 트래픽으로 간주하여 즉시 유실(Drop)됩니다.

4.5 중앙 ALB (Application Load Balancer) 구성

Hub VPC의 Public 영역에 위치하며, 외부 트래픽을 안전하게 Service VPC 내부로 전달합니다.

파일 경로: infrastructure/prod/04-alb/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

dependency "hub_vpc" {
  config_path = "../01-hub-vpc"
}

terraform {
  source = "tfr:///terraform-aws-modules/alb/aws?version=8.7.0"
}

inputs = {
  name               = "hub-central-alb"
  load_balancer_type = "application"
  vpc_id             = dependency.hub_vpc.outputs.vpc_id
  subnets            = dependency.hub_vpc.outputs.public_subnets # 외부에 노출
  
  # HTTP 80 포트 리스너 설정
  http_tcp_listeners = [
    {
      port               = 80
      protocol           = "HTTP"
      target_group_index = 0
    }
  ]

  # 타겟 그룹 (Target Group) 설정
  target_groups = [
    {
      name             = "service-nginx-tg"
      backend_protocol = "HTTP"
      backend_port     = 80
      # 다른 VPC의 인스턴스를 지정하기 위해 반드시 'ip' 대상 유형(Target Type) 사용
      target_type      = "ip" 
      # 이 부분의 IP는 다음 단계에서 생성될 EC2의 Private IP를 콘솔에서 수동/자동 매핑해야 합니다.
    }
  ]
}

4.6 컴퓨팅 자원 배포 (Service VPC 내부 Nginx)

실제 서비스를 담당할 인스턴스를 외부에서 접근 불가능한 프라이빗 환경에 안전하게 배포합니다.

파일 경로: infrastructure/prod/05-ec2-nginx/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

dependency "service_vpc" {
  config_path = "../02-service-vpc"
}

terraform {
  source = "tfr:///terraform-aws-modules/ec2-instance/aws?version=5.0.0"
}

inputs = {
  name = "nginx-service-instance"
  
  instance_type = "t3.micro"
  ami           = "ami-0c9c942bd7bf113a2" # Ubuntu 22.04 LTS (ap-northeast-2 기준)
  
  vpc_security_group_ids = [dependency.service_vpc.outputs.default_security_group_id]
  subnet_id              = dependency.service_vpc.outputs.private_subnets[0]

  # User Data를 통한 Nginx 자동 설치 셸 스크립트
  # 인터넷(아웃바운드)이 정상 연결되어야 apt-get 패키지 다운로드가 성공합니다.
  user_data = <<-EOF
              #!/bin/bash
              apt-get update -y
              apt-get install -y nginx
              systemctl start nginx
              systemctl enable nginx
              echo "Welcome to Centralized Network Nginx!" > /var/www/html/index.html
              EOF

  tags = {
    Environment = "prod"
    Service     = "web"
  }
}

5. 실행 방법 및 플러그인 버전 관리

대규모 다중 모듈 환경에서는 도구와 플러그인의 버전 충돌을 방지하고, 전체 의존성 트리(Dependency Tree)를 검증하는 과정이 필수적입니다. 모든 코드가 작성되었다면 최상위 디렉터리(infrastructure/prod/)에서 다음 명령어들을 순차적으로 실행합니다.

Terminal (Git Bash / PowerShell)
# 1. 플러그인 강제 초기화 및 업그레이드 (Init & Upgrade)
# 로컬 캐시의 구버전 프로바이더 플러그인을 무시하고, 코드에 명시된 최신 호환 버전으로 환경을 갱신합니다.
terragrunt run-all init -upgrade

# 2. 변경 사항 전체 미리보기 및 검증 (Plan)
# 클라우드에 실제 변경을 가하기 전, 논리적 오류와 생성될 리소스 목록을 화면에 출력하여 사전 검증합니다.
terragrunt run-all plan

# 3. 인프라 실제 배포 적용 (Apply)
# 검증이 완료되면 실제 AWS 클라우드에 리소스를 프로비저닝합니다. 
# 01-hub-vpc부터 05-ec2-nginx까지 의존성 그래프에 따라 병렬/순차적으로 완벽하게 자동 배포됩니다.
terragrunt run-all apply
[실전 팁] run-all 환경에서의 Plan 파일 사용 주의사항

단일 디렉터리 구조에서는 plan -out=file 옵션으로 실행 계획을 파일로 저장하는 것이 정석입니다. 하지만 Terragrunt의 run-all 다중 모듈 환경에서는 -out 옵션을 사용할 경우, 파편화된 다수의 임시 해시(Hash) 캐시 폴더 안으로 계획 파일이 흩어져 apply 단계에서 경로 추적 에러(Path Resolution Failure)가 발생합니다. 따라서 run-all 명령어 사용 시에는 -out 옵션을 배제하고, 화면에 출력되는 의존성 검증 로그를 육안으로 승인(Approve)하는 방식을 채택해야 합니다.[3]

[부록] CLI (Command Line Interface, 명령줄 인터페이스) 도구 버전 관리

현업에서는 프로젝트마다 도구의 버전을 일치시키기 위해 버전 관리 매니저를 사용하는 것이 표준입니다.

Terminal (버전 관리 도구 활용)
# tfenv(Terraform Version Manager, 테라폼 버전 관리자)를 통한 테라폼 버전 세팅
tfenv install latest
tfenv use latest

# tgswitch(Terragrunt Version Switcher, 테라그런트 버전 전환기)를 통한 자동 스위칭
tgswitch    

# Windows (Chocolatey) 환경 (관리자 권한 실행 필요)
choco upgrade terraform -y
choco upgrade terragrunt -y

6. 브라우저 확인 방법 및 검증

배포가 완료되면 터미널에 출력된 ALB의 DNS (Domain Name System, 도메인 네임 시스템) 주소를 복사합니다.

  • 인바운드 라우팅 확인: 웹 브라우저 주소창에 ALB DNS 주소를 입력합니다. Nginx의 환영 페이지("Welcome to Centralized Network Nginx!")가 보인다면 외부 → Hub → TGW → Service 로의 인바운드 라우팅이 완벽하게 구성된 것입니다.
  • 아웃바운드 라우팅 확인:
    AWS SSM (Systems Manager, 시스템 매니저)의 Session Manager를 통해 Service-VPC 내부의 Nginx 인스턴스에 안전하게 접속합니다. 터미널에서 curl https://www.google.com 명령어를 실행하여 HTML 코드가 반환되는지 확인합니다. 이는 트래픽이 Service → TGW → Hub VPC NAT → 외부 인터넷으로 완벽하게 나갔음을 증명합니다.

1. Hub & Spoke 아키텍처: 자전거 바퀴의 중심(Hub)과 빗살(Spoke) 모양에서 유래한 네트워크 토폴로지입니다. 모든 통신이 중앙 Hub를 거치게 함으로써, 각 Spoke VPC마다 NAT Gateway나 방화벽을 개별적으로 두지 않아도 되어 관리의 편의성과 비용 절감 효과를 얻을 수 있는 엔터프라이즈급 표준 아키텍처입니다.

2. Target Type (대상 유형): 로드밸런서가 트래픽을 전달할 대상의 종류입니다. 인스턴스 ID(Instance), IP 주소(IP), 람다 함수(Lambda) 등이 있으며, TGW를 넘어가는 트래픽은 식별을 위해 IP 주소를 직접 매핑하는 방식을 사용해야 합니다.

3. Plan File (계획 파일, .out 파일): IaC 도구가 코드를 분석하여 실제로 어떤 리소스를 생성, 수정, 삭제할지 미리 계산해 둔 바이너리 파일입니다. 이를 통해 멱등성(Idempotence, 연산을 여러 번 적용하더라도 결과가 달라지지 않는 성질)을 안전하게 보장합니다.
