심층 방어 아키텍처: IDS/IPS 개념 및 AWS Network Firewall 성능 분석

침입 탐지 및 방지 시스템의 본질과 패러다임 변화

네트워크 인프라를 보호하기 위한 방어선은 끊임없이 진화해 왔습니다. 단순한 IP와 Port 기반의 접근 제어만으로는 패킷(Packet) 내부에 교묘하게 숨겨진 악성 페이로드(Payload)를 식별할 수 없습니다. 이러한 한계를 극복하고 Deep Packet Inspection (DPI, 심층 패킷 분석)을 수행하여 악의적인 행위를 식별하는 시스템이 바로 IDS (Intrusion Detection System, 침입 탐지 시스템)와 IPS (Intrusion Prevention System, 침입 방지 시스템)입니다.

* Deep Packet Inspection (심층 패킷 분석): 네트워크 트래픽의 헤더 정보(IP, Port)뿐만 아니라 실제 데이터가 담긴 페이로드(Payload) 영역까지 뜯어보어 악성코드나 비정상적인 프로토콜 사용을 검사하는 고도화된 기술입니다.

IDS (침입 탐지 시스템)의 한계와 역할

IDS는 네트워크 트래픽을 '수동적(Passive)'으로 모니터링하여 공격의 징후를 발견하고 관리자에게 경고(Alert)를 보내는 감시 카메라와 같은 역할을 합니다. 네트워크 스위치의 Mirror Port (미러 포트)나 SPAN (Switched Port Analyzer, 스위치 포트 분석기) 포트에 연결되어 실제 트래픽의 '복사본'을 분석합니다.

IDS는 실제 트래픽 흐름에 개입하지 않으므로 시스템 장애가 발생하더라도 전체 네트워크 서비스에는 영향을 미치지 않습니다(Out-of-band 방식). 하지만 트래픽의 복사본을 분석하여 경고를 띄우는 시점에는 이미 공격 패킷이 대상 서버에 도달해버린 후일 가능성이 높다는 치명적인 한계가 있습니다. 즉, 사후 대응(Post-incident Response)에 그치게 됩니다.

IPS (침입 방지 시스템)의 등장과 능동적 방어

이러한 IDS의 한계를 극복하기 위해 등장한 것이 능동적 방어 체계인 IPS입니다. IPS는 트래픽이 지나가는 경로의 중간에 직접 위치하는 In-line (인라인) 방식으로 구성됩니다. 들어오는 모든 패킷을 실시간으로 검사하고, 악의적인 패킷으로 판명될 경우 관리자에게 경고를 보내는 동시에 해당 패킷을 즉시 폐기(Drop) 하거나 연결을 강제로 끊어(Reset) 공격이 내부 서버에 도달하는 것을 원천 차단합니다.

* Mirror Port / SPAN: 네트워크 스위치를 통과하는 트래픽의 복사본을 만들어 특정 포트로 전달하는 기술입니다. 모니터링 장비(IDS 등)가 실제 통신에 영향을 주지 않고 데이터를 분석할 수 있게 해줍니다.
* In-line (인라인): 데이터 통신 경로의 한가운데 장비가 물리적 또는 논리적으로 배치되어, 모든 데이터가 해당 장비를 통과해야만 목적지로 갈 수 있는 구성 방식을 의미합니다.

아키텍처 도식화: IDS와 IPS의 트래픽 흐름 비교

아래의 도식은 Out-of-band(대역 외)로 구성되는 IDS와 In-line(대역 내)으로 구성되는 IPS의 물리적, 논리적 트래픽 흐름 차이를 명확하게 보여줍니다.

그림 1. IDS와 IPS의 네트워크 위치 및 방어 매커니즘 비교

IPS의 핵심 탐지 및 방어 매커니즘

성능이 뛰어난 IPS는 수많은 트래픽 속에서 정상적인 요청과 악의적인 공격을 지연 시간(Latency) 없이 순식간에 구별해 내야 합니다. 이를 위해 크게 세 가지의 엔진 기술을 복합적으로 사용합니다.

  • Signature-based Detection (시그니처 기반 탐지): 가장 널리 쓰이는 방식으로, 알려진 공격 패턴(바이러스 서명 등)의 데이터베이스와 현재 패킷을 1:1로 비교합니다. 정확도가 매우 높고 오탐률(False Positive)이 낮지만, 알려지지 않은 공격(Zero-Day Attack)은 탐지할 수 없습니다.
  • Anomaly-based Detection (이상 징후 기반 탐지): 머신 러닝이나 통계적 모델링을 통해 네트워크의 '정상적인 기준선(Baseline)'을 학습합니다. 이 기준선을 벗어나는 비정상적인 트래픽 급증이나 기이한 포트 스캔 행위가 발생하면 공격으로 간주합니다. 미지의 공격을 막을 수 있지만 오탐률이 상대적으로 높습니다.
  • Stateful Protocol Analysis (상태 기반 프로토콜 분석): 단순히 패킷 하나만 보는 것이 아니라, 클라이언트와 서버 간에 맺어진 전체 세션(Session)의 연결 상태를 추적합니다. 프로토콜의 표준 규격(RFC)을 위반하는 기형적인 패킷이나 순서가 맞지 않는 조작된 패킷을 차단합니다.
자주 오해하거나 실수하는 부분: "IPS와 WAF는 같은 거 아닌가요?" 매우 자주 혼동되는 개념입니다. 아키텍트로서 이 둘을 명확히 구분하여 설계해야 합니다.

IPS (L3 ~ L7 전반): 네트워크 전체를 대상으로 하며 모든 포트와 프로토콜(TCP, UDP, ICMP 등)의 악성 페이로드 및 비정상 트래픽을 방어합니다. 주로 운영체제 취약점 공격, 웜 바이러스, DDoS 초기 탐지에 강합니다.
WAF (오직 L7 웹 통신): 오직 80(HTTP) 및 443(HTTPS) 포트로 들어오는 웹 애플리케이션 트래픽만 집중적으로 분석합니다. SQL 인젝션, 크로스 사이트 스크립팅(XSS) 등 웹 애플리케이션의 취약점을 노리는 공격 방어에 특화되어 있습니다. 완벽한 보안을 위해서는 IPS와 WAF를 직렬로 함께 배치(Defense in Depth, 심층 방어)해야 합니다.

AWS 환경에서의 IPS 아키텍처: AWS Network Firewall

전통적인 데이터센터(IDC)에서는 물리적인 IPS 장비를 랙(Rack)에 마운트하여 사용했습니다. 하지만 동적으로 확장되는 클라우드 환경에서는 고정된 용량의 하드웨어 장비는 병목 현상(Bottleneck)의 원인이 됩니다. AWS는 과거 파트너사의 가상 어플라이언스(Virtual Appliance)를 거쳐야 했던 복잡성을 해소하기 위해 완전 관리형 서비스인 AWS Network Firewall (AWS 네트워크 방화벽)을 출시했습니다.

참조: AWS Network Firewall 공식 문서

AWS Network Firewall은 단순한 L4 방화벽이 아닙니다. 이 서비스의 핵심은 오픈 소스 기반의 강력한 IPS 엔진인 Suricata (수리카타)를 내장하고 있다는 점입니다. 이를 통해 클라우드 환경에서도 온프레미스 수준 이상의 강력한 Stateful(상태 기반) 패킷 검사와 IPS 기능을 완벽하게 수행합니다.

* AWS Network Firewall: 모든 Amazon VPC(가상 사설 클라우드)에 필수적인 네트워크 보호 기능을 손쉽게 배포할 수 있도록 지원하는 고가용성 관리형 보안 서비스입니다.
* Suricata (수리카타): 개방형 정보 보안 재단(OISF)에서 개발한 강력한 오픈 소스 기반의 차세대 침입 탐지 및 방지 시스템(IDS/IPS) 엔진입니다. 복잡한 룰셋을 통해 심층 패킷 검사를 수행합니다.

AWS Network Firewall의 고유한 특징 및 아키텍처 구성요소

  1. VPC Endpoint (가상 사설 클라우드 엔드포인트) 기반의 투명한 라우팅: AWS Network Firewall을 생성하면, 지정된 서브넷에 Gateway Load Balancer Endpoint (GWLBE) 기술 기반의 방화벽 엔드포인트가 생성됩니다. 사용자는 VPC의 라우팅 테이블(Route Table)을 수정하여 인터넷으로 나가거나 들어오는 모든 트래픽이 이 엔드포인트를 거치도록 유도(Traffic Steering)함으로써 투명하게 In-line IPS를 구성할 수 있습니다.
  2. Suricata 호환 규칙 지원: 기업에서 이미 사용 중인 기존의 Suricata IPS 시그니처나, Proofpoint(Emerging Threats) 등 외부 위협 인텔리전스 피드에서 제공하는 룰셋을 그대로 AWS에 임포트(Import)하여 사용할 수 있습니다.
  3. 상태 기반(Stateful) 룰 그룹 유연성: 패킷의 방향(Forward/Return)을 이해하고 세션을 추적하며, 프로토콜 서명, 포트 번호, 그리고 정규 표현식을 사용한 페이로드 검사까지 완벽히 지원합니다.

AWS IPS (Network Firewall)의 성능 한계와 확장성 (Performance & Scalability)

클라우드 아키텍트가 보안 솔루션을 도입할 때 가장 우려하는 것은 "보안 장비가 전체 네트워크의 속도를 저하시키지 않을까?" 하는 점입니다. AWS Network Firewall은 클라우드 네이티브 아키텍처를 통해 이러한 우려를 해소합니다.

성능 지표 (Performance Metrics) 상세 설명 및 아키텍트 고려사항
고가용성 및 무한 확장 (Auto Scaling) 방화벽 인스턴스는 사용자의 개입 없이 트래픽 로드에 따라 백그라운드에서 자동으로 확장(Scale-out) 및 축소(Scale-in)됩니다. 트래픽 스파이크(Spike)가 발생해도 병목 현상이 없습니다.
처리량 (Throughput) 가용 영역(Availability Zone, AZ) 당 최대 100 Gbps의 방화벽 처리 성능을 제공합니다. 다중 AZ로 구성할 경우 수백 Gbps 규모의 엔터프라이즈 트래픽도 안정적으로 IPS 검사가 가능합니다.
지연 시간 (Latency) 심층 패킷 분석(DPI)을 수행함에도 불구하고 일반적인 상황에서 마이크로초(Microseconds) 단위에서 수 밀리초(ms) 단위의 매우 낮은 지연 시간을 보장하여 서비스 품질(QoS)에 영향을 주지 않습니다.
아키텍트 필수 체크: AWS Network Firewall 과금 모델 매우 높은 과금 발생 주의 AWS Network Firewall은 구축형 장비와 달리 상당히 높은 종량제 비용이 발생하므로 예산 산정이 필수적입니다.

1. 엔드포인트 시간당 요금: 방화벽이 구성된 가용 영역(AZ)의 개수마다 시간당 고정 비용이 발생합니다. (보통 최소 2개 이상의 AZ에 구성하므로 월 기본 유지비가 수백 달러에 달합니다.)
2. 데이터 처리 요금 (GB당): 방화벽 엔진(IPS)을 통과하여 검사받는 모든 트래픽 용량(GB) 단위로 과금이 발생합니다.

따라서 동영상 스트리밍이나 대용량 파일 다운로드와 같이 굳이 IPS 패킷 딥 인스펙션(DPI)이 필요 없는 대규모 트래픽은 방화벽을 우회하도록 라우팅 설계를 최적화하는 것이 비용 절감의 핵심입니다.

엔터프라이즈를 위한 중앙 집중식 IPS 아키텍처 도식 (Centralized Inspection)

수십, 수백 개의 VPC를 운영하는 엔터프라이즈 환경에서 모든 VPC마다 방화벽을 개별적으로 구축하면 비용이 기하급수적으로 증가하고 정책 관리가 불가능해집니다. 최적의 아키텍처는 AWS Transit Gateway (TGW, 전송 게이트웨이)와 전용 Inspection VPC (검사 전용 VPC)를 결합하여 모든 트래픽을 한 곳으로 모아 IPS 검사를 수행하는 중앙 집중식 방화벽 아키텍처입니다.

* AWS Transit Gateway (TGW): 클라우드 라우터 역할을 하며 수천 개의 VPC와 온프레미스 네트워크를 중앙에서 연결해 주는 네트워크 허브 서비스입니다.
* Inspection VPC (검사 전용 VPC): 애플리케이션 서버를 배치하지 않고, 오직 보안 검사 장비(Network Firewall, 타사 IPS 등)만을 배치하여 트래픽을 필터링하는 목적의 전용 네트워크 공간입니다.
그림 2. AWS Transit Gateway를 활용한 중앙 집중식 Network Firewall (IPS) 검사 아키텍처

성공적인 IPS 운영을 위한 실전 팁 및 고려사항

아키텍트로서 IPS 장비를 클라우드에 성공적으로 안착시키기 위해서는 라우팅의 흐름과 보안 정책의 속성을 깊이 이해해야 합니다.

실무 아키텍트의 팁 1: 비대칭 라우팅 (Asymmetric Routing) 방지 네트워크 아키텍처에서 가장 자주 겪는 장애 원인입니다. In-line 모드의 IPS는 Stateful(상태 기반) 검사를 수행하므로, 클라이언트가 서버로 보내는 요청(Request) 패킷과 서버가 응답(Response)하는 패킷이 반드시 동일한 IPS 방화벽 장비(엔드포인트)를 통과해야 합니다. 들어올 때는 A 가용 영역의 방화벽을 탔는데 나갈 때는 B 가용 영역의 방화벽을 타게 되면, 방화벽은 비정상적인 세션으로 인식하고 즉시 패킷을 버립니다(Drop). VPC 라우팅 테이블 및 Transit Gateway 어플라이언스 모드(Appliance Mode) 설정을 정교하게 구성하여 비대칭 라우팅이 발생하지 않도록 강제해야 합니다.
실무 아키텍트의 팁 2: 프로덕션 환경 적용 전 Count 모드 활용 IPS는 공격을 능동적으로 차단(Drop)하기 때문에, 룰셋이 잘못 설정되면 정상적인 고객의 결제 요청이나 로그인 트래픽마저 차단되는 대형 장애(False Positive)를 유발할 수 있습니다. AWS Network Firewall에 새로운 Suricata 룰을 적용할 때에는, 초기 1~2주간 액션을 'Drop'이 아닌 'Alert/Count (로깅만 수행)' 모드로 설정하여 운영해야 합니다. 로그를 분석하여 정상 트래픽 차단이 발생하지 않음을 완벽히 검증한 후, 최종적으로 'Drop' 액션으로 전환하는 것이 안전한 시스템 운영의 기본 원칙입니다.

결론적으로, 현대의 보안 인프라에서 IPS 기능은 필수불가결합니다. 전통적인 IDS의 사후 보고 한계를 넘어 실시간 방어를 제공하며, AWS 환경에서는 Network Firewall과 Suricata 엔진의 결합을 통해 클라우드의 유연성(Auto Scaling)과 온프레미스의 강력한 보안 정책을 모두 충족하는 고도의 방어 체계를 구축할 수 있습니다.
