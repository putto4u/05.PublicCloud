AWS Network Firewall 보안 정책 및 규칙 설계 전문가 가이드

본 가이드는 대규모 클라우드 인프라 환경에서 Network Firewall (NFW, 네트워크 방화벽)[유료]을 안전하고 효율적으로 설계하기 위한 실무 권장 사항을 다룹니다. 규칙 그룹의 계층적 구성부터 패킷의 심층 검사, 도메인 필터링, 그리고 고가용성 아키텍처 연동까지 아키텍트 관점에서의 최적화 방안을 제시합니다.

1. 방화벽 정책(Firewall Policy) 아키텍처 및 구성 요소

AWS Network Firewall (NFW, 네트워크 방화벽)[유료]은 단일 장애점(SPOF) 없이 트래픽을 검사할 수 있도록 설계된 관리형 서비스입니다. 방화벽의 설정은 논리적으로 세 가지 계층으로 구분됩니다.

  • Firewall (방화벽): VPC 내의 특정 서브넷에 생성되는 물리적/논리적 엔드포인트입니다.
  • Firewall Policy (방화벽 정책): 트래픽이 어떤 순서로 규칙을 통과할지, 그리고 규칙에 맞지 않는 트래픽을 어떻게 처리할지(Default Action)를 정의하는 뼈대입니다.
  • Rule Group (규칙 그룹): 실제 패킷을 허용(Pass), 차단(Drop), 또는 경고(Alert)할지 결정하는 상세 조건들의 모음입니다.
실전 팁: 방화벽 정책은 여러 방화벽(Firewall) 객체에서 재사용할 수 있습니다. 개발(Dev), 검증(Staging), 운영(Prod) 환경에 동일한 보안 기준을 적용하려면 중앙에서 하나의 '방화벽 정책'을 만들어 여러 VPC의 방화벽에 연결하는 것이 효율적입니다.
  • NFW (Network Firewall, 네트워크 방화벽): VPC 트래픽에 대해 정밀한 제어 및 가시성을 제공하는 고가용성 관리형 방화벽 서비스입니다. (엔드포인트 및 트래픽 처리량에 따라 과금)
  • SPOF (Single Point of Failure, 단일 장애점): 시스템 구성 요소 중 동작하지 않으면 전체 시스템이 중단되는 요소입니다. AWS NFW는 이를 방지하기 위해 Multi-AZ로 확장되도록 설계되었습니다.
출처: AWS Network Firewall Developer Guide

2. 정책 기본 동작(Default Action) 및 평가 순서 설계

네트워크 방화벽 정책 설계의 첫걸음은 Default Action (기본 동작)의 정의입니다. 트래픽이 방화벽에 도달하면 먼저 Stateless (상태 비저장) 엔진을 거치고, 여기서 처리되지 않은 트래픽이 Stateful (상태 저장) 엔진으로 넘어갑니다.

권장 사항: Default Deny (기본 거부) 아키텍처

실무 환경에서는 반드시 명시적으로 허용된 트래픽 외에는 모두 차단하는 Default Deny (기본 거부) 전략을 적용해야 합니다.

구분 권장 설정값 설명
Stateless Default Action Forward to stateful rule groups 기본적인 IP/Port 필터링을 통과한 모든 패킷은 심층 검사를 위해 상태 저장 엔진으로 넘깁니다.
Stateful Default Action Drop established / Drop all 상태 저장 규칙(허용 리스트)에 매칭되지 않는 모든 잔여 트래픽은 안전하게 폐기합니다.
  • Stateless (상태 비저장): 네트워크 연결의 컨텍스트(세션)를 무시하고 각 패킷을 독립적으로 검사하는 방식입니다.
  • Stateful (상태 저장): 트래픽의 방향, 연결 상태(3-Way Handshake 등)를 추적하며 검사하는 방식입니다.
  • Default Deny (기본 거부): 보안의 기본 원칙으로, 정책에 정의되지 않은 모든 접근을 암묵적으로 차단하는 것을 말합니다.

3. 상태 비저장 규칙 그룹(Stateless Rule Groups) 최적화

Stateless Rule (상태 비저장 규칙)은 패킷의 페이로드(내용)를 보지 않고 헤더(IP, Port, Protocol)만 검사하므로 처리 속도가 매우 빠릅니다. 이 계층을 어떻게 활용하느냐에 따라 방화벽 전체의 성능과 Data Processing (데이터 처리)[유료] 비용이 결정됩니다.

실전 팁: 성능 튜닝 및 비용 절감
모든 트래픽을 상태 저장 엔진(Stateful)으로 보내면 성능 저하와 검사 비용이 증가합니다. 완전히 신뢰할 수 있는 내부 통신(예: 모니터링 서버 통신, 백업 트래픽)은 상태 비저장 규칙에서 Pass (통과) 동작을 적용하여 상태 저장 검사를 Bypass (건너뛰기) 하도록 설계하십시오. 반대로, 알려진 악성 IP 대역은 여기서 Drop (차단)하여 엔진 부하를 줄입니다.

상태 비저장 규칙은 Priority (우선순위) 값을 가집니다. 값이 낮을수록 우선적으로 평가되므로, Drop 규칙을 가장 낮은 숫자(최우선)로 배치하는 것이 안전합니다.

  • Data Processing (데이터 처리): 방화벽 엔진을 통과하는 트래픽 용량(GB)에 따라 부과되는 과금 항목입니다.
  • Bypass (바이패스): 특정 보안 검사 단계를 거치지 않고 트래픽을 직접 통과시키는 네트워크 제어 방식입니다.
  • Priority (우선순위): 여러 규칙이 충돌할 때 어떤 규칙을 먼저 적용할지 결정하는 순서 값입니다.

4. 상태 저장 규칙(Stateful Rule) 및 Suricata IPS 엔진

Stateful Rule (상태 저장 규칙)은 NFW의 핵심인 Intrusion Prevention System (IPS, 침입 방지 시스템) 역할을 수행합니다. 내부적으로 오픈소스 IPS 엔진인 Suricata (수리카타)를 사용하여 Deep Packet Inspection (DPI, 심층 패킷 검사)를 실행합니다.

규칙 평가 순서 제어 (Strict Order)

과거에는 'Action Order(동작 순서)'가 기본이었으나, 실무에서는 반드시 Strict Order (엄격한 순서)를 사용해야 합니다. Strict Order를 활성화하면 관리자가 지정한 순서대로만 규칙이 평가되므로, 복잡한 정책 환경에서 예상치 못한 허용/차단을 방지할 수 있습니다.

자주 하는 실수: 비대칭 라우팅(Asymmetric Routing) 환경에서는 상태 저장 엔진이 TCP 세션의 흐름을 제대로 추적하지 못해 정상 트래픽을 Drop할 수 있습니다. 들어오는 경로와 나가는 경로가 동일한 NFW 엔드포인트를 거치도록 라우팅 테이블을 설계해야 합니다.
  • IPS (Intrusion Prevention System, 침입 방지 시스템): 네트워크에서 악의적인 활동을 탐지하고 능동적으로 차단하는 보안 시스템입니다.
  • Suricata (수리카타): 강력한 실시간 네트워크 위협 탐지 및 방지 기능을 제공하는 오픈소스 기반의 고성능 IPS 엔진입니다.
  • Deep Packet Inspection (DPI, 심층 패킷 검사): 네트워크 패킷의 헤더뿐만 아니라 데이터 본문(Payload) 영역까지 분석하여 악성코드나 비정상 통신을 식별하는 기술입니다.
  • Strict Order (엄격한 순서): 규칙의 우선순위를 관리자가 명시적으로 지정한 숫자 순서대로만 평가하도록 강제하는 옵션입니다.

5. 아웃바운드 제어: 도메인 리스트(FQDN) 필터링

내부 서버가 외부 인터넷으로 나갈 때(Egress), IP 주소 기반의 차단은 클라우드 환경에서 한계가 있습니다. NFW는 Fully Qualified Domain Name (FQDN, 전체 서비스 도메인 이름) 기반의 필터링을 지원하여, 특정 도메인(예: *.github.com)에 대한 접근만 허용하는 화이트리스트(Whitelist) 구현이 가능합니다.

동작 원리 및 프로토콜 제약

도메인 필터링은 HTTP의 Host 헤더 또는 HTTPS의 Server Name Indication (SNI, 서버 이름 지시자)를 분석하여 동작합니다. 즉, HTTP/HTTPS 트래픽에서만 유효하며, 일반 TCP/UDP 기반의 자체 프로토콜 트래픽은 도메인 이름으로 차단할 수 없습니다.

설계 권장 사항: 내부망의 NAT Gateway (네트워크 주소 변환 게이트웨이)[유료]를 거쳐 나가는 아웃바운드 트래픽에 대해, OS 업데이트(Ubuntu, RedHat) 및 필수 API 도메인만 허용(Allow)하고 나머지는 모두 차단(Drop)하도록 규칙 그룹을 구성하십시오.
  • FQDN (Fully Qualified Domain Name, 전체 서비스 도메인 이름): 호스트 이름과 도메인 이름이 결합되어 인터넷상에서 특정 대상을 절대적으로 식별할 수 있는 전체 도메인 이름입니다.
  • SNI (Server Name Indication, 서버 이름 지시자): HTTPS 통신 초기 단계(TLS Handshake)에서 클라이언트가 접속하려는 도메인 이름을 암호화 전에 평문으로 서버에 알려주는 확장 기술입니다.
  • Whitelist (화이트리스트): 모든 접근을 기본적으로 차단하고, 목록에 명시된 안전한 대상(IP 또는 도메인)과의 통신만 허용하는 보안 정책입니다.

6. AWS 관리형 규칙 그룹(Managed Rule Groups) 활용

자체적으로 모든 Suricata 규칙을 작성하고 유지보수하는 것은 막대한 운영 공수가 듭니다. AWS는 최신 위협 인텔리전스를 기반으로 자동 업데이트되는 Managed Rule Groups (관리형 규칙 그룹)을 제공합니다.

필수 적용 권장 관리형 규칙

규칙 그룹 범주 설명 및 권장 사항
Botnet C&C (명령 제어) 내부 감염된 서버가 외부 해커의 Command and Control (C&C, 명령 제어) 서버와 통신하는 것을 차단합니다. (필수 적용)
Malware (악성코드) 알려진 악성코드 유포지 다운로드 및 통신 패턴을 식별하고 방어합니다.
Threat Intelligence (위협 인텔리전스) AWS 보안 팀 및 외부 보안 파트너가 수집한 악성 IP 및 도메인 블랙리스트를 적용합니다.
실전 팁: 관리형 규칙을 처음 도입할 때는 즉시 Drop(차단)으로 설정하지 마십시오. 처음 1~2주는 Alert(경고) 모드로 설정하여 정상적인 비즈니스 트래픽이 오탐(False Positive)으로 차단되지 않는지 로그를 분석한 후 차단 모드로 전환해야 합니다.
  • Managed Rule Groups (관리형 규칙 그룹): 클라우드 제공자(AWS)가 최신 보안 위협을 분석하여 규칙을 생성하고 자동으로 업데이트해 주는 서비스 기능입니다.
  • Command and Control (C&C, 명령 제어 서버): 해커가 악성코드에 감염된 봇넷(좀비 PC)에 원격으로 명령을 내리고 탈취한 데이터를 수집하는 서버입니다.
  • False Positive (오탐지): 정상적인 통신이나 접근을 보안 시스템이 악성 위협으로 잘못 판단하여 차단하는 현상입니다.

7. TLS Inspection (암호화 트래픽 복호화 검사)

현대의 웹 트래픽은 90% 이상이 암호화(HTTPS)되어 있습니다. 기존의 IPS는 암호화된 트래픽 내부의 악성 페이로드를 탐지할 수 없습니다. 이를 해결하기 위해 NFW는 TLS Inspection (TLS 검사) 기능을 제공합니다.

이 기능은 AWS Certificate Manager (ACM, AWS 인증서 관리자)[유료]를 사용하여 트래픽을 복호화(인그레스)하거나, 자체 CA(인증 기관)를 통해 다시 암호화(이그레스)하여 내부의 숨겨진 공격(예: 암호화된 SQL 인젝션)을 검사합니다.

아키텍트 주의사항 (성능 및 개인정보): TLS 검사는 방화벽 엔진에 막대한 연산 부하를 일으킵니다. 성능 저하를 방지하기 위해 반드시 필요한 고위험 구간에만 선별적으로 적용해야 합니다. 또한, 의료/금융 정보 등 민감한 데이터가 포함된 트래픽을 복호화할 경우 컴플라이언스(법규 준수) 위반 소지가 있으므로 법무팀과 협의가 필수적입니다.
  • TLS Inspection (TLS 검사): 암호화된 네트워크 트래픽(HTTPS 등)을 방화벽 내부에서 일시적으로 복호화하여 악성코드 은닉 여부를 검사한 뒤 다시 암호화하여 전송하는 보안 기술입니다.
  • ACM (AWS Certificate Manager, AWS 인증서 관리자): AWS 서비스 및 내부 연결에 사용할 공인 및 사설 SSL/TLS 인증서를 프로비저닝, 관리, 배포하는 서비스입니다.
  • Compliance (컴플라이언스, 규정 준수): 기업이 비즈니스를 수행함에 있어 관련 법률, 규정, 지침 등을 철저히 준수하는 것을 의미합니다.

8. 로깅 아키텍처 및 트래픽 모니터링

NFW는 강력한 방어 도구일 뿐만 아니라, 훌륭한 네트워크 가시성 확보 도구입니다. 보안 사고 발생 시 감사(Audit) 및 포렌식을 위해 로그를 안전하게 보관해야 합니다.

NFW는 두 가지 형태의 로그를 지원하며, Amazon S3 (단순 스토리지 서비스)[유료] 또는 CloudWatch Logs (클라우드워치 로그)[유료]로 전송할 수 있습니다.

  • Flow Logs (흐름 로그): 방화벽을 통과하는 모든 패킷의 5-Tuple(출발지/목적지 IP, 포트, 프로토콜) 정보를 기록합니다. 트래픽 량 분석 및 라우팅 디버깅에 사용됩니다.
  • Alert Logs (알림 로그): 상태 저장 규칙(Suricata)에 의해 Drop 또는 Alert 처리된 트래픽의 상세 정보(매칭된 페이로드, 룰 번호 등)를 기록합니다. 보안 관제(SOC)의 핵심 데이터입니다.
  • Amazon S3 (Simple Storage Service, 단순 스토리지 서비스): 데이터 백업, 아카이브, 빅데이터 분석 등에 사용되는 확장성과 내구성이 뛰어난 객체 스토리지 서비스입니다.
  • CloudWatch Logs (클라우드워치 로그): 시스템 및 애플리케이션의 로그 파일을 실시간으로 모니터링, 검색, 보관할 수 있는 서비스입니다.
  • SOC (Security Operations Center, 보안 관제 센터): 조직의 IT 인프라에 대한 보안 위협을 24시간 모니터링하고 분석하여 대응하는 중앙 통제 조직입니다.

9. 네트워크 라우팅 통합 설계 (Hub VPC)

NFW가 정상적으로 동작하기 위한 가장 중요한 전제조건은 트래픽이 방화벽을 거쳐가도록 하는 Traffic Steering (트래픽 스티어링, 라우팅 우회)입니다. 중앙 집중식 보안(Centralized Inspection) 구조에서 라우팅은 다음과 같이 설계되어야 합니다.

경로 지점 라우팅 설정 핵심
인터넷 게이트웨이 (IGW) VPC Ingress Routing (VPC 인그레스 라우팅)을 사용하여, ALB (Application Load Balancer)[유료]의 사설 IP(`/32`)를 목적지로 하는 트래픽을 NFW 엔드포인트로 먼저 가로챕니다(Intercept).
전송 게이트웨이 (TGW)[유료] Spoke VPC에서 넘어온 외부행 트래픽(`0.0.0.0/0`)을 Hub VPC 내의 NAT Gateway로 보내고, NAT는 다시 NFW 엔드포인트로 트래픽을 전달하는 체인을 구성합니다.
라우팅 루프(Routing Loop) 주의: 방화벽 서브넷의 라우팅 테이블에서 외부(`0.0.0.0/0`) 타겟을 또다시 방화벽 엔드포인트나 TGW로 설정하면 패킷이 갇히게 됩니다. NFW 서브넷의 외부 타겟은 반드시 IGW여야 합니다.
  • Traffic Steering (트래픽 스티어링): 네트워크 트래픽을 원래의 목적지로 직접 보내지 않고, 검사나 최적화를 위해 특정 보안 장비나 경로로 강제로 우회시키는 네트워크 기법입니다.
  • VPC Ingress Routing (VPC 인그레스 라우팅): 인터넷 게이트웨이(IGW)나 가상 프라이빗 게이트웨이(VGW)에 라우팅 테이블을 연결하여 유입되는 트래픽의 경로를 제어하는 기능입니다.
  • TGW (Transit Gateway, 전송 게이트웨이): 여러 VPC와 온프레미스 네트워크를 단일 게이트웨이로 상호 연결하는 중앙 집중식 허브 라우터 서비스입니다.

10. 고가용성(HA) 아키텍처 및 비용 최적화

엔터프라이즈 환경에서 NFW는 단일 가용 영역(Single-AZ)의 장애에 대비하여 Multi-AZ (다중 가용 영역)로 구축되어야 합니다. 각 AZ마다 전용 Firewall Subnet (방화벽 서브넷)을 생성하고 NFW 엔드포인트를 배치해야 합니다.

비용 최적화(Cost Optimization) 설계 방안

NFW는 강력한 만큼 높은 비용을 수반합니다. 비용 관리를 위해 다음 사항을 설계에 반영해야 합니다.

  • 엔드포인트 비용 절감: NFW는 엔드포인트(생성된 AZ 개수) 단위로 시간당 과금됩니다. 모든 VPC에 개별적으로 NFW를 설치(Distributed 모델)하지 말고, TGW를 활용한 Hub & Spoke 모델을 구축하여 NFW를 중앙(Hub VPC)에서만 운영하십시오.
  • 데이터 처리 비용 절감: 방화벽을 통과하는 GB당 요금이 부과됩니다. 내부 서버 간의 대용량 데이터 복제 트래픽은 TGW 라우팅에서 NFW를 우회(Bypass)하도록 설정하여 불필요한 과금을 방지하십시오.
  • Multi-AZ (Multiple Availability Zones, 다중 가용 영역): 데이터 센터(AZ) 한 곳에 재해나 장애가 발생해도 서비스가 중단되지 않도록, 물리적으로 분리된 여러 AZ에 인프라를 분산 배치하는 고가용성 아키텍처 설계 방식입니다.
  • Hub & Spoke Model (허브 앤 스포크 모델): 중앙의 허브 네트워크(Hub VPC)가 모든 트래픽 통제와 보안 검사를 담당하고, 주변의 스포크 네트워크(Spoke VPC)들은 허브를 통해서만 통신하는 효율적인 중앙 집중형 네트워크 토폴로지입니다.
  • Cost Optimization (비용 최적화): 클라우드 리소스의 사용량, 성능, 아키텍처를 분석하여 불필요한 지출을 제거하고 가장 경제적인 방식으로 시스템을 운영하는 아키텍처 설계 원칙입니다.
