심층 방어 아키텍처: DoS/DDoS 공격의 진화와 AWS 통합 방어 전략

서비스 가용성 파괴의 본질: DoS와 DDoS의 개념

IT 인프라 보안에 있어서 기밀성(Confidentiality)과 무결성(Integrity)만큼이나 중요한 핵심 가치가 바로 Availability (가용성)입니다. 시스템이 정상적인 사용자에게 적절한 시간에 서비스를 제공할 수 있는 능력을 의미하며, 이 가용성을 무너뜨리기 위해 고안된 가장 원초적이면서도 파괴적인 공격 형태가 DoS (Denial of Service, 서비스 거부) 및 DDoS (Distributed Denial of Service, 분산 서비스 거부) 공격입니다.

DoS (서비스 거부 공격)

단일 공격자(단일 IP 또는 단일 시스템)가 타겟 시스템의 자원(네트워크 대역폭, CPU, 메모리, 데이터베이스 커넥션 등)을 고갈시켜 정상적인 서비스 요청을 처리하지 못하도록 방해하는 공격입니다. 초창기 인터넷 환경에서 위협적이었으나, 방어 측의 네트워크 대역폭과 컴퓨팅 파워가 기하급수적으로 커지고, 단일 IP를 방화벽이나 IPS (Intrusion Prevention System, 침입 방지 시스템)에서 쉽게 차단할 수 있게 되면서 그 실효성을 잃었습니다.

DDoS (분산 서비스 거부 공격)

DoS의 한계를 극복하기 위해 진화한 형태입니다. 공격자는 악성코드에 감염되어 원격 제어가 가능한 수만에서 수백만 대의 Zombie PC (좀비 PC) 또는 IoT (Internet of Things, 사물인터넷) 기기들을 네트워크로 연결하여 Botnet (봇넷)을 구성합니다. 이후 C&C (Command and Control, 명령 및 제어) 서버를 통해 일제히 특정 타겟을 공격하도록 명령을 하달합니다. 방어자 입장에서는 전 세계에 분산된 수많은 IP에서 정상적인 요청으로 위장한 트래픽이 쏟아져 들어오기 때문에, 단순한 IP 차단 방식으로는 방어가 거의 불가능에 가깝습니다.

* Availability (가용성): 인가된 사용자가 정보나 시스템을 필요로 할 때 지연이나 거부 없이 즉시 사용할 수 있도록 보장하는 보안의 3대 요소(CIA Triad) 중 하나입니다.
* Botnet (봇넷): 악성코드에 감염되어 해커의 명령을 일제히 수행할 수 있도록 묶여 있는 수많은 컴퓨터나 기기들의 네트워크 집합체입니다.
* C&C (Command and Control) 서버: 해커가 감염된 좀비 기기(봇넷)들에게 해킹 명령을 내리고 조종하기 위해 사용하는 중앙 통제 서버입니다.

DDoS 공격의 구조적 분류 및 주요 기법

아키텍트는 방어선을 구축하기 위해 적의 공격 형태를 정확히 이해해야 합니다. 최신 DDoS 공격은 OSI 7계층 모델을 기준으로 크게 세 가지 카테고리로 분류되며, 복합적으로 섞여서(Multi-Vector) 들어오는 경우가 많습니다.

공격 유형 (카테고리) 설명 및 공격 원리 주요 공격 기법 (Vectors)
볼륨 기반 공격
(Volumetric Attacks)		 OSI 3, 4 계층
가장 흔한 형태입니다. 타겟 네트워크의 대역폭(Bandwidth) 자체를 가득 채워버려, 정상적인 패킷이 아예 물리적 회선으로 들어오지 못하게 막는 무식하지만 강력한 방법입니다. 주로 Amplification (증폭) 기법을 사용합니다.
  • UDP Reflection/Amplification: 공격자가 출발지 IP를 타겟 IP로 위조(Spoofing)한 후, 응답 크기가 매우 큰 서버(NTP, DNS, Memcached 등)에 질의를 보냅니다. 거대한 응답이 타겟 서버로 집중되어 대역폭을 고갈시킵니다.
  • ICMP Flood (Ping Flood): 무수히 많은 Ping 요청을 보내 네트워크 회선을 마비시킵니다.
프로토콜/네트워크 공격
(State-Exhaustion)		 OSI 3, 4 계층
서버나 방화벽, 로드 밸런서 등 네트워크 장비가 유지할 수 있는 동시 연결(Connection) 세션 수(State Table)의 한계를 고갈시키는 공격입니다.
  • SYN Flood: TCP 연결의 3-Way Handshake 취약점을 노립니다. 공격자가 SYN 패킷만 무수히 보내고 최종 ACK를 보내지 않아, 서버가 연결 대기 상태(Half-open)로 메모리를 낭비하다 다운되게 만듭니다.
  • Ping of Death: 규격(65,535 바이트)을 초과하는 기형적인 패킷을 보내 시스템 파괴를 유도합니다.
애플리케이션 계층 공격
(Application Layer)		 OSI 7 계층
네트워크 대역폭은 조금만 사용하면서, 웹 서버(Apache, Nginx 등)나 백엔드 데이터베이스의 CPU, 메모리 자원을 집중적으로 고갈시킵니다. 트래픽 량이 적어 탐지가 매우 어렵습니다.
  • HTTP Flood: 봇넷을 이용해 웹 서버에 무수히 많은 HTTP GET/POST 요청(예: 무거운 DB 쿼리를 유발하는 검색 요청)을 지속적으로 보냅니다.
  • Slowloris (Slow HTTP): HTTP 헤더를 비정상적으로 아주 천천히 전송하여, 웹 서버의 커넥션 스레드를 오랫동안 점유하고 놓아주지 않아 결국 서비스 불능에 빠지게 합니다.
* Amplification (증폭) 기법: 작은 크기의 요청(Request)을 보내면 수십에서 수만 배 큰 응답(Response)을 반환하는 특정 프로토콜의 취약점을 이용해 공격 트래픽의 규모를 폭발적으로 키우는 기법입니다.
* TCP 3-Way Handshake: 신뢰성 있는 통신을 위해 클라이언트와 서버가 데이터를 주고받기 전 'SYN -> SYN/ACK -> ACK' 3단계로 서로 통신 준비를 확인하는 과정입니다.

역사적인 주요 DDoS 실제 사고 사례 (Case Study)

IT 인프라 기술이 발전함에 따라 방어 기술도 진보했지만, 그만큼 공격의 규모와 지능화 수준도 기하급수적으로 상상을 초월하게 발전해 왔습니다.

1. Mirai (미라이) 봇넷의 등장과 Dyn DNS 마비 사태 (2016년)

기존의 PC 봇넷이 아닌, 보안이 취약한 가정용 IP 카메라, 공유기 등 수십만 대의 IoT 기기를 악성코드(Mirai)로 감염시켜 사상 최대 규모의 봇넷을 구성했습니다. 이들은 미국의 주요 DNS 서비스 업체인 'Dyn'을 공격(약 1.2 Tbps 규모)하여, Dyn을 이용하던 트위터, 넷플릭스, 레딧, 아마존 등 미국 인터넷의 절반을 마비시키는 초유의 사태를 발생시켰습니다.

2. GitHub 대상 Memcached 증폭 공격 (2018년)

전 세계 개발자들의 플랫폼인 GitHub가 무려 1.35 Tbps (초당 1.35 테라비트)라는 당시 기준 역대 최대 규모의 트래픽 공격을 받았습니다. 공격자들은 인터넷에 무방비로 노출된 Memcached 서버(인메모리 캐시 서버)의 취약점을 이용했습니다. 1바이트의 요청으로 최대 51,200배의 트래픽을 증폭시켜 GitHub로 쏟아부었으나, GitHub가 이용 중이던 클라우드 스크러빙 센터(트래픽 정제소)로 트래픽을 우회시켜 약 10분 만에 방어에 성공한 성공적인 대응 사례로 꼽힙니다.

3. AWS의 CLDAP 반사 공격 자체 방어 기록 (2020년)

클라우드 환경에서는 클라우드 제공자(CSP) 자체가 거대한 방어막입니다. AWS는 2020년 1분기에 자사 인프라를 향해 들어온 무려 2.3 Tbps 규모의 공격을 AWS Shield (AWS 쉴드)를 통해 무사히 차단했다고 발표했습니다. 이는 Connectionless LDAP(CLDAP) 프로토콜을 이용한 반사/증폭 공격이었습니다.

4. HTTP/2 Rapid Reset 제로데이 공격 (2023년)

최근에는 L7 계층에서의 공격이 극에 달했습니다. 2023년 하반기, AWS, Google Cloud, Cloudflare는 초당 무려 3억 9,800만 RPS (Requests Per Second)라는 상상을 초월하는 규모의 L7 DDoS 공격을 탐지하고 방어했습니다. 이는 HTTP/2 프로토콜이 다중 스트림을 요청(가짜 요청)하고 즉시 취소(RST_STREAM)하는 과정을 반복할 때 서버의 자원이 심각하게 고갈되는 프로토콜 자체의 취약점(CVE-2023-44487)을 악용한 고도의 공격이었습니다.

AWS 환경에서의 완벽한 멀티 레이어 DDoS 방어 아키텍처

온프레미스(물리적 데이터센터) 환경에서 Terabit 단위의 공격을 막아내기 위해 전용 회선과 하드웨어 스크러빙 장비를 구축하는 것은 천문학적인 비용이 듭니다. 그러나 AWS와 같은 글로벌 클라우드 인프라에서는 Anycast (애니캐스트) 네트워크와 거대한 글로벌 엣지(Edge) 네트워크를 바탕으로 트래픽을 분산 흡수하는 아키텍처를 설계할 수 있습니다.

참조: AWS DDoS 복원력 모범 사례 백서

1. 엣지 기반의 분산 방어 (Front-Door)

공격 트래픽이 실제 애플리케이션 서버(EC2)가 있는 리전(Region, 예: 서울)까지 도달하기 전에, 전 세계 수백 개의 PoP (Point of Presence, 엣지 로케이션)에서 1차적으로 트래픽을 흡수하고 차단해야 합니다.

  • Amazon Route 53 (DNS): DNS 쿼리 폭주 공격을 Anycast 라우팅을 통해 전 세계 엣지로 분산시켜 100% 가용성을 보장합니다.
  • Amazon CloudFront (CDN): 정적/동적 콘텐츠를 엣지에서 캐싱(Caching)하여 제공함으로써, 백엔드 서버로 가는 트래픽 자체를 기하급수적으로 줄입니다. 유효하지 않은 포트나 프로토콜(L3/L4 볼륨 공격)은 CloudFront 엣지에서 원천 드랍(Drop)됩니다.

2. AWS Shield: 관리형 DDoS 보호 서비스

AWS는 자사 인프라 보호를 위해 구축한 보안 시스템을 고객에게 직접 제공합니다. 두 가지 티어로 나뉩니다.

  • AWS Shield Standard 기본 무료 제공: 모든 AWS 고객에게 자동으로 적용됩니다. 가장 흔하게 발생하는 L3/L4 네트워크 계층의 볼륨 기반 공격(SYN Flood, UDP Reflection 등)을 자동으로 실시간 탐지하고 완화(Mitigation)합니다.
  • AWS Shield Advanced 초고가 프리미엄 서비스 (월 $3,000 + 데이터 전송 요금): 엔터프라이즈급의 정교한 보호를 제공합니다. 단순 네트워크 공격뿐만 아니라 애플리케이션 계층(L7)의 복잡한 공격까지 방어하며, 24시간 연중무휴로 AWS SRT(Shield Response Team, 보안 침해 사고 대응팀) 전문가의 직접적인 지원을 받을 수 있습니다. 가장 핵심적인 혜택은 비용 보호(Cost Protection)입니다. DDoS 공격으로 인해 Auto Scaling이 발생하여 CloudFront나 EC2 인프라 비용이 폭탄으로 나오더라도, 그 초과 비용을 면제해 줍니다.

3. 애플리케이션 계층 심층 보호

  • AWS WAF (Web Application Firewall): CloudFront나 ALB (Application Load Balancer)에 연결되어 L7 계층의 HTTP/HTTPS 트래픽을 심층 분석합니다. Rate Limiting (속도 제한) 규칙을 적용하여 특정 IP가 5분 내에 비정상적으로 많은 요청을 보내면 자동으로 차단합니다.
* Anycast (애니캐스트): 동일한 IP 주소를 전 세계 여러 대의 서버에 할당하여, 사용자가 접속을 시도할 때 네트워크상 가장 가깝고 빠른 경로의 서버로 자동 연결되게 해주는 네트워크 라우팅 기술입니다.
* PoP (Point of Presence) / Edge Location: 클라우드 제공자가 전 세계 주요 도시에 구축해 놓은 네트워크 접점 통신망 허브입니다.
그림 2. AWS 글로벌 인프라를 활용한 심층적 멀티 레이어 DDoS 방어 아키텍처

아키텍트 실전 팁 및 흔한 오해 (Misconceptions)

클라우드 아키텍처를 설계할 때, 기술에 대한 맹신은 가장 큰 보안 홀(Hole)을 만듭니다. 현장에서 겪게 되는 핵심 통찰을 정리합니다.

자주 오해하거나 실수하는 부분: "DDoS 오면 Auto Scaling(오토 스케일링)이 알아서 서버 늘려주니까 안전한 거 아니야?" 매우 위험하고 순진한 생각입니다. L7 계층의 HTTP Flood 공격을 그대로 백엔드로 흘려보낸 상태에서 Auto Scaling만 구성되어 있다면, 공격 트래픽이 늘어나는 만큼 AWS 자원이 무한히 생성됩니다. 서비스는 죽지 않을지 몰라도, 월말에 청구된 천문학적인 AWS 인프라 비용 때문에 회사가 파산할 수 있습니다.

이를 보안 업계에서는 EDoS (Economic Denial of Sustainability, 경제적 지속 가능성 거부 공격)라고 부릅니다. 인프라가 공격을 견디더라도 재정적으로 서비스를 유지할 수 없게 만드는 치명적인 공격입니다. 따라서 최전방 엣지(CloudFront, WAF)에서 공격을 1차적으로 필터링(Rate Limiting 등)하여 백엔드로 들어오는 트래픽 자체를 차단하는 설계가 필수적입니다.
실무 아키텍트의 팁: 완벽한 차단보다 '완충(Buffering)' 전략 DDoS 공격을 100% 탐지하고 IP를 하나하나 차단하는 것은 사실상 불가능합니다. 훌륭한 아키텍처는 공격 트래픽을 유연하게 '소화'시킵니다.
1. CloudFront 적극 활용: 정적 이미지, JS, CSS뿐만 아니라 변하지 않는 API 응답값도 적극적으로 Edge에 캐싱하십시오. 백엔드(DB) 히트율을 낮추는 것이 가장 강력한 방어입니다.
2. WAF의 Rate Based Rule(속도 기반 규칙): 단일 IP에서 5분당 특정 임계치(예: 100회) 이상의 요청이 오면 무조건 일시 차단하거나 CAPTCHA를 띄우도록 설정하십시오. 이것 하나만으로도 어설픈 L7 봇넷 공격의 80% 이상을 방어할 수 있습니다.
3. VPC 내부 숨기기: ALB와 EC2에는 절대 공인 IP(Public IP)를 직접 할당하지 마십시오. 오직 CloudFront와 WAF를 통해서만 사설 IP(Private IP) 망으로 들어오도록 보안 그룹(Security Group)을 꽉 잠가야 합니다(Origin Cloaking). 공격자가 IP를 직접 치고 들어오는 것을 원천 봉쇄해야 합니다.

네트워크를 마비시키는 물리적 공격(L3/L4)은 AWS Shield가 담당하고, 서비스를 마비시키는 지능형 공격(L7)은 AWS WAF와 아키텍트의 설계(캐싱, 속도 제한, Auto Scaling)가 방어하는 것, 이것이 현대 클라우드 보안 아키텍처의 정수입니다.
