Terragrunt Dependency (의존성) 핵심 개념

Terragrunt (테라그런트)는 순수 Terraform(테라폼)의 불편함을 해소하기 위해 만들어진 래퍼(Wrapper) 도구입니다. 순수 테라폼에서는 다른 모듈의 상태를 읽기 위해 terraform_remote_state라는 데이터 소스를 사용해야 하지만, Terragrunt 환경에서는 dependency (디펜던시, 의존성) 블록이라는 훨씬 더 강력하고 DRY (Don't Repeat Yourself, 중복 배제)한 방식을 사용합니다.

이러한 아키텍처는 네트워크 층, 데이터베이스 층, 애플리케이션 층을 독립된 디렉토리로 분리하여 관리하는 Decoupling (디커플링, 결합 분리)을 완벽하게 지원하며, terragrunt run-all apply 실행 시 인프라의 생성 순서를 자동으로 결정해 주는 결정적인 역할을 합니다.

운영 사례: Dependency 블록을 활용한 모듈 간 데이터 주입

네트워크 전담팀이 구축한 VPC (Virtual Private Cloud, 가상 프라이빗 클라우드)의 서브넷 ID를, 애플리케이션 개발팀이 EC2 (Elastic Compute Cloud) 인스턴스를 배포할 때 동적으로 전달받아야 하는 상황을 가정해 보겠습니다. 애플리케이션 팀은 dependency 블록을 통해 네트워크 팀의 Output (출력값)을 직접 읽어와 inputs으로 주입합니다.

도식 1. Terragrunt Dependency 블록을 활용한 모듈 간 데이터 파이프라인

[App 팀 코드] Dependency를 통해 데이터를 가져오는 방법

terragrunt.hcl 파일에서 네트워크 모듈의 경로를 지정하고, 해당 모듈이 내뱉는 Output을 inputs 속성으로 전달하는 실무 코드입니다.

# app/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

# 1. 네트워크 인프라에 대한 의존성(Dependency) 선언
dependency "network" {
  config_path = "../network" # 네트워크 모듈의 상대 경로

  # [핵심] 네트워크 인프라가 아직 생성되지 않았을 때 plan 명령어가 실패하는 것을 방지
  mock_outputs = {
    public_subnet_id = "subnet-00000000mock"
  }
  mock_outputs_allowed_terraform_commands = ["validate", "plan"]
}

# Terraform 공식 모듈 소스 지정
terraform {
  source = "tfr:///terraform-aws-modules/ec2-instance/aws?version=5.6.0"
}

# 2. 의존성 모듈의 Output을 Input으로 주입 [과금 발생]
inputs = {
  name          = "Web-Server-01"
  ami           = "ami-0c9c942bd01f50564"
  instance_type = "t3.micro"

  # dependency.[이름].outputs.[출력변수명] 형태로 직접 접근
  subnet_id     = dependency.network.outputs.public_subnet_id
}

[자주 오해하거나 실수하는 부분: Mock Outputs의 중요성]

초기 인프라 구축 시, 전체 모듈에 대해 terragrunt run-all plan을 실행하면 의존성 모듈(네트워크)이 아직 생성되지 않아 상태 파일(State)에 값이 존재하지 않습니다. 이 때문에 plan 명령어가 에러를 뱉으며 중단됩니다. 이를 방지하기 위해 mock_outputs (가짜 출력값)를 반드시 선언하여, Terragrunt가 plan 단계에서는 임시 값을 사용해 종속성 검사를 통과하도록 설계해야 합니다.

운영 사례: 3계층(3-Tier) 아키텍처 다중 의존성(Multi-Dependencies) 주입

엔터프라이즈 환경의 표준인 네트워크(Network) 층, 데이터베이스(Database) 층, 애플리케이션(Application) 층으로 구성된 3계층 아키텍처에서는 다중 의존성 관리가 필수적입니다. 애플리케이션 모듈은 네트워크 모듈로부터 Subnet ID (서브넷 식별자)를 받아야 하고, 동시에 데이터베이스 모듈로부터 DB Endpoint (데이터베이스 접속 주소)를 받아야 합니다.

💡 [아키텍트의 실전 팁: 의존성 블록의 다중 선언]

순수 테라폼에서는 여러 개의 terraform_remote_state를 선언하고 관리하는 것이 번거롭지만, Terragrunt에서는 필요한 만큼 dependency 블록을 추가 선언하기만 하면 됩니다. 이렇게 하면 run-all 명령어 실행 시 Terragrunt가 네트워크 → 데이터베이스 → 애플리케이션 순서로 Dependency Graph (의존성 그래프)를 자동 생성하여 배포 순서를 완벽하게 제어합니다.

[App 팀 코드] Network와 Database 모듈의 데이터를 동시에 받아오는 방법

# app/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

# 1. 네트워크 계층 의존성 (VPC, Subnet 정보 획득)
dependency "network" {
  config_path = "../network"
  mock_outputs = {
    private_subnet_ids = ["subnet-0001mock", "subnet-0002mock"]
    vpc_id             = "vpc-0000mock"
  }
  mock_outputs_allowed_terraform_commands = ["validate", "plan"]
}

# 2. 데이터베이스 계층 의존성 (접속 엔드포인트 획득)
dependency "database" {
  config_path = "../database"
  mock_outputs = {
    db_endpoint = "mock-db.cluster-xxxx.ap-northeast-2.rds.amazonaws.com"
  }
  mock_outputs_allowed_terraform_commands = ["validate", "plan"]
}

# 3. 애플리케이션 모듈에 다중 데이터 주입 [과금 발생]
inputs = {
  name        = "Enterprise-App-Server"
  vpc_id      = dependency.network.outputs.vpc_id
  subnet_ids  = dependency.network.outputs.private_subnet_ids
  
  # EC2 인스턴스의 환경 변수(User Data 등)로 DB 주소 전달
  db_endpoint = dependency.database.outputs.db_endpoint
}

다중 모듈 스크립트 실행 및 결과 검증 (Execution & Verification)

다중 의존성이 설정된 환경에서는 개별 디렉토리에 들어가서 명령어를 칠 필요가 없습니다. 최상위 디렉토리에서 run-all 명령어를 사용하면 전체 인프라를 한 번에 오케스트레이션(Orchestration, 자동화된 제어)할 수 있습니다.

Step 1. 전체 초기화 및 실행 계획 확인

terragrunt run-all init
terragrunt run-all plan

작동 원리: 모든 하위 디렉토리를 순회하며 .terraform 폴더를 초기화합니다. plan 단계에서는 의존성에 따라 Network → DB → App 순서로 계획이 수립되는 것을 터미널 로그에서 확인할 수 있습니다.

Step 2. 전체 인프라 일괄 프로비저닝 [과금 발생 지점]

terragrunt run-all apply

작동 원리: 의존성 그래프에 기반하여 네트워크 생성이 완료되어야만 데이터베이스 생성을 시작하고, 데이터베이스 생성이 완료되어야만 애플리케이션 생성을 시작합니다. 도중에 오류가 발생하면 그 시점에서 안전하게 중단됩니다.

Step 3. 상태 검증 및 확인 (Cross-Validation)

terragrunt run-all output

작동 원리: 배포가 완료된 후, 각 모듈이 내뱉는 최종 출력값(VPC ID, DB Endpoint 등)을 터미널에서 한눈에 확인합니다. 이후 AWS 관리 콘솔 (GUI)에 접속하여 Enterprise-App-Server EC2 인스턴스의 사용자 데이터 (User Data)나 환경 변수에 RDS (Relational Database Service) 엔드포인트가 정상적으로 주입되었는지 교차 검증합니다.

[필수 보안 및 관리 작업: 일괄 삭제를 통한 과금 방지]

실습이 완료된 인프라는 방치할 경우 지속적으로 과금이 발생합니다. terragrunt run-all destroy 명령어를 실행하면 Terragrunt가 생성의 역순(App → DB → Network)으로 의존성을 안전하게 해제하며 모든 리소스를 일괄 파기합니다.

terragrunt run-all destroy

운영 사례: Generate 블록을 통한 동적 Data Source 주입

Terragrunt는 코드를 동적으로 생성해내는 Generate (제너레이트, 코드 생성) 기능을 지원합니다. 클라우드 제공자의 API를 직접 쿼리(Query)하여 가장 최신의 AMI (Amazon Machine Image, 아마존 머신 이미지)를 가져오는 Terraform Data Source 블록을 terragrunt.hcl에서 직접 주입하여 코드의 유연성과 이식성을 극대화할 수 있습니다.

💡 [아키텍트의 실전 팁: 하드코딩 제거 (No Hardcoding)]

AWS의 AMI ID는 리전(Region)마다 다르며 주기적으로 보안 패치 버전이 새로 릴리즈되어 ID가 변경됩니다. ami = "ami-123456" 처럼 코드에 박아두면 몇 달 뒤 인스턴스 교체 시 에러가 발생합니다. Terragrunt의 Generate 기능을 통해 "가장 최신의 Amazon Linux 2023 이미지"를 조회하는 코드를 런타임에 동적으로 삽입하는 것이 자동화의 핵심(Best Practice)입니다.

Generate 블록을 활용하여 Data Source 코드를 자동 생성하는 방법

# terragrunt.hcl

# 1. 실행 시점에 data.tf 파일을 동적으로 생성하여 모듈 내부에 삽입
generate "data_ami" {
  path      = "data.tf"
  if_exists = "overwrite"
  contents  = <# AWS 공식 이미지로 한정

  # 필터 조건을 통해 원하는 운영체제 식별
  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }
}
EOF
}

# 2. 생성된 데이터 소스를 로컬 모듈 코드에서 참조
terraform {
  source = "./modules/my-custom-ec2"
}

# ※ modules/my-custom-ec2/main.tf 에서는 아래와 같이 사용합니다:
# ami = data.aws_ami.latest_amazon_linux.id
* Data Source의 생명주기: terragrunt plan 또는 apply 명령이 실행되는 시점에 클라우드 API를 실시간으로 호출하여 데이터를 읽어옵니다. 따라서 코드 실행 시점의 최신 상태를 항상 보장받을 수 있습니다. Data Source 호출 자체는 클라우드 벤더에서 과금하지 않습니다.

본 교재는 기업 환경의 실제 인프라스트럭처 설계 및 구축 운영 노하우를 바탕으로 작성되었습니다.

참고: Terragrunt Dependencies 공식 문서

요청하신 대로 기존 순수 Terraform의 `terraform_remote_state` 방식에서 Terragrunt(테라그런트)의 핵심 기능인 `dependency` 블록을 활용하는 실무 사례로 문서를 완전히 개편하였습니다. 도식 그림 역시 Terragrunt 아키텍처에 맞게 수정하고, Terragrunt만의 특별한 기능인 `mock_outputs`와 `generate` 블록의 개념을 상세히 추가했습니다. 이니셜 설명, 과금 안내, 존댓말/대화체 배제 등 모든 개인 맞춤 설정 사항을 철저히 확인하고 적용하였습니다.
