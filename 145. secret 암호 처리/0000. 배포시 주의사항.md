    <!-- Section 1: The Critical Risk -->
    <section class="mb-24">
        <h2 class="text-3xl font-bold text-slate-800 mb-8 flex items-center gap-3">
            <span class="bg-slate-900 text-white w-10 h-10 flex items-center justify-center rounded-lg">1</span>
            상태 파일을 공유 저장소에 노출할 수 없는 3대 이유
        </h2>
        <p class="text-lg">
            테라폼의 <span class="highlight">State File (상태 파일, 테라폼이 관리하는 인프라의 현재 정보를 담은 JSON 형식의 파일)</span>은 프로젝트의 설계도이자 현재 인프라의 정확한 스냅샷입니다. 이를 GitHub과 같은 공개 또는 공유 저장소에 업로드하는 행위는 다음과 같은 치명적인 위험을 내포합니다.
        </p>

        <div class="grid md:grid-cols-3 gap-6 mt-8">
            <div class="tech-box">
                <h4 class="font-bold text-red-600 mb-2">① 민감 정보 노출</h4>
                <p class="text-sm">테라폼은 변수에 `sensitive = true`를 설정하더라도, 상태 파일 내부에는 <span class="important">Plaintext (평문, 암호화되지 않은 일반 텍스트)</span>로 모든 정보를 저장합니다. DB 암호, API 키가 코드 이력에 영구히 남게 됩니다.</p>
            </div>
            <div class="tech-box">
                <h4 class="font-bold text-amber-600 mb-2">② 상태 드리프트</h4>
                <p class="text-sm"><span class="highlight">State Drift (상태 드리프트, 실제 리소스의 상태와 상태 파일에 기록된 정보가 일치하지 않는 현상)</span>가 발생합니다. 여러 작업자가 각자의 로컬 파일을 커밋하면 인프라 형상이 꼬이게 됩니다.</p>
            </div>
            <div class="tech-box">
                <h4 class="font-bold text-blue-600 mb-2">③ 동시성 제어 불가</h4>
                <p class="text-sm">Git은 단순 이력 관리 도구일 뿐입니다. 인프라가 변경 중일 때 다른 사람이 수정을 시도하는 것을 막는 <span class="highlight">Locking (잠금)</span> 기능이 없어 리소스 중복 생성이나 삭제 사고가 발생합니다.</p>
            </div>
        </div>

        <div class="footnote">
            <strong>용어 상세 설명:</strong>
            <ul class="list-disc list-inside space-y-1">
                <li><strong>JSON (JavaScript Object Notation, 자바스크립트 객체 표기법):</strong> 데이터를 구조적으로 저장하기 위한 경량의 텍스트 기반 형식입니다.</li>
                <li><strong>Credential (자격 증명):</strong> 시스템에 접근하기 위한 암호, 토큰, 인증서 등의 민감한 정보를 통칭합니다.</li>
            </ul>
        </div>
    </section>

        <p class="text-lg">
            엔터프라이즈 환경에서는 로컬 PC가 아닌 보안이 강화된 외부 저장소인 <span class="highlight">Backend (백엔드, 테라폼 상태 파일이 저장되는 물리적 저장 위치)</span>를 사용해야 합니다. AWS 환경에서는 <strong>S3</strong>와 <strong>DynamoDB</strong>의 조합이 공식적으로 권장되는 아키텍처입니다.

        <div class="tech-box">
            <h3 class="font-bold text-slate-800 mb-4">핵심 컴포넌트의 역할</h3>
            <ul class="space-y-4">
                <li>
                    <strong>Amazon S3 (Simple Storage Service, 아마존 단순 저장 서비스):</strong> 
                    객체 기반 저장소로 <span class="highlight">SSE (Server-Side Encryption, 서버측 암호화)</span>를 통해 데이터를 저장 시 자동으로 암호화하고, 버전 관리 기능을 통해 손상된 상태 파일의 복구를 지원합니다.
                </li>
                <li>
                    <strong>Amazon DynamoDB (아마존 다이너모디비, 완전 관리형 NoSQL 데이터베이스):</strong> 
                    상태 잠금을 위한 고성능 키-값 저장소입니다. 테라폼 명령이 실행되면 고유한 <span class="highlight">LockID (잠금 식별자)</span>를 생성하여 다른 작업자의 동시 접근을 원천 차단합니다.
                </li>
            </ul>
        </div>

        <div class="warning-box">
            <p class="font-bold text-red-800">💰 과금 및 유료 서비스 주의사항:</p>
            <ul class="text-sm list-disc list-inside mt-2">
                <li><strong>Amazon S3:</strong> 데이터 저장량 및 API 호출 횟수(PUT, GET)에 따라 비용이 발생합니다.</li>
                <li><strong>Amazon DynamoDB:</strong> <span class="important">On-demand (온디맨드, 요청당 과금)</span> 방식을 권장하지만, 프로비저닝 방식을 사용할 경우 고정 비용이 발생할 수 있습니다. (테라폼 잠금용으로는 프리 티어 범위 내에서 충분히 운영 가능합니다.)</li>
            </ul>

    <section class="mb-24">
        <h2 class="text-3xl font-bold text-slate-800 mb-8 flex items-center gap-3">
            <span class="bg-slate-900 text-white w-10 h-10 flex items-center justify-center rounded-lg">3</span>
            실무 백엔드 설정 및 가이드라인
        </h2>
        
        <p class="text-lg">
            공식 문서에서 권장하는 <span class="highlight">Partial Configuration (부분 설정, 일부 설정값을 변수나 별도 파일로 분리하여 유연성을 높이는 기법)</span>을 활용하여 범용적인 백엔드 코드를 작성합니다.
        </p>

        <div class="code-block">
<pre>
# backend.tf 또는 main.tf 내부에 정의
terraform {
  backend "s3" {
    bucket         = "tf-state-prod-shared"         # 상태 파일이 담길 S3 버킷 이름
    key            = "infra/vpc/terraform.tfstate"  # 저장 경로 및 파일명
    region         = "ap-northeast-2"               # 서울 리전
    dynamodb_table = "terraform-lock-table"         # 잠금 관리를 위한 테이블명
    encrypt        = true                           # SSE(서버측 암호화) 활성화 필수
  }
}
</pre>
        </div>

        <div class="expert-tip">
            <h4 class="font-bold text-blue-800 mb-2">🛡️ 테라폼 아키텍트의 보안 권고 사항</h4>
            <ul class="list-disc list-inside text-sm space-y-2">
                <li><strong>S3 버킷 정책:</strong> <span class="highlight">IAM (Identity and Access Management, 식별 및 액세스 관리)</span> 정책을 통해 승인된 엔지니어만 버킷에 접근하도록 제한하십시오.</li>
                <li><strong>MFA Delete 활성화:</strong> 실수로 상태 파일을 삭제하지 못하도록 <span class="highlight">MFA (Multi-Factor Authentication, 다요소 인증)</span>가 활성화되어야 삭제가 가능하게 설정하십시오.</li>
                <li><strong>환경별 분리:</strong> 운영(Production)과 개발(Dev) 환경은 반드시 <span class="important">서로 다른 버킷 혹은 서로 다른 경로(Key)</span>를 사용하여 충돌을 방지하십시오.</li>
            </ul>

    <section class="mb-24">
        <h2 class="text-3xl font-bold text-slate-800 mb-8 flex items-center gap-3">
            <span class="bg-slate-900 text-white w-10 h-10 flex items-center justify-center rounded-lg">4</span>
            최후의 방어선: .gitignore 설정
        </h2>
        <p class="text-lg">
            원격 백엔드를 설정하더라도 로컬 작업 중에 생성되는 부산물들이 Git에 포함될 수 있습니다. 프로젝트 루트에 반드시 아래 내용을 포함한 <span class="highlight">.gitignore (깃 제외 설정 파일)</span>를 작성하십시오.
        </p>

        <div class="code-block">
<pre>
# .gitignore

# 테라폼 로컬 상태 파일 및 백업 (이미 S3에 있으므로 로컬 파일은 무시)
*.tfstate
*.tfstate.backup

# 테라폼 작업 디렉터리 (플러그인, 모듈 데이터 포함)
.terraform/

# 변수 정의 파일 (절대로 민감 정보가 포함된 파일을 올리지 마십시오)
*.tfvars
*.tfvars.json

# 크래시 로그
crash.log
</pre>
        </div> 

        <div class="warning-box" style="border-left-color: #f59e0b; background-color: #fffbeb;">
            <p class="font-bold text-amber-800">⚠️ 만약 이미 커밋을 했다면?</p>
            <p class="text-sm mt-1">단순히 파일을 삭제하는 것으로는 부족합니다. Git의 커밋 히스토리(History)에는 데이터가 남아 있으므로, <span class="important">BFG Repo-Cleaner</span>와 같은 도구를 사용하여 과거 이력에서 해당 정보를 완전히 삭제하거나, 모든 자격 증명을 <span class="highlight">Rotation (로테이션, 기존 암호를 파기하고 새로운 암호로 교체)</span>해야 합니다.</p>
        </div>
