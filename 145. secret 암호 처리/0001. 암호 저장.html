테라폼 변수 아키텍처 및 민감 정보 보호

Hardcoding(하드코딩, 고정값 입력) 탈피와 보안 거버넌스 수립

1 민감 정보 보호의 당위성

현대적인 IaC (Infrastructure as Code, 코드형 인프라) 아키텍처에서 가장 빈번하게 발생하는 보안 사고는 자격 증명(Credential)의 노출입니다. 코드 내부에 암호를 직접 기록하는 하드코딩 방식은 VCS (Version Control System, 버전 관리 시스템)를 통해 전파되어 돌이킬 수 없는 피해를 초래합니다. 이번 섹션에서는 변수 분리와 민감 정보 가리기 기능을 통해 이를 방어하는 전략을 학습합니다.

[도식 1] 선언(정의)과 주입(값)의 논리적 분리 아키텍처

2 실습: 변수 분리 및 민감 정보 보호

1. 변수 정의 및 민감 설정 (variables.tf)

variables.tf
variable "db_admin_password" {
  description = "MySQL Employees DB 관리자 비밀번호"
  type        = string
  # sensitive를 true로 설정하면 terraform plan/apply 시 화면에 값이 표시되지 않습니다.
  sensitive   = true
}

2. 메인 코드 수정 (main.tf)

main.tf
resource "local_file" "mysql_admin_config" {
  content  = <<EOF
[client]
host=mysql-employees-db.example.com
user=admin
password=${var.db_admin_password}
port=3306
EOF

  filename = "${path.module}/mysql_admin.cnf"
}

# 출력값에도 sensitive 속성이 전파됩니다.
output "db_password_output" {
  value     = var.db_admin_password
  sensitive = true
}

3. 데이터 주입 (terraform.tfvars)

⚠️ 아키텍트 주의사항:

이 파일은 환경별로 다른 값을 가지며 민감 정보를 포함하므로 절대 공유 저장소(Git)에 커밋하지 않습니다.

terraform.tfvars
db_admin_password = "EnhancedSecurityPass!2026"

3 실행 및 보안 가시성 검증

명령어를 실행하여 테라폼이 민감 정보를 어떻게 처리하는지 CLI (Command Line Interface, 명령 줄 인터페이스) 상에서 확인합니다.

$ terraform plan

# ... 생략 ...
+ resource "local_file" "mysql_admin_config" {
    + content              = (sensitive value)
    + filename             = "./mysql_admin.cnf"
    # ... 생략 ...
}

Plan: 1 to add, 0 to change, 0 to destroy.

Changes to Outputs:
  + db_password_output = (sensitive value)

💡 전문가 분석:

`sensitive = true` 옵션이 설정되면 테라폼은 해당 변수가 포함된 리소스의 속성이나 출력값을 화면에 표시할 때 (sensitive value)로 마스킹 처리합니다. 이는 로깅 시스템이나 협업 도구에서 실수로 비밀번호가 노출되는 것을 방지하는 일차적인 방어선입니다.

4 아키텍처적 한계: State 파일 노출

[도식 2] 화면 마스킹과 상태 파일 평문 저장의 괴리

명령어를 실행한 후 `terraform.tfstate` 파일을 열어보십시오. 화면에서는 가려졌던 비밀번호가 파일 내부에는 평문(Plaintext)으로 기록되어 있음을 확인할 수 있습니다. 이것은 테라폼의 설계 철학 상 인프라의 현재 상태를 완벽하게 유지해야 하기 때문입니다.

5 최종 보안 거버넌스 수립 조치

1. 원격 백엔드 전환

상태 파일을 로컬이 아닌 S3와 같은 외부 저장소로 옮기고, SSE (Server-Side Encryption, 서버 측 암호화)를 강제 적용합니다.

2. 접근 제어 (IAM)

상태 파일이 저장된 위치에 대한 접근 권한을 IAM (Identity and Access Management, 식별 및 액세스 관리) 정책을 통해 엄격히 제한합니다.

주석 및 용어 정의:

  • HCL (HashiCorp Configuration Language, 하시코프 설정 언어): 테라폼에서 사용하는 인프라 정의 언어입니다.
  • VCS (Version Control System, 버전 관리 시스템): Git과 같이 소스 코드의 변경 이력을 관리하는 시스템입니다.
  • Backend (백엔드): 테라폼의 상태 파일이 저장되는 위치를 의미하며 로컬과 원격으로 나뉩니다.
  • KMS (Key Management Service, 키 관리 서비스): 데이터 암호화에 사용되는 암호화 키를 생성하고 관리하는 서비스입니다.
  • 비용 안내: S3 원격 백엔드 사용 시 S3 저장 비용 및 API 호출 비용이 발생하며, KMS 암호화 적용 시 추가 과금이 발생합니다. (AWS 프리티어 범위를 확인하십시오.)
