테라폼 변수 보안: 대화형 입력 실습

Interactive Input(대화형 입력)을 활용한 민감 정보 처리 및 아키텍처적 한계 분석

1 개요 및 아키텍트의 시각

인프라스트럭처 프로비저닝(Infrastructure Provisioning, 인프라 자원 준비 및 배치) 과정에서 데이터베이스 암호와 같은 민감 정보가 코드나 파일에 기록되는 순간, 보안 사고의 위험은 기하급수적으로 증가합니다. 이를 방어하기 위해 테라폼은 파일에 값을 저장하지 않고 실행 시점에 입력을 요구하는 Interactive Input(대화형 입력) 방식을 지원합니다.

💡 아키텍트의 진실 확인 (Truth Check):

사용자가 키보드로 입력한 값은 VCS(Version Control System, 버전 관리 시스템)의 추적을 피할 수 있지만, 테라폼 엔진이 인프라 상태를 추적하는 State File(상태 파일)에는 여전히 평문으로 남습니다. 즉, 입력 방식의 변경만으로는 완전한 보안을 달성할 수 없으며, 반드시 상태 파일의 격리 및 암호화가 병행되어야 합니다.

2 실습 환경 및 코드 구현

기존 설정 파일들과의 충돌을 방지하기 위해 독립된 작업 디렉터리(Directory, 컴퓨터 폴더)를 생성합니다.

$ mkdir terraform-interactive-input $ cd terraform-interactive-input

1. 변수 선언 (variables.tf)

기본값(Default)을 명시하지 않아야 테라폼이 실행 시 값을 요청하게 됩니다.

variables.tf
variable "db_admin_password" {
  description = "MySQL Database 관리자 비밀번호"
  type        = string
  sensitive   = true  # 화면 출력 시 마스킹 처리를 위한 속성
}

2. 리소스 정의 (main.tf)

main.tf
resource "local_file" "mysql_admin_config" {
  content  = <<EOF
[client]
host=mysql-secure-db.example.com
user=admin
password=${var.db_admin_password}
EOF

  filename = "${path.module}/secure_mysql_admin.cnf"
}

3 실행 계획 수립 및 대화형 주입

실무 아키텍처 표준인 Plan Output(실행 계획 파일 저장) 전략을 사용합니다. 이는 보안 정보 입력의 번거로움을 줄이고 운영 사고를 방지하는 핵심 기법입니다.

[도식 3] 대화형 입력과 실행 계획 파일을 결합한 보안 워크플로우

$ terraform init $ terraform plan -out=secure_plan.tfplan

위 명령 실행 시 비밀번호를 묻는 프롬프트가 나타나면 키보드로 암호를 입력합니다.

$ terraform apply "secure_plan.tfplan"

💡 실전 아키텍처 전략:

`-out` 플래그를 사용하지 않으면 `plan` 시점에 한 번, `apply` 시점에 또 한 번 비밀번호를 입력해야 합니다. 이는 자동화된 CI/CD Pipeline(지속적 통합/배포 파이프라인) 구축 시 사용자 응답을 기다리며 무한 대기하는 현상을 유발하므로, 실행 계획 저장 방식이 아키텍처 설계의 기본입니다.

4 보안 검증 및 잔존 위협 분석

실행 후 터미널 화면과 물리적 파일에 저장된 암호의 형태를 비교하여 기술적 한계를 분석합니다.

[도식 4] 테라폼 보안의 역설: 화면 출력 마스킹과 상태 파일 평문 저장의 괴리

⚠️ 보안 전문가의 긴급 진단:

`cat terraform.tfstate` 명령을 통해 확인한 결과, 키보드로 정성껏 입력했던 암호가 최종 파일에는 고스란히 저장되어 있습니다. 이는 테라폼이 인프라를 관리하기 위해 필요한 정보를 속성(Attribute)에 모두 저장해야 하는 구조적 한계 때문입니다. 따라서 로컬 환경에 상태 파일을 남기는 것은 보안상 매우 위험합니다.

5 최종 아키텍처 설계 제안

최고 수준의 보안 가버넌스를 위해 다음과 같은 단계별 조치를 강력히 권고합니다.

  • Phase 1

    상태 파일 원격 격리: AWS S3와 같이 서버 측 암호화(SSE)와 IAM 정책 제어가 가능한 원격 백엔드로 상태 파일을 즉시 이전하십시오.

  • Phase 2

    버전 관리 제외: `.gitignore` 파일을 작성하여 상태 파일과 부산물이 Git 저장소에 절대 포함되지 않도록 원천 차단하십시오.

  • Phase 3

    과금 관리: 원격 백엔드 운영 시 발생하는 S3 저장 및 데이터 처리 비용을 주기적으로 모니터링하십시오. (S3는 저장 용량 대비 저렴하지만 API 호출 비용이 수반됩니다.)

주석 및 용어 정의:

  • CLI (Command Line Interface, 명령 줄 인터페이스): 사용자가 텍스트 형식의 명령어를 입력하여 프로그램을 제어하는 환경입니다.
  • Binary File (이진 파일): 텍스트가 아닌 컴퓨터가 직접 해석할 수 있는 형식으로 구성된 파일로, 일반 편집기로는 내용 확인이 불가능합니다.
  • Prompt (프롬프트): 컴퓨터가 사용자에게 입력을 기다리고 있음을 알리는 신호 또는 입력창을 의미합니다.
  • Masking (마스킹): 민감한 데이터의 일부 또는 전체를 특정 문자(예: *)로 가려 노출을 방지하는 기술입니다.
  • AWS Network Firewall 비용 안내: 이 실습 자체는 로컬 파일을 다루므로 과금되지 않으나, 추후 AWS S3 백엔드 및 네트워크 방화벽 도입 시 시간당 사용료와 데이터 처리 GB당 비용이 발생하므로 주의가 필요합니다.
